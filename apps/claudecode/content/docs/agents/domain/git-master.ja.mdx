## 概要

Git Master はクリーンでアトミックな git 履歴を作成するスペシャリストエージェントです。
コミット分割、スタイル一貫性のあるメッセージ作成、安全な履歴操作、ブランチ管理を処理します。

Git 履歴は将来のためのドキュメントです。
15 ファイルを含む 1 つの巨大なコミットはバイセクト、レビュー、または復帰できません。
それぞれ 1 つのことを実行するアトミックコミットは履歴を有用にします。

Git Master は独立して動作し、他のエージェントをスポーンしません。

## 使用時機

- 複数ファイルの変更を論理的に別々のコミットに分割するとき
- プロジェクトのコミットスタイルに合うコミットメッセージを書くとき
- リベース操作が必要なとき
- git 履歴をクリーンアップするとき

## 使用例

```
"Split these changes into atomic commits"
"Rebase the recent commits"
"Find when this bug was introduced in the git history"
```

## ワークプロセス

1. **スタイル検出**: `git log -30` を使用して最後の 30 コミットの言語（English/Korean）とフォーマット（semantic/plain/short）を決定
2. **変更分析**: `git status` と `git diff --stat` を使用して変更ファイルを論理的な関心事にマッピング
3. **関心ごとに分割**: 異なるディレクトリ/モジュール = 分割、異なるコンポーネントタイプ = 分割、独立して復帰可能 = 分割
4. **アトミックコミットを作成**: 依存順序でコミット、検出されたスタイルに合わせる
5. **検証**: git log 出力で結果を確認

## コミット分割ガイドライン

| 変更ファイル数 | 最小コミット数 |
|-------------|-------------|
| 3+ ファイル | 2+ コミット |
| 5+ ファイル | 3+ コミット |
| 10+ ファイル | 5+ コミット |

各コミットは独立して復帰した場合、ビルドを破壊してはいけません。

## セーフティルール

- main/master ブランチでのリベースなし
- 常に `--force` の代わりに `--force-with-lease` を使用
- リベース前にダーティファイルをスタッシュ
- コミットスタイル（言語、フォーマット）はプロジェクトの既存の規約に合わせる

## 他のエージェントとの組み合わせ

- **executor**: Executor が実装を完了した後、Git Master がコミットを整理
- **code-reviewer**: コードレビュー後、適切に修正をコミット
- **verifier**: コミット後のビルド検証は verifier の責務
