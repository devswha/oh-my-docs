## 概要

Writer は開発者が読みたいと思うクリアで正確な技術ドキュメントを作成するエージェントです。
README、API ドキュメント、アーキテクチャドキュメント、ユーザーガイド、コード コメントを処理します。

デフォルトモデルは haiku で、高速でコスト効率的に実行できます。

コア原則は正確性です。
不正確なドキュメントはドキュメントがないより悪い — 開発者を誤った方向に導きます。
すべてのコード例をテストし、すべてのコマンドを検証する必要があります。

## 使用時機

- README またはプロジェクトドキュメントを書くとき
- API ドキュメントを作成するとき
- 移行ノートまたはチェンジログを書くとき
- コード コメント を追加するとき

## 使用例

```
"Write a README for this project"
"Document the auth API endpoints"
"Add JSDoc comments to this module"
```

## 記述プロセス

1. **リクエスト分析**: 正確なドキュメント タスクを特定
2. **コードベース探索**: Glob/Grep/Read を使用してドキュメント化するコードを理解（並列実行）
3. **既存ドキュメントスタイルを確認**: 既存ドキュメントの構造、スタイル、規約を理解
4. **検証されたコード例を含むドキュメントを記述**: すべての例とコマンドをテスト
5. **結果を報告**: 検証結果を含める

## 記述原則

- **能動態、直接言語、不要な単語なし**
- **スキャン可能な構造**: 見出し、コードブロック、テーブル、箇条書きを使用
- **正確なスコープ**: リクエストされたもののみをドキュメント化、スコープ蔓延を防止
- **検証**: ドキュメントに含める前にコード例とコマンドをテスト。テストできない場合は明示的に状態

Writer は記述パスのみを実行します。
同じコンテキストで自己レビューまたは自己承認しません。
レビューが必要な場合、別のレビュー/検証パスに委譲します。

## 他のエージェントとの組み合わせ

- **explore**: ドキュメント化するコードの構造を検索・理解するのに使用
- **executor**: ドキュメント更新がコード変更と並行して必要な場合、executor と並列実行
- **document-specialist**: 外部 API/SDK ドキュメント検索は document-specialist で処理
