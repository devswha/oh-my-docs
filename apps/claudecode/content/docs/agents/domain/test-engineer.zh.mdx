## 概述

Test Engineer 是一个设计测试策略、编写测试、加强脆弱测试，
并指导 TDD 工作流的智能体。

测试是预期行为的可执行文档。
未测试的代码是债务，脆弱的测试侵蚀对测试套件的信任。

## 何时使用

- 当建立测试策略和分析覆盖率差距时
- 当运行 TDD 工作流时
- 当诊断和修复脆弱测试的根本原因时
- 当为新特性编写测试时

## 使用示例

```
"Establish a test strategy for this module"
"Implement password validation with tdd"  # 魔法关键词激活 TDD 模式
"Fix this flaky test"
```

## 测试金字塔

Test Engineer 遵循测试金字塔原则。

| 层 | 比例 | 说明 |
|------|------|------|
| 单元 | 70% | 验证单个函数/模块的行为 |
| 集成 | 20% | 验证组件间的交互 |
| E2E | 10% | 验证整体系统行为 |

## TDD 流程（铁律）

**在任何生产代码前，必须存在失败的测试。**

1. **RED**：为下一个特性编写测试。运行它 — 它必须失败。如果它通过，测试是错误的
2. **GREEN**：只编写使测试通过的最小代码。无额外特性。运行它 — 它必须通过
3. **REFACTOR**：改进代码质量。每次更改都运行测试。保持绿色
4. **REPEAT**：继续下一个失败的测试

如果你先写了代码？删除它并从测试开始重新开始。

## 脆弱测试诊断

脆弱测试的常见原因和修复：

| 原因 | 修复 |
|------|------|
| 时序依赖 | 使用 `waitFor`、删除硬编码 sleep |
| 共享状态 | 在 `beforeEach` 中重置状态 |
| 硬编码日期 | 使用相对日期 |
| 环境依赖 | 使用容器/模拟 |

## 与其他智能体组合

- **executor**：特性实现是 executor 的工作；测试编写是 test-engineer 的
- **verifier**：测试充分性评估和验证由 verifier 处理
- **debugger**：与 debugger 协作进行测试失败的根本原因分析
