## 概要

QA Tester は tmux セッションで実際のアプリケーション動作を対話的に検証するエージェントです。
サービス起動、コマンド送信、出力キャプチャ、期待値に対する検証、クリーンアップを処理します。

ユニットテストはコードロジックを検証しますが、QA テストは実際の動作を検証します。
アプリケーションはすべてのユニットテストに合格しても、実際に実行されるときに失敗する可能性があります。
tmux 経由の対話的テストは起動失敗、統合問題、ユーザー対面バグを捉えます。

## 使用時機

- 自動テストスイートが問題の動作をカバーしていないとき
- 対話的 CLI 入力/出力シミュレーションが必要なとき
- サービス起動/シャットダウン テストが必要なとき
- ストリーミング/リアルタイム動作検証が必要なとき

### 使用しない場合

- プロジェクトに機能をカバーするテストがあるとき（テスト実行がより効率的）
- 単純なコマンド検証の場合（直接実行がより効率的）
- 静的コード分析の場合（architect がより適切）

## 使用例

```
"Actually test that the API server responds correctly"
"Validate the interactive features of this CLI tool"
"Check the /health endpoint after service startup"
```

## テストプロセス

1. **前提条件確認**: tmux がインストールされているか、ポートが利用可能か、ディレクトリが存在するか確認。満たされない場合は即座に失敗
2. **セットアップ**: ユニークな名前（`qa-{service}-{test}-{timestamp}`）を持つ tmux セッション を作成、サービスを起動、準備シグナルを待機
3. **実行**: テストコマンドを送信、出力を待機、`tmux capture-pane` で キャプチャ
4. **検証**: キャプチャされた出力を期待パターンと比較。PASS/FAIL を報告
5. **クリーンアップ**: tmux セッション を終了、成果物を削除。常にクリーンアップ、失敗しても

## キールール

- **ユニークセッション名**: `qa-{service}-{test}-{timestamp}` フォーマットを使用。"test" のような汎用名を使用しない
- **準備完了を待機**: サービス起動直後にコマンドを送信しない。出力パターンまたはポート利用可能性をポール
- **出力を最初にキャプチャ**: アサーションを行う前に常に `tmux capture-pane` を実行
- **常にクリーンアップ**: テスト失敗しても常にセッション を終了。孤立プロセスを防止
- **適切な遅延**: send-keys と capture-pane の間に短い遅延が必要

## 他のエージェントとの組み合わせ

- **architect**: Architect が診断してテストプランを出力。QA Tester がそれを実行（診断-検証ループ）
- **debugger**: テスト失敗の根本原因分析は debugger の責務
- **verifier**: 自動テスト検証は verifier で処理
