## 概要

Test Engineer はテスト戦略を設計し、テストを書き、不安定なテストを強化し、
TDD ワークフローをガイドするエージェントです。

テストは期待される動作の実行可能なドキュメントです。
テストされていないコードは負債であり、不安定なテストはテストスイートへの信頼を低下させます。

## 使用時機

- テスト戦略を確立してカバレッジギャップを分析するとき
- TDD ワークフローを実行するとき
- 不安定なテストの根本原因を診断して修正するとき
- 新しい機能のテストを書くとき

## 使用例

```
"Establish a test strategy for this module"
"Implement password validation with tdd"  # magic keyword activates TDD mode
"Fix this flaky test"
```

## テストピラミッド

Test Engineer はテストピラミッド原則に従います。

| レイヤー | 比率 | 説明 |
|------|------|------|
| ユニット | 70% | 個々の関数/モジュールの動作を検証 |
| 統合 | 20% | コンポーネント間のインタラクションを検証 |
| E2E | 10% | 全体的なシステム動作を検証 |

## TDD プロセス（鉄則）

**本番コードを書く前に、失敗するテストが存在する必要があります。**

1. **RED**: 次の機能のテストを書く。実行 — 失敗する必要があります。合格すれば、テストが間違っています
2. **GREEN**: テストを合格させるための最小限のコードを書く。余分な機能なし。実行 — 合格する必要があります
3. **REFACTOR**: コード品質を向上。すべての変更でテストを実行。グリーンを保ちます
4. **REPEAT**: 次の失敗するテストに進む

コードを最初に書きましたか？削除して、テストからやり直します。

## 不安定なテスト診断

不安定なテストの一般的な原因と修正：

| 原因 | 修正 |
|------|------|
| タイミング依存 | `waitFor` を使用、ハードコードされたスリープを削除 |
| 共有状態 | `beforeEach` で状態をリセット |
| ハードコードされた日付 | 相対日付を使用 |
| 環境依存 | コンテナ/モックを使用 |

## 他のエージェントとの組み合わせ

- **executor**: 機能実装は executor の仕事。テスト記述は test-engineer の仕事
- **verifier**: テスト妥当性評価と検証は verifier で処理
- **debugger**: テスト失敗の根本原因分析のため debugger と協力
