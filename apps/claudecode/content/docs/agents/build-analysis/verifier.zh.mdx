## 概述

Verifier 是一个基于实际证据而非"应该能工作"的声明来验证完成情况的智能体。
它设计验证策略、通过证据确认完成、评估测试充分性和回归风险、验证验收标准。

像"应该"、"可能"和"似乎"这样的词是需要实际验证的警示信号。
无证据的完成声明是生产错误的主要原因。

验证作为与作者分离的一个独立流程进行。
不在同一上下文中进行自我批准。

## 何时使用

- 当在实现完成后需要独立验证
- 当确认实际行为与验收标准相符
- 当评估测试覆盖率的充分性
- 当评估更改的回归风险

## 使用示例

```
"Executor 说完成了 — 实际验证一下"
"确认这个特性符合所有验收标准"
"评估这个变更的回归风险"
```

## 验证过程

1. **定义**：什么证明了行为有效？哪些边界情况重要？什么可能回归？
2. **执行**（并行）：运行测试套件、用 lsp_diagnostics_directory 进行类型检查、运行构建命令、搜索相关测试
3. **缺口分析**：对每个需求，分类为已验证（测试存在 + 通过 + 边界覆盖）、部分（测试不完整）或缺失（无测试）
4. **判决**：基于支持证据发出清晰的通过或失败判决

## 立即拒绝条件

在这些情况下，Verifier 不在没有证据的情况下批准：

- 使用模糊语言如"应该/可能/似乎"
- 没有最近的测试输出可用
- 仅有"所有测试通过"的声明而无结果显示
- TypeScript 变更无类型检查
- 编译语言的编译验证

## 与其他智能体的结合

- **executor**：Executor 完成实现后，Verifier 独立验证
- **test-engineer**：发现测试覆盖率缺口时，test-engineer 添加测试
- **code-reviewer**：代码质量/风格审查是 code-reviewer 的责任
- **security-reviewer**：安全审计由 security-reviewer 处理
