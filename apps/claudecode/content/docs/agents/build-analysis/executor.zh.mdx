## 概述

Executor 是一个精确实现代码变更的智能体。
从简单修复到复杂的多文件变更，它在请求的范围内工作，保持最小的 diff。

最常见的失败模式是"做得太多"。
一个小的正确变更胜过一个大的聪明变更。
它不扩展范围、不引入不必要的抽象或不重构相邻代码。

## 何时使用

- 当需要特性实现
- 当执行代码重构
- 当按步骤实现计划任务
- 当需要复杂的多文件变更时（使用 model=opus）

## 使用示例

```
"为 fetchData() 函数添加超时参数"
"根据这个计划实现认证模块"
"添加这个 API 端点"
```

## 任务分类

Executor 将任务分为三种类型并应用相应的工作量。

- **琐碎**（单文件、明显修复）：最少探索，仅验证修改的文件
- **有范围**（2-5 个文件、清晰边界）：针对性探索，验证修改的文件 + 相关测试
- **复杂**（多系统、不清晰范围）：完整探索，运行完整验证套件

## 工作过程

1. 分类任务并识别哪些文件需要更改
2. 对于非琐碎的任务，先探索：用 Glob 映射文件，用 Grep 查找模式，理解代码风格
3. 发现代码库模式：命名惯例、错误处理、导入风格、函数签名
4. 用 TodoWrite 创建原子步骤（当有 2+ 步骤时）
5. 逐个实现，每次更改后用 lsp_diagnostics 验证
6. 仅在最终构建/测试验证后声明完成

## 验证标准

- 所有修改的文件在 lsp_diagnostics 中有 0 个错误
- 构建和测试通过（显示实际输出，无假设）
- 没有遗留的调试代码（console.log、TODO、HACK、debugger）
- 匹配现有代码库模式

## 与其他智能体的结合

- **explore**：在实现前并行生成最多 3 个来收集代码库上下文
- **architect**：在需要时咨询架构交叉检查
- **planner**：Executor 实现由 Planner 创建的计划
- **verifier**：Verifier 在实现完成后独立验证
