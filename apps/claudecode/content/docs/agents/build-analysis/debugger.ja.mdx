## 概要

Debugger は、バグを根本原因にトレースし、最小限の修正を提案するエージェントです。
ビルド失敗の解決も、最小限の変更で処理します。

核となる原則は「症状ではなく原因を修正する」ことです。
「undefined」になっているのはなぜか？を追跡するのであって、どこにでも null チェックを追加するわけではありません。
修正は常に最小限の diff であり、リファクタリングやアーキテクチャ変更は含みません。

## 使用時期

- ランタイム エラーの根本原因を見つける場合
- ビルド/コンパイル エラー（型エラー、インポート エラーなど）を修正する場合
- リグレッションを分離する場合
- スタック トレースを分析する場合

## 使用例

```
"TypeError: Cannot read property 'name' of undefined エラーをトレースする"
"ビルド エラーを修正する"
"このテストが急に失敗し始めたのはなぜか見つけよ"
```

## ランタイム バグ調査プロトコル

1. **再現**: バグが確実にトリガーされることを確認
2. **証拠収集**（並列）: 完全なエラー メッセージ/スタック トレースを読む、git log/blame で最近の変更を確認、同様に動作するコードを見つける
3. **仮説形成**: 壊れたコード vs 動作するコードを比較、入力からエラーまでデータフローをトレース
4. **修正**: 一度に 1 つの変更だけを提案。同じパターンが他の場所に存在しないかチェック
5. **サーキット ブレーカー**: 3 つの失敗した仮説後、停止して Architect にエスカレート

## ビルド エラー解決プロトコル

1. マニフェスト ファイルからプロジェクト タイプを検出
2. `lsp_diagnostics_directory` またはビルド コマンドを使用してすべてのエラーを収集
3. エラーを分類: 型推論、定義の不足、インポート/エクスポート、設定
4. 各エラーを最小限の変更で修正: 型注釈、null チェック、インポート修正
5. 修正後の検証: 変更されたファイルで lsp_diagnostics を実行
6. 最終検証: ビルド コマンドが終了コード 0 で終了することを確認
7. 進捗追跡: 「X/Y エラーを修正」を報告

## 他のエージェントとの組み合わせ

- **architect**: 3 つの失敗した仮説後、architect レベルの分析にエスカレート
- **executor**: Debugger が原因を見つけたら、Executor が修正を実装
- **test-engineer**: 修正後のリグレッション テスト作成は test-engineer の責任
- **verifier**: Verifier が修正が実際に問題を解決したことを検証
