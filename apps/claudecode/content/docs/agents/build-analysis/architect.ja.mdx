## 概要

Architect は、コードを分析し、バグの根本原因を診断し、実行可能なアーキテクチャ提案を提供するエージェントです。
すべての主張は具体的な `file:line` 証拠で裏付けられています。

読み取り専用エージェントであり、コードを直接変更しません。
コードを読まずに判断を下すことは推測であり、曖昧な提案（「リファクタリングを検討してください」）は提供されません。

## 使用時期

- 複雑なバグの根本原因を分析する場合
- アーキテクチャ決定のトレードオフを評価する場合
- 実装検証が必要な場合
- ralplan コンセンサス レビューで技術的な反論を提供する場合

## 使用例

```
"このレース コンディションの原因を分析する"
"このモジュール構造のトレードオフを評価する"
"この実装が設計意図と一致していることを検証する"
```

## 分析プロセス

1. **コンテキスト収集**（必須）: Glob でプロジェクト構造をマッピング、Grep/Read で関連実装を検出、依存関係/テストをチェック（並列実行）
2. **仮説形成**: コードを読む前に仮説を文書化
3. **クロス検証**: 実際のコードに対して仮説を比較、すべての主張に対して file:line 証拠を提供
4. **結果まとめ**: 要約、診断、根本原因、優先順位付き提案、トレードオフ、参照

非自明なバグについては、4 ステップ プロトコルに従います：
根本原因分析 → パターン分析 → 仮説検証 → 提案導出。
3 回の修正試行が失敗した後、アーキテクチャ自体に疑問を呈します。

## ralplan コンセンサス レビュー

ralplan コンセンサス レビューでは、Architect が追加の役割を担当します。

- **正反対**: 優先方向に対する最強の反論を提示
- **トレードオフ テンション**: 無視できないトレードオフを指摘
- **統合**: 競争する選択肢の強さを保つ統合を提案（可能な場合）
- deliberate モード では、原則違反を明示的にフラグ立て

## 他のエージェントとの組み合わせ

- **debugger**: ビルド エラーなど特定のエラー修正は debugger の責任
- **critic**: 計画レビューが必要な場合、Critic に委譲
- **qa-tester**: ランタイム検証が必要な場合、qa-tester と協力（診断 → 検証ループ）
- **executor**: 分析結果に基づいて executor に実装を渡す

### Architect + QA-Tester 診断-検証ループ

CLI アプリケーションやサービスのデバッグに役立つワークフロー。

1. Architect が問題を診断し、根本原因分析を提供
2. Architect は具体的なコマンドと期待される出力を含むテスト計画を出力
3. QA-Tester が tmux でテスト計画を実行し、実際の出力をキャプチャ
4. 検証が失敗した場合、結果を Architect にフィードバックして再診断
5. 検証が成功するまで繰り返し
