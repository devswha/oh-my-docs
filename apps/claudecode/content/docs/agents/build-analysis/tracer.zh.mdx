## 概述

Tracer 是一个通过有纪律的、基于证据的因果追踪来解释观察结果的智能体。
它将观察与解释分离，生成竞争假说，收集赞成和反对每个假说的证据，
按证据强度排列解释，并推荐最快减少不确定性的下一步探查。

区别于普通调试的是，它不从症状直接跳到首选解释。
它维持替代解释直到证据消除它们，并明确处理不确定性。

## 何时使用

- 当追踪复杂现象的原因
- 当有多个可能的原因且不清楚哪个是正确的
- 当需要深度分析"为什么会这样？"
- 当调查原因不明的问题，如性能回归或间歇性失败

## 使用示例

```
"追踪为什么这个基准性能下降了 25%"
"分析为什么工作进程分配停滞"
"调查为什么这个测试间歇性失败"
```

## 追踪协议

1. **观察**：准确描述观察的结果，不带解释
2. **框架**：定义追踪目标 — 正在回答哪个具体的"为什么"问题
3. **假说**：故意从不同框架（代码路径、配置/环境、测量误差、架构假设不匹配等）生成竞争因果解释
4. **收集证据**：为每个假说收集赞成和反对证据。引用特定的 file:line 证据
5. **应用视角**：应用系统视角（边界、重试、反馈循环）、预测视角和科学视角
6. **反驳**：运行反驳轮，其中最强的替代方案挑战当前领导者
7. **排列/收敛**：降低与证据矛盾或需要额外假设的解释
8. **综合**：陈述当前最佳解释以及为什么它优于替代方案
9. **探查**：推荐下一步调查，它将最快地减少不确定性

## 证据强度层级

Tracer 按强度顺序评估证据。

1. 受控复现、直接实验、唯一区别假说的真实来源
2. 直接证据：带时间戳的日志、追踪事件、指标、配置快照、git 历史
3. 多个独立来源汇聚到相同解释
4. 单一来源推论，符合观察但尚未唯一区别
5. 弱的间接线索（命名、时间接近、堆栈位置）
6. 直觉/类比/猜测

## 与其他智能体的结合

- **debugger**：简单的错误修复是 debugger 的责任。原因不明的复杂现象由 tracer 处理
- **architect**：需要架构级别分析时与 architect 协作
- **scientist**：结合 scientist 进行需要数据分析的追踪
