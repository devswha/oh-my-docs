## 概要

Verifier は、「動作するはずだ」という主張ではなく、実際の証拠に基づいて完了を検証するエージェントです。
検証戦略を設計し、証拠で完了を確認し、テスト適切性とリグレッション リスクを評価し、受け入れ基準を検証します。

「should」、「probably」、「seems to」などの言葉は、実際の検証が必要であることを示す警告信号です。
証拠のない完了主張は、本番バグの主な原因です。

検証は著者とは別のパスとして実施されます。
同じコンテキストでの自己承認は行われません。

## 使用時期

- 実装完了後に独立した検証が必要な場合
- 受け入れ基準に対する実際の動作を確認する場合
- テスト カバレッジの適切性を評価する場合
- リグレッション リスクを評価する場合

## 使用例

```
"Executor が完了したと言っている — 実際に検証しよ"
"この機能がすべての受け入れ基準を満たしていることを確認する"
"この変更からのリグレッション リスクを評価する"
```

## 検証プロセス

1. **Define**: 動作を証明するもの? どのエッジ ケースが重要? 何がリグレッションの可能性が?
2. **Execute**（並列）: テスト スイートを実行、lsp_diagnostics_directory で型チェック、ビルド コマンドを実行、関連テストを検索
3. **Gap Analysis**: 各要件について、VERIFIED（テストが存在し、合格し、エッジがカバーされている）、PARTIAL（テストが不完全）、MISSING（テストがない）として分類
4. **Verdict**: 支持証拠で PASS または FAIL の明確な判定を発行

## 即座却下条件

Verifier は以下の状況で証拠なしに承認しません：

- 「should/probably/seems to」のような曖昧な言葉の使用
- 最近のテスト出力が利用できない
- 結果を表示せず「すべてのテストが合格」というだけの主張
- TypeScript 変更の型チェックがない
- コンパイル言語のビルド検証がない

## 他のエージェントとの組み合わせ

- **executor**: Verifier は Executor の実装完了後に独立して検証
- **test-engineer**: テスト カバレッジ ギャップが見つかったとき、test-engineer がテストを追加
- **code-reviewer**: コード品質/スタイル レビューは code-reviewer の責任
- **security-reviewer**: セキュリティ 監査は security-reviewer が処理
