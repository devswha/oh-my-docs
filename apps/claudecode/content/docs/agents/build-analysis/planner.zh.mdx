## 概述

Planner 是一个进行结构化用户访谈并生成可行任务计划的智能体。
当给定"执行 X"这样的请求时，它不直接实现 — 而是创建"X 的任务计划"。

有两个核心原则：
1. 仅询问用户关于偏好/优先级的问题；用 explore 智能体直接验证代码库事实
2. 生成可行的计划，包含 3-6 个步骤（既不是 30 个微步骤也不是 2 个模糊指令）

计划保存到 `.omc/plans/{name}.md`。

## 何时使用

- 当需要新特性或项目的执行计划
- 当组织重构或迁移的步骤
- 当理解复杂任务的顺序和依赖关系
- 当运行 ralplan（基于共识的规划）

## 使用示例

```
"为添加深色模式创建计划"
"为这个模块制作重构计划"
"使用 ralplan 为重新设计认证系统创建计划"
```

## 规划过程

1. **意图分类**：确定这是简单修复/重构/新构建/中等规模任务
2. **代码库调查**：用 explore 智能体理解现有模式和结构
3. **用户访谈**：一次问一个关于偏好、时间表和范围决策的问题
4. **分析师咨询**：在生成计划前分析需求缺口
5. **计划生成**：包括上下文、目标、护栏、任务工作流、详细的 TODO 和成功标准
6. **用户确认**：仅在收到明确批准后转到执行阶段

## 共识模式（ralplan）

用 `/plan --consensus` 或 `/ralplan` 运行激活基于共识的规划模式。

- **RALPLAN-DR 摘要**：原则（3-5 个）、决策驱动因素（前 3 个）、可行选项（2+ 个）
- **包含 ADR**：决策、驱动因素、考虑的替代方案、选择原因、后果、后续行动
- `--deliberate` 标志：为高风险工作添加前期预测和扩展测试规划

## 与其他智能体的结合

- **explore**：自动生成来验证代码库事实
- **analyst**：在计划生成前咨询需求缺口分析
- **critic**：审查生成的计划的质量
- **executor**：计划批准后处理实现
