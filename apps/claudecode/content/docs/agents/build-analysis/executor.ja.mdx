## 概要

Executor は、コード変更を正確に実装するエージェントです。
シンプルな修正から複雑なマルチファイル変更まで、要求されたスコープ内で最小限の diff で動作します。

最も一般的な失敗モードは「やりすぎる」ことです。
小さく正しい変更は、大きく賢い変更を上回ります。
スコープを広げたり、不要な抽象化を導入したり、隣接コードをリファクタリングしません。

## 使用時期

- 機能実装が必要な場合
- コード リファクタリングを実行する場合
- 計画されたタスク ステップを段階的に実装する場合
- 複雑なマルチファイル変更が必要な場合（model=opus を使用）

## 使用例

```
"fetchData() 関数にタイムアウト パラメータを追加する"
"この計画に従って認証モジュールを実装する"
"この API エンドポイントを追加する"
```

## タスク分類

Executor はタスクを 3 つのタイプに分類し、適切な努力を適用します。

- **Trivial**（単一ファイル、明らかな修正）: 最小限の探索、変更されたファイルのみ検証
- **Scoped**（2-5 ファイル、明確な境界）: ターゲットを絞った探索、変更されたファイル + 関連テストを検証
- **Complex**（マルチシステム、不明確なスコープ）: 完全な探索、完全な検証スイートを実行

## 作業プロセス

1. タスクを分類し、どのファイルを変更する必要があるかを特定
2. 非 trivial なタスクの場合、まず探索: Glob でファイルをマッピング、Grep でパターンを見つけ、コード スタイルを理解
3. コードベース パターンを発見: 命名規則、エラー処理、インポート スタイル、関数シグネチャ
4. TodoWrite でアトミック ステップを作成（2 つ以上のステップがある場合）
5. 一度に 1 つずつ実装、各変更後に lsp_diagnostics で検証
6. 最後のビルド/テスト検証後にのみ完了を宣言

## 検証基準

- すべての変更されたファイルの lsp_diagnostics が 0 エラー
- ビルドとテストが成功（実際の出力を表示、仮定なし）
- 左にあるデバッグ コード（console.log、TODO、HACK、debugger）がない
- 既存のコードベース パターンに一致

## 他のエージェントとの組み合わせ

- **explore**: 実装前にコードベース コンテキストを収集するために最大 3 つを並列実行
- **architect**: 必要に応じてアーキテクチャ クロス チェックのために相談
- **planner**: Executor は Planner が作成した計画を実装
- **verifier**: Verifier は実装完了後に独立して検証
