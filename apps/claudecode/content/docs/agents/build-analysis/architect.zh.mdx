## 概述

Architect 是一个分析代码、诊断错误根本原因并提供可行架构建议的智能体。
所有声明都由特定的 `file:line` 证据支持。

它是一个只读智能体，不直接修改代码。
未读代码就做出判断是猜测，不提供模糊建议（如"考虑重构"）。

## 何时使用

- 当分析复杂错误的根本原因
- 当评估架构决策的权衡
- 当需要实现验证
- 当在 ralplan 共识审查中提供技术反驳论点

## 使用示例

```
"分析这个竞态条件的原因"
"评估这个模块结构的权衡"
"验证这个实现是否符合设计意图"
```

## 分析过程

1. **上下文收集**（必需）：用 Glob 映射项目结构，用 Grep/Read 查找相关实现，检查依赖项/测试（并行运行）
2. **假说形成**：在读代码前记录假说
3. **交叉验证**：将假说与实际代码对比，为所有声明提供 file:line 证据
4. **结果总结**：摘要、诊断、根本原因、优先级建议、权衡、参考

对于非明显的错误，遵循 4 步协议：
根本原因分析 → 模式分析 → 假说验证 → 建议推导。
在 3 次失败的修复尝试后，质疑架构本身。

## ralplan 共识审查

在 ralplan 共识审查中，Architect 承担额外角色。

- **反驳**：针对首选方向提出最强的反驳论证
- **权衡张力**：指出不能忽视的权衡
- **综合**：提出一个综合方案，保留竞争选项的优势（如果可能）
- 在深思熟虑模式中，明确标记原则违反

## 与其他智能体的结合

- **debugger**：特定的错误修复（如构建错误）是 debugger 的责任
- **critic**：需要计划审查时委托给 critic
- **qa-tester**：需要运行时验证时与 qa-tester 协作（诊断 → 验证循环）
- **executor**：根据分析结果将实现传给 executor

### Architect + QA-Tester 诊断-验证循环

一个对调试 CLI 应用或服务有用的工作流。

1. Architect 诊断问题并提供根本原因分析
2. Architect 输出测试计划，包含特定命令和预期输出
3. QA-Tester 在 tmux 中执行测试计划并捕获实际输出
4. 若验证失败，将结果反馈给 Architect 进行重新诊断
5. 重复直到验证成功
