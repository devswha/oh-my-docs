## 概要

Tracer は、観察された結果を規律のある証拠ベースの因果トレースで説明するエージェントです。
観察と解釈を分離し、競合仮説を生成し、各仮説の支持証拠と反対証拠を収集し、
証拠の強さで説明をランク付けし、最も早く不確実性を低減する次のプローブを推奨します。

通常のデバッグとの違いは、症状から優先される説明に直接ジャンプしないことです。
証拠が代替説明を排除するまで、代替説明を維持し、不確実性を明示的に処理します。

## 使用時期

- 複雑な現象の原因をトレースする場合
- 複数の可能な原因があり、どちらが正しいか不明確な場合
- 「なぜこれが起きたのか?」の深い分析が必要な場合
- 原因が不明な問題（パフォーマンス リグレッション、断続的な障害など）を調査する場合

## 使用例

```
"このベンチマーク パフォーマンスが 25% 低下したのはなぜかをトレースする"
"ワーカー割り当てがスタールしているのはなぜかを分析する"
"このテストが断続的に失敗するのはなぜかを調査する"
```

## トレース プロトコル

1. **OBSERVE**: 観察された結果を解釈せずに正確に説明
2. **FRAME**: トレース対象を定義 — どの具体的な「なぜ」の質問に答えるか
3. **HYPOTHESIZE**: 異なるフレーム（コード パス、設定/環境、測定エラー、アーキテクチャ前提の不一致など）から意図的に競合する因果説明を生成
4. **GATHER EVIDENCE**: 各仮説の支持証拠と反対証拠を収集。具体的な file:line 証拠を引用
5. **APPLY LENSES**: システム レンズ（境界、リトライ、フィードバック ループ）、プリモーテム レンズ、科学的レンズを適用
6. **REBUT**: リバッタル ラウンドを実行し、最強の代替案が現在のリーダーに異議を唱える
7. **RANK/CONVERGE**: 証拠と矛盾するか追加の前提が必要な説明をダウンランク
8. **SYNTHESIZE**: 現在最良の説明とそれが代替案より優れている理由を述べる
9. **PROBE**: 最も不確実性を低減する次の調査を推奨

## 証拠強度階層

Tracer は証拠を強度の順序で評価します。

1. 制御された再現、直接実験、仮説を一意に区別する真実のソース
2. 直接証拠: タイムスタンプ付きログ、トレース イベント、メトリクス、設定スナップショット、git 履歴
3. 同じ説明に収束する複数の独立したソース
4. 観察に適合するが仮説をまだ一意に区別しない単一ソース推論
5. 弱い傍証の手がかり（命名、時間的近接性、スタック位置）
6. 直感/類推/推測

## 他のエージェントとの組み合わせ

- **debugger**: シンプルなバグ修正は debugger の責任。原因が不明な複雑な現象は tracer が処理
- **architect**: アーキテクチャ レベルの分析が必要な場合、Architect と協力
- **scientist**: データ分析が必要なトレースの場合、Scientist と組み合わせ
