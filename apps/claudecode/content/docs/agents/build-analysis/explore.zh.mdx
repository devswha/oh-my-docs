## 概述

Explore 是一个发现专家智能体，快速查找代码库中的文件、代码模式和符号之间的关系。
其默认模型是 haiku，使其快速且便宜，专门查找"东西在哪里"。

Explore 是一个只读智能体。
它不创建或修改文件，以消息文本形式返回搜索结果。

它处理内部代码库探索。
外部文档、学术论文和 API 参考的查找由 `document-specialist` 处理。

## 何时使用

- 当需要查找实现特定特性的文件
- 当追踪代码模式或符号的使用位置
- 当理解项目结构和文件之间的关系
- 当其他智能体（planner、executor 等）需要在开始工作前收集代码库上下文

## 使用示例

```
"找出认证相关代码在哪里"
"查找所有调用这个函数的文件"
"映射项目的路由结构"
```

Explore 从多个角度并行运行搜索，而不是单一搜索。
例如，当查找"认证相关代码"时，它同时搜索控制器、中间件、令牌验证、会话管理等。

## 使用的工具

- **Glob**：按名称/模式查找文件
- **Grep**：按文本模式搜索代码内容
- **ast_grep_search**：搜索结构模式，如函数形状和类结构
- **lsp_document_symbols**：查看文件的符号概览（函数、类、变量）
- **lsp_workspace_symbols**：在整个工作区搜索符号名称
- **Bash**：用 git 命令查询历史/更改记录

## 上下文保守策略

Explore 保守地使用上下文窗口空间。

- 对于超过 200 行的文件，它首先用 `lsp_document_symbols` 检查概览
- 对于超过 500 行的文件，它优先使用结构工具而不是读整个文件
- 对于大文件，它仅用 `offset`/`limit` 参数读取所需的部分
- 并行读取限制为一次 5 个文件

## 与其他智能体的结合

- **planner**：先运行 Explore 来收集代码库上下文，然后创建计划
- **executor**：生成 explore 智能体来理解现有模式和依赖项，然后实现
- **architect**：在架构分析初期用于文件映射
- **debugger**：用于快速定位与错误相关的代码
