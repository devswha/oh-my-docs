## 概述

Debugger 是一个追踪错误到根本原因并提出最小修复的智能体。
它也处理以最小更改解决构建失败。

核心原则是"修复原因，而不是症状"。
与其到处添加 null 检查，不如追踪"为什么这个是未定义的？"
修复始终是最小的 diff，不涉及重构或架构改变。

## 何时使用

- 当查找运行时错误的根本原因
- 当修复构建/编译错误（类型错误、导入错误等）
- 当隔离回归
- 当分析堆栈跟踪

## 使用示例

```
"追踪 TypeError: Cannot read property 'name' of undefined 错误"
"修复构建错误"
"找出为什么这个测试突然开始失败"
```

## 运行时错误调查协议

1. **复现**：确认错误可以可靠地触发
2. **收集证据**（并行）：读取完整错误消息/堆栈跟踪，用 git log/blame 检查最近的更改，查找类似的工作代码
3. **形成假说**：比较损坏代码与工作代码，追踪从输入到错误的数据流
4. **修复**：每次只提出一个更改。检查相同模式是否在其他地方存在
5. **断路器**：在 3 次失败的假说后停止并升级给 architect

## 构建错误解决协议

1. 从清单文件检测项目类型
2. 用 `lsp_diagnostics_directory` 或构建命令收集所有错误
3. 分类错误：类型推断、缺失定义、导入/导出、配置
4. 用最小更改修复每个错误：类型注解、null 检查、导入修复
5. 验证后修复：在修改的文件上运行 lsp_diagnostics
6. 最终验证：确认构建命令以代码 0 退出
7. 跟踪进度：报告"X/Y 错误已修复"

## 与其他智能体的结合

- **architect**：在 3 次失败的假说后升级给 architect 级别的分析
- **executor**：一旦 debugger 找到原因，executor 实现修复
- **test-engineer**：修复后编写回归测试是 test-engineer 的责任
- **verifier**：验证修复是否真的解决了问题
