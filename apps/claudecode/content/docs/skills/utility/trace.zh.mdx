## 概述

Trace 是一个处理模糊的、因果的、证据关键问题如"为什么会发生这个结果？"的技能。它陈述观察，生成竞争解释，并行收集证据，对解释进行排名，并建议最可能减少不确定性的下一步调查。

与简单的调试摘要或代码修复循环不同，它通过假设间的竞争找到最令人信服的解释。

## 使用

```bash
/oh-my-claudecode:trace "why is the API returning 500 errors intermittently?"
```

## 好的用例

- 运行时错误和回归
- 性能 / 延迟 / 资源行为分析
- 架构前/事后评估
- 实验结果追踪
- 解释配置 / 路由 / 编排行为

## 核心 Trace 约定

始终维护以下区分：

1. **观察**：实际观察到的内容
2. **假设**：竞争解释
3. **支持证据**：支持每个解释的内容
4. **反对证据 / 缺口**：矛盾或缺失的内容
5. **当前最佳解释**：当前领先的解释
6. **关键未知**：区分最佳假设的缺失事实
7. **区分探针**：最高价值的下一步以减少不确定性

## 默认假设通道

3 个追踪通道在团队模式中并行运行：

1. **代码路径 / 实现原因**：在代码行为中追踪原因
2. **配置 / 环境 / 编排原因**：在基础设施/配置中追踪原因
3. **测量 / 工件 / 假设不匹配原因**：追踪观察本身可能的错误

## 证据强度层级

证据排名（强度优先）：

1. 受控再现 / 直接实验 / 唯一区分工件
2. 来自严格来源的主要来源工件（日志、指标、git 历史）
3. 多个独立来源指向相同解释
4. 单个源代码路径或行为推理
5. 弱推测线索（时间、相似性）
6. 直觉 / 类比 / 推测

## 证伪轮次

在得出结论前，对领先假设进行证伪轮次：

- 非领先通道针对领导者提出最佳反驳
- 领导者用证据（不是断言）回应
- 如果反驳削弱领导者，排名调整
- 如果两个假设收敛于相同机制，它们合并
