## 概要

Trace は「この結果がなぜ起きたのか？」のような曖昧で因果的で証拠批判的な質問に対処するスキル。観察を言い換え、競合する説明を生成し、証拠を並列に集め、説明をランク付けし、不確実性を減らす可能性が最も高い次の調査を提案します。

単純なデバッグサマリーまたはコード修正ループとは異なり、仮説間の競争を通じて最も説得力のある説明を見つけます。

## 使用方法

```bash
/oh-my-claudecode:trace "why is the API returning 500 errors intermittently?"
```

## 良いユースケース

- ランタイムバグとリグレッション
- パフォーマンス / レイテンシ / リソース動作分析
- アーキテクチャ事前/事後モーテム
- 実験結果トレーシング
- 設定 / ルーティング / オーケストレーション動作説明

## コア Trace コントラクト

常に以下の違いを維持：

1. **観察**：実際に観察されたこと
2. **仮説**：競合する説明
3. **証拠賛成**：各説明をサポートするもの
4. **証拠反対 / ギャップ**：矛盾または欠落しているもの
5. **現在の最良説明**：現在の主導的説明
6. **クリティカル不明**：トップ説明を区別する欠落したファクト
7. **判別プローブ**：不確実性を減らすための最高価値の次のステップ

## デフォルト仮説レーン

team モードで 3 つのトレーサーレーンが並列実行：

1. **コードパス / 実装原因**：コード動作の原因をトレース
2. **設定 / 環境 / オーケストレーション原因**：インフラ/設定の原因をトレース
3. **測定 / アーティファクト / 仮定ミスマッチ原因**：観察自体の可能なエラーをトレース

## 証拠強度階級

証拠は（最強順）ランク付け：

1. 制御された再現 / 直接実験 / 一意に判別的なアーティファクト
2. 厳密なプロベナンスの一次ソースアーティファクト（ログ、メトリクス、git 履歴）
3. 同じ説明を指す複数の独立したソース
4. 単一のソースコードパスまたは動作推論
5. 弱い状況的な手がかり（タイミング、類似性）
6. 直感 / 類推 / 推測

## 反証ラウンド

結論前に、主導的仮説に対する反証ラウンドを実行：

- 非主導レーンが主導に対する最良の反論を提示
- 主導が証拠で応答（主張ではなく）
- 反論が主導を弱めると、ランキング調整
- 2 つの仮説が同じメカニズムに収束すると、マージ
