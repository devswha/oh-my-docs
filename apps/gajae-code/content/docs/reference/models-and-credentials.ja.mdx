Gajae-Code（`gjc`）は各具体プロバイダーモデルをすべて保持し、その上に正規（canonical）レイヤーを構築します。これにより正確な `provider/model` を固定することも、複数プロバイダーをまたいで統合される正規 id を選ぶこともできます。これらはすべて 1 つのファイル — `~/.gjc/agent/models.yml` — と認証解決・ランタイム検出で駆動します。

本ページは実用リファレンスで、スキーマ全体は貼りません。

## 設定ファイルの場所

```text
~/.gjc/agent/models.yml
```

レガシー動作：`models.yml` が無く同じ場所に `models.json` があれば `models.yml` へ移行されます。プログラムから渡す場合は明示的な `.json` / `.jsonc` パスも引き続きサポートされます。`models.yml` の検証に失敗すると、レジストリは組み込みモデルで動作を続け、エラーを UI に表示します。

## `models.yml` の構造

トップレベルは 3 セクション — `providers`、`equivalence`、`modelBindings` です。

```yaml
providers:
  <provider-id>:
    # プロバイダーレベル設定（baseUrl、api、auth、models ...）
equivalence:
  overrides:
    <provider-id>/<model-id>: <canonical-model-id>
  exclude:
    - <provider-id>/<model-id>
modelBindings:
  modelRoles:
    default: <provider/model-id または canonical-id>[:thinkingLevel]
  agentModelOverrides:
    executor: <provider/model-id>
```

`models.yml` は **厳格（strict）** です。未知の provider/model キーはディスパッチ前の検証で失敗するため、古いキーは削除してください。

### プロバイダーレベルのフィールド

```yaml
providers:
  my-provider:
    baseUrl: https://api.example.com/v1
    apiKey: MY_PROVIDER_API_KEY
    api: openai-completions
    auth: apiKey
    disableStrictTools: false  # `strict` を拒否する Anthropic 互換エンドポイントは true
    models:
      - id: some-model-id
        input: [text]
        contextWindow: 128000
        maxTokens: 16384
```

許可される `api` 値：`openai-completions`、`openai-responses`、`openai-codex-responses`、`azure-openai-responses`、`bedrock-converse-stream`、`anthropic-messages`、`google-generative-ai`、`google-vertex`、`google-gemini-cli`、`ollama-chat`、`cursor-agent`。`auth` 値：`apiKey`（既定）、`none`、`oauth`。

検証：完全なカスタムプロバイダーは `baseUrl`、`api`、`auth: none` でなければ `apiKey` が必要です。`id` は必須、`contextWindow`/`maxTokens` は指定時に正の値であること。

## プロバイダープリセット

よくある MiniMax・GLM/zAI 設定はプリセット推奨です。

```sh
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

TUI 内でも同じプリセットが使えます。

```text
/provider add --preset minimax
/provider add --preset glm
/provider add zai
```

プリセットは文書化された環境変数名（`MINIMAX_CODE_API_KEY`、`MINIMAX_CODE_CN_API_KEY`、`ZAI_API_KEY`）を参照する `models.yml` エントリのみを書き込み、実際の認証情報は保存・検証しません。GLM のエイリアス（`glm`、`zai`、`z-ai`）は `glm-proxy` という OpenAI 互換カスタムプロバイダーを書き込み、ファーストクラスの `zai` プロバイダーは置き換えません。

## モデルロールと思考レベル

対応ロール：`default`、`smol`、`slow`、`vision`、`plan`、`designer`、`commit`、`task`。

ロール値は `provider/modelId`（具体バリアントを固定）または正規 id（プロバイダー統合を許可）で、思考レベルのサフィックス `:off | :minimal | :low | :medium | :high | :xhigh` を付けられます（例：`default: layofflabs/gpt-5.5:high`）。

関連設定：`modelRoles`、`enabledModels`（パスプレフィックスでスコープ可）、`modelProviderOrder`、`providers.kimiApiFormat`、`providers.openaiWebsockets`。

## 正規等価

正規 id は公式アップストリーム id のみ（例：`anthropic-model-opus-4-6`、`gpt-5.3-openai-code`）。`equivalence` ブロックでグルーピングを設定します。

```yaml
equivalence:
  overrides:
    zenmux/openai-code: gpt-5.3-openai-code
  exclude:
    - demo/openai-code-preview
```

ビルド順：ユーザー `overrides` → 組み込み公式 id 一致 → 保守的ヒューリスティック正規化（`anthropic/`・`openai/` プレフィックス除去、`4.6 -> 4-6`）→ 自身の id へフォールバック。複数バリアントが同じ正規 id を共有する場合、可用性＋認証、`modelProviderOrder`、レジストリ順で解決し、無効・未認証プロバイダーはスキップします。セッションとトランスクリプトは常に実際に実行した具体 provider/model を記録します。

## ランタイム検出

明示設定しない場合、ローカルランタイム向けに鍵不要（keyless）の検出プロバイダーが暗黙追加されます。

| プロバイダー | 既定 base URL（環境変数で上書き） | API |
| --- | --- | --- |
| `ollama` | `http://127.0.0.1:11434`（`OLLAMA_BASE_URL`） | `openai-responses` |
| `llama.cpp` | `http://127.0.0.1:8080`（`LLAMA_CPP_BASE_URL`） | `openai-responses` |
| `lm-studio` | `http://127.0.0.1:1234/v1`（`LM_STUDIO_BASE_URL`） | `openai-completions` |

`discovery.type` は `ollama`、`llama.cpp`、`lm-studio` を受け付け、プロバイダーレベルの `api` が必要です。

## 認証と API キー解決順序

<Steps>
  <Step>ランタイム上書き（CLI `--api-key`）。</Step>
  <Step>`agent.db` に保存された API キー認証情報。</Step>
  <Step>`agent.db` に保存された OAuth 認証情報（リフレッシュ込み）。</Step>
  <Step>環境変数マッピング（`OPENAI_API_KEY`、`ANTHROPIC_API_KEY` など）。</Step>
  <Step>ModelRegistry フォールバックリゾルバー（`models.yml` の provider `apiKey`）。</Step>
</Steps>

`models.yml` の `apiKey` 値はまず環境変数名として扱われ、該当変数が無ければリテラル文字列をトークンとして使います。`authHeader: true` かつ `apiKey` 設定時は `Authorization: Bearer <resolved-key>` を注入します。`auth: none` は認証情報なしで利用可能とみなされます。`GJC_AUTH_BROKER_URL`（または `auth.broker.url`）を設定すると、ローカル SQLite 認証ストアがリモート broker に置き換わります。
