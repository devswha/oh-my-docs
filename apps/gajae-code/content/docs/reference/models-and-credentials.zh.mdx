Gajae-Code（`gjc`）保留每个具体的 provider 模型，并在其上构建一个规范（canonical）层，因此你既能固定某个精确的 `provider/model`，也能选择一个跨 provider 合并的规范 id。这一切都由一个文件 —— `~/.gjc/agent/models.yml` —— 加上鉴权解析与运行时发现来驱动。

本页是实用参考，不粘贴完整 schema。

## 配置文件位置

```text
~/.gjc/agent/models.yml
```

遗留行为：若缺少 `models.yml` 而同位置存在 `models.json`，会迁移为 `models.yml`；以编程方式传入时仍支持显式 `.json` / `.jsonc` 路径。若 `models.yml` 校验失败，注册表会回退到内置模型并在 UI 中暴露错误。

## `models.yml` 结构

顶层有三段 —— `providers`、`equivalence`、`modelBindings`：

```yaml
providers:
  <provider-id>:
    # provider 级配置（baseUrl、api、auth、models ...）
equivalence:
  overrides:
    <provider-id>/<model-id>: <canonical-model-id>
  exclude:
    - <provider-id>/<model-id>
modelBindings:
  modelRoles:
    default: <provider/model-id 或 canonical-id>[:thinkingLevel]
  agentModelOverrides:
    executor: <provider/model-id>
```

`models.yml` 是**严格**的：未知 provider/model 键会在分发前校验失败，请删除过时键。

### Provider 级字段

```yaml
providers:
  my-provider:
    baseUrl: https://api.example.com/v1
    apiKey: MY_PROVIDER_API_KEY
    api: openai-completions
    auth: apiKey
    disableStrictTools: false  # 拒绝 `strict` 的 Anthropic 兼容端点设为 true
    models:
      - id: some-model-id
        input: [text]
        contextWindow: 128000
        maxTokens: 16384
```

允许的 `api` 值：`openai-completions`、`openai-responses`、`openai-codex-responses`、`azure-openai-responses`、`bedrock-converse-stream`、`anthropic-messages`、`google-generative-ai`、`google-vertex`、`google-gemini-cli`、`ollama-chat`、`cursor-agent`。`auth` 值：`apiKey`（默认）、`none`、`oauth`。

校验：完整自定义 provider 需要 `baseUrl`、`api`，以及非 `auth: none` 时的 `apiKey`；`id` 必填，`contextWindow`/`maxTokens` 若给定须为正数。

## Provider 预设

常见 MiniMax 与 GLM/zAI 建议用预设：

```sh
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

TUI 内同样可用：

```text
/provider add --preset minimax
/provider add --preset glm
/provider add zai
```

预设只写入引用文档化环境变量名（`MINIMAX_CODE_API_KEY`、`MINIMAX_CODE_CN_API_KEY`、`ZAI_API_KEY`）的条目，不存储或校验真实凭证。GLM 别名（`glm`、`zai`、`z-ai`）写入名为 `glm-proxy` 的 OpenAI 兼容自定义 provider，不替换一等的 `zai` provider。

## 模型角色与思考级别

支持的角色：`default`、`smol`、`slow`、`vision`、`plan`、`designer`、`commit`、`task`。

角色值可以是 `provider/modelId`（固定具体变体）或规范 id（允许跨 provider 合并），并可附加思考级别后缀：`:off | :minimal | :low | :medium | :high | :xhigh`，例如 `default: layofflabs/gpt-5.5:high`。

相关设置：`modelRoles`、`enabledModels`（可按路径前缀作用域）、`modelProviderOrder`、`providers.kimiApiFormat`、`providers.openaiWebsockets`。

## 规范等价

规范 id 仅为官方上游 id，如 `anthropic-model-opus-4-6`、`gpt-5.3-openai-code`。用 `equivalence` 配置分组：

```yaml
equivalence:
  overrides:
    zenmux/openai-code: gpt-5.3-openai-code
  exclude:
    - demo/openai-code-preview
```

分组顺序：用户 `overrides` → 内置官方 id 匹配 → 保守启发式归一化（剥离 `anthropic/`、`openai/` 前缀，`4.6 -> 4-6`）→ 回退到自身 id。多个变体共享同一规范 id 时，按可用性+鉴权、`modelProviderOrder`、再注册表顺序解析；跳过禁用/未鉴权 provider。会话与记录始终保存实际执行该轮的具体 provider/model。

## 运行时发现

未显式配置时自动添加无鉴权（keyless）的本地运行时 provider：

| Provider | 默认 base URL（环境变量覆盖） | API |
| --- | --- | --- |
| `ollama` | `http://127.0.0.1:11434`（`OLLAMA_BASE_URL`） | `openai-responses` |
| `llama.cpp` | `http://127.0.0.1:8080`（`LLAMA_CPP_BASE_URL`） | `openai-responses` |
| `lm-studio` | `http://127.0.0.1:1234/v1`（`LM_STUDIO_BASE_URL`） | `openai-completions` |

`discovery.type` 接受 `ollama`、`llama.cpp`、`lm-studio`，并需要 provider 级 `api`。

## 鉴权与 API 密钥解析顺序

<Steps>
  <Step>运行时覆盖（CLI `--api-key`）。</Step>
  <Step>`agent.db` 中存储的 API 密钥凭证。</Step>
  <Step>`agent.db` 中存储的 OAuth 凭证（含刷新）。</Step>
  <Step>环境变量映射（`OPENAI_API_KEY`、`ANTHROPIC_API_KEY` 等）。</Step>
  <Step>ModelRegistry 回退解析器（`models.yml` 的 provider `apiKey`）。</Step>
</Steps>

`models.yml` 的 `apiKey` 先按环境变量名解析；无对应变量则当作字面 token。`authHeader: true` 且设置 `apiKey` 时注入 `Authorization: Bearer <resolved-key>`。`auth: none` 视为无需凭证即可用。设置 `GJC_AUTH_BROKER_URL`（或 `auth.broker.url`）时，本地 SQLite 凭证库由远程 broker 替换。
