---

Gajae-Code(`gjc`)는 공개 표면을 의도적으로 작게 유지합니다. 워크플로 스킬 네 개(`deep-interview`, `ralplan`, `team`, `ultragoal`)와 역할 에이전트 네 개(`executor`, `architect`, `planner`, `critic`)가 전부입니다. 핵심은 작업에 맞는 가장 작은 워크플로를 고른 다음, 그 워크플로가 나머지 작업을 끝까지 끌고 가도록 두는 것입니다.

## 작업에 맞는 가장 작은 워크플로를 고릅니다

대부분의 작업은 전체 루프까지 필요하지 않습니다. 작업이 요구하는 만큼만 사다리를 올라가세요.

1. **직접 편집** — 파일 경로, 함수 이름, 수용 기준이 명확하고 위험이 낮은 작업입니다. 워크플로 없이 바로 처리합니다.
2. **`deep-interview`** — 의도, 범위, 수용 기준이 모호할 때 사용합니다. 소크라테스식 질문으로 계획이나 코드 변경 전에 모호함을 제거합니다.
3. **`ralplan`** — 계획은 세울 수 있지만 접근, 순서, 검증에 합의가 필요할 때 사용합니다. Planner/Architect/Critic 루프가 변경 전에 계획을 비평합니다.
4. **`ultragoal`** — 지속 가능하고 감사 가능한 다단계 goal에 사용합니다. 작업이 `.gjc/` 아래에 남는 goal 원장으로 나뉘고, 구현·수정·검증을 거쳐 진행됩니다.
5. **`team`** — *승인된* 작업이 병렬 tmux 워커로 이득을 볼 때만 사용합니다.

계획 산출물은 실행 경로를 명시적으로 승인하기 전까지 `pending approval` 상태로 남습니다. `deep-interview`와 `ralplan`은 스스로 구현을 시작하지 않습니다. 스펙이나 계획을 만들고 멈춥니다.

| 워크플로 | 언제 사용하나요 |
| --- | --- |
| 직접 편집 | 정확히 설명할 수 있는, 명확하고 위험이 낮은 변경. |
| `deep-interview` | 의도 / 범위 / 수용 기준이 모호함. |
| `ralplan` | 계획은 가능하지만 접근 / 순서 / 검증에 합의가 필요함. |
| `ultragoal` | 지속 원장을 가진 지속 가능하고 감사 가능한 goal. |
| `team` | 병렬 tmux 워커로 이득을 보는 승인된 작업. |

## 유스케이스

### 버그 수정

전형적인 전체 루프입니다. 명확히 하고, 계획하고, 끌고 가고, 검증합니다. 각 단계는 "코드를 좀 썼다"가 아니라 증거로 끝납니다.

- **언제 사용하나요:** 코드를 건드리기 전에 원인이나 수정 범위를 못 박아야 하는 실제 결함.
- **스킬 / 에이전트:** `deep-interview` → `ralplan` → `ultragoal`. `critic`이 계획을 검토하고 `executor`가 범위가 한정된 수정을 합니다. 병렬 작업에만 `team`을 더합니다.

<Steps>

<Step>

#### `deep-interview`로 버그를 명확히 합니다

계획에 들어가기 전에 버그, 영향받는 동작, 비목표, 수용 검사를 못 박습니다.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
```

이 단계에서는 코드 변경이 아니라 정리된 버그 리포트, 즉 `pending approval`로 표시된 스펙을 얻습니다.

</Step>

<Step>

#### `ralplan`으로 검토된 계획으로 바꿉니다

`ralplan`은 정리된 리포트를 Planner/Architect/Critic 합의 루프에 통과시킵니다. 코드를 건드리기 전에 약한 순서나 빈약한 검증을 잡아냅니다.

```text
/skill:ralplan turn the clarified bug report into a reviewed fix plan
```

검토된 계획도 `pending approval` 상태로 남습니다. 접근 방식을 명시적으로 승인합니다.

</Step>

<Step>

#### `ultragoal`로 지속 가능한 goal을 만듭니다

승인된 계획을 `ultragoal`에 넘기면 작업을 지속 가능한 goal로 추적합니다. 계획은 `--brief-file`로 전달합니다.

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
```

작업이 `.gjc/ultragoal/` 아래의 감사 가능한 goal 원장이 됩니다.

</Step>

<Step>

#### (선택) `team`으로 작업을 나눕니다

병렬 워커가 진짜로 도움이 될 때만, 구현과 검증을 tmux 기반 `executor` 워커들에게 나눕니다.

```sh
gjc team 2:executor "split implementation and verification for this bug fix"
```

`team`은 선택적 실행 모드입니다. 반드시 거쳐야 하는 단계가 아니므로 단일 스레드 수정에는 건너뜁니다. `team`은 tmux 세션 안(`$TMUX` 설정)에서만 동작합니다. 먼저 셸에서 `gjc` CLI를 실행하세요(예: `gjc --tmux`).

</Step>

<Step>

#### goal을 완료하고 증거를 모읍니다

`ultragoal`은 작업을 최종 증거 요약까지 끌고 갑니다.

```sh
gjc ultragoal complete-goals
```

무엇을 바꿨는지, 발견 후 무엇을 수정했는지, 어떤 검사가 돌았는지, 그리고 `.gjc/` 아래에 남은 흔적을 돌려받습니다.

</Step>

</Steps>

참고용으로, 전체 흐름을 한 블록에 담으면 다음과 같습니다.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# Optional only for parallel work:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

### 새 기능 구축

- **언제 사용하나요:** 원하는 바는 대충 알지만, 요구사항에 드러낼 만한 가정이 남아 있는 새 기능.
- **스킬 / 에이전트:** 범위가 모호하면 `deep-interview`로 시작하고, 합의된 계획을 위해 `ralplan`, 그다음 `ultragoal`로 끌고 갑니다. `executor`가 구현하고 `architect`와 `critic`이 검토합니다.

기능이 크거나 요구사항이 모호하면, 권장 체인은 명확성 → 실현 가능성 → 지속 가능한 goal입니다.

```text
/skill:deep-interview clarify the feature, scope, non-goals, and acceptance criteria
/skill:ralplan turn the clarified spec into a reviewed implementation plan
gjc ultragoal create-goals --brief-file <approved-plan>
gjc ultragoal complete-goals
```

요구사항이 이미 바로 계획할 만큼 명확하면 `deep-interview`를 건너뛰고 `ralplan`부터 시작하세요.

### 안전한 리팩터링과 정리

- **언제 사용하나요:** 검증 전까지 메인 체크아웃에서 떨어뜨려 두고 싶은 구조적 변경.
- **스킬 / 에이전트:** 격리된 Git worktree에 더해, 접근 방식을 위한 `ralplan`과 변경 추적을 위한 `ultragoal`. `architect`가 읽기 전용으로 검토합니다.

메인 체크아웃을 깨끗하게 유지하도록 격리된 Git worktree 안에서 세션을 실행합니다.

```sh
gjc --tmux --worktree <path>
```

일회성이거나 브랜치별 작업에는 전용 경로를 쓰세요. worktree 안에서 `ralplan`으로 계획하고, `ultragoal`로 변경을 추적합니다.

### 낯선 코드베이스 탐색

- **언제 사용하나요:** 바꾸기 전에 시스템을 이해해야 할 때.
- **스킬 / 에이전트:** 먼저 읽기 전용 `architect`와 `planner`. `architect`는 아키텍처 및 코드 리뷰 평가를, `planner`는 순서 정리와 수용 기준을 담당합니다. 둘 다 파일을 변경하지 않습니다.

읽기 전용 역할을 앞세워 시스템을 파악합니다. 구체적인 변경을 설명할 수 있게 되면 `ralplan`으로 넘어갑니다. 승인할 만한 계획이 나오기 전까지 탐색은 읽기 전용으로 유지하세요.

### 길고 손 떼는, 또는 병렬 작업

- **언제 사용하나요:** 하나의 추론 단위를 넘어 살아남아야 하는 다단계 goal, 또는 워커들로 깔끔하게 나뉘는 작업.
- **스킬 / 에이전트:** 지속 가능한 리더 소유 원장을 위한 `ultragoal`, 보이는 tmux 실행 엔진으로서의 `team`.

`ultragoal`은 `.gjc/ultragoal/goals.json`과 `.gjc/ultragoal/ledger.jsonl`을 소유합니다. `team`은 작업과 검증 증거를 리더에게 돌려줍니다. 리더가 그 증거로 체크포인트를 찍습니다. 워커는 직접 `ultragoal`을 체크포인트하지 않습니다.

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
gjc team 2:executor "<task description>"
gjc ultragoal complete-goals
```

### 빠른 답변이나 작은 편집

- **언제 사용하나요:** 질문, 한 줄 수정, 또는 정확히 설명할 수 있는 변경.
- **스킬 / 에이전트:** 없음. 직접 실행이 맞는 도구입니다.

워크플로가 필요 없습니다. 파일 경로, 함수 이름, 수용 기준 같은 명확한 앵커가 있는 요청이면 바로 처리하기에 충분합니다. 모호함이나 지속성이 진짜로 필요할 때만 워크플로를 꺼내세요.
