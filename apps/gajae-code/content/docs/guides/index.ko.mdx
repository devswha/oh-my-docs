이 가이드는 하나의 작업을 Gajae-Code 작업 루프로 끝까지 따라갑니다. 대상은 구체적인 버그 수정입니다. README의 운영자 순서를 그대로 따릅니다. `deep-interview`로 명확히 하고, `ralplan`으로 접근을 검토하고, `ultragoal`로 구현·수정·검증까지 작업을 끌고 가며, 병렬 워커가 도움이 될 때 선택적으로 `team` 단계를 더합니다.

이 워크스루의 목적은 각 단계에서 무엇을 기대해야 하는지, 그리고 어떤 증거를 돌려받는지 보여주는 것입니다. 이 순서는 운영자의 흐름을 설명할 뿐 숨겨진 자동화가 아닙니다. 에이전트는 무엇을 바꿨는지, 발견 후 무엇을 수정했는지, 어떤 검사를 돌렸는지, 수정을 뒷받침하는 증거가 무엇인지 계속 보고합니다.

## 시작하기 전에

권장되는 tmux 기반 세션을 실행합니다. 버그 수정에서는 보통 메인 체크아웃을 깨끗하게 유지하기 위해 브랜치 로컬 작업 공간을 원하므로, 격리된 Git worktree 안에서 실행하세요.

```sh
gjc --tmux --worktree <path>
```

일회성이거나 브랜치별 작업에는 전용 경로를 쓰세요. 여기서부터 루프는 `deep-interview -> ralplan -> ultragoal`이고, `team`은 조율된 병렬 워커가 작업에 도움이 될 때만 더합니다.

## 워크스루

<Steps>

<Step>

### `deep-interview`로 버그를 명확히 하기

계획이나 코드 변경 전에 모호함을 제거하는 것으로 시작합니다. `deep-interview`를 실행해 버그, 영향받는 동작, 비목표, 그리고 수정이 동작함을 입증할 수용 기준을 못 박습니다.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
```

**무엇을 기대할까요:** 코드 변경이 아니라 정리된 버그 리포트입니다. 이 산출물은 `pending approval` 상태로 남고, `deep-interview`는 스스로 구현을 시작하지 않습니다. 이 단계를 마치면 무엇이 깨졌고 "고쳐졌다"가 무엇을 의미하는지에 대한 합의를 갖게 됩니다.

</Step>

<Step>

### `ralplan`으로 검토된 계획으로 바꾸기

의도와 수용 기준이 명확해지면, 변경 전에 계획을 세우고 비평합니다. `ralplan`은 정리된 버그 리포트를 수정 계획으로 바꾸고, 읽기 전용 `critic`에게 통과시켜 약한 순서나 빈약한 검증을 코드를 건드리기 전에 잡아냅니다.

```text
/skill:ralplan turn the clarified bug report into a reviewed fix plan
```

**무엇을 기대할까요:** 마찬가지로 `pending approval` 상태로 남는 검토된 수정 계획입니다. 아직 구현되는 것은 없습니다. 접근, 순서, 제안된 검사가 만족스러울 때 계획을 명시적으로 승인합니다.

</Step>

<Step>

### `ultragoal`로 지속 가능한 goal 만들기

승인된 계획을 `ultragoal`에 넘깁니다. `ultragoal`은 구현·수정·검증을 거치며 작업을 지속 가능한 goal로 추적합니다. 계획은 `--brief-file`로 전달합니다.

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
```

**무엇을 기대할까요:** 작업이 감사 가능한 goal 원장으로 나뉩니다. 워크플로 원장, 계획, 스펙은 `.gjc/` 아래에 남아 나중에도 검토할 수 있습니다.

</Step>

<Step>

### (선택) `team`으로 작업 나누기

조율된 병렬 워커가 작업에 도움이 될 때만 `team` 단계를 더합니다. 예를 들어 버그 수정의 구현과 검증을 tmux 기반 `executor` 워커 두 개로 나눕니다.

```sh
gjc team 2:executor "split implementation and verification for this bug fix"
```

**무엇을 기대할까요:** `team`은 선택적 실행 모드이며 반드시 거쳐야 하는 핸드오프가 아닙니다. 단일 스레드 수정에서는 건너뛰고, 병렬 tmux 워커가 정말 도움이 될 때 꺼내 쓰세요.

</Step>

<Step>

### goal을 완료하고 증거 모으기

goal을 완료까지 끌고 갑니다. `ultragoal`은 "코드를 좀 썼다"에서 멈추지 않고 최종 증거 요약까지 작업을 끌고 갑니다.

```sh
gjc ultragoal complete-goals
```

**무엇을 기대할까요:** 어떤 검사가 돌았고 무엇을 보여줬는지가 함께 따라오는 완료된 작업입니다. 에이전트는 무엇을 바꿨는지, 발견 후 무엇을 수정했는지, 어떤 검사를 돌렸는지, 수정을 뒷받침하는 증거가 무엇인지 보고합니다.

</Step>

</Steps>

## 전체 운영자 순서

참고용으로, 한 패스 전체를 한 블록에 모으면 다음과 같습니다.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# 병렬 작업이 필요할 때만:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

## 증거는 어떤 모습일까요

완료된 `ultragoal` 패스는 diff에서 끝나지 않습니다. 에이전트는 다음을 보고합니다.

- **무엇을 바꿨는지** — `executor`가 한 구체적인 편집, 수정, 리팩터링.
- **발견 후 무엇을 수정했는지** — 검증이나 리뷰가 문제를 드러낸 뒤 한 조정.
- **어떤 검사를 돌렸는지** — 계획에서 정한 검증 단계와 그 결과.
- **수정을 뒷받침하는 증거** — `.gjc/` 아래에 남아 결과를 나중에 감사할 수 있게 하는 흔적.

## 라우팅: 맞는 가장 작은 워크플로를 쓰기

이 버그 수정 패스는 전체 루프를 쓰지만, 대부분의 작업은 더 적게 필요합니다. 요청을 만족하는 가장 작은 워크플로를 고르세요.

1. 명확하고 위험이 낮은 편집은 직접 구현합니다.
2. 의도, 범위, 수용 기준이 모호하면 `deep-interview`.
3. 계획을 세울 만큼 요구사항이 명확하지만 아키텍처, 순서, 검증에 합의가 필요하면 `ralplan`.
4. 작업을 감사 가능한 원장과 함께 지속 가능한 goal로 나눠야 하면 `ultragoal`.
5. 승인된 작업이 병렬 워커로 이득을 보면 `team`.
