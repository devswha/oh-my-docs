Gajae-Code(`gjc`)は公開表面を意図的に小さく保ちます。4 つのワークフロースキル(`deep-interview`、`ralplan`、`team`、`ultragoal`)と 4 つのロールエージェント(`executor`、`architect`、`planner`、`critic`)だけです。要点は、タスクに合う最小のワークフローを選び、その後はそのワークフローに作業を最後まで運ばせることです。

## タスクに合う最小のワークフローを選ぶ

ほとんどのタスクに完全なループは不要です。作業が求める分だけ、はしごを登ってください。

1. **直接編集** —— ファイルパス、関数名、受け入れ基準が明確で低リスクの作業。ワークフロー不要、そのまま実行します。
2. **`deep-interview`** —— 意図・範囲・受け入れ基準が曖昧なとき。計画やコード変更の前に、ソクラテス式の質問で曖昧さを取り除きます。
3. **`ralplan`** —— 計画は立てられるが、アプローチ・順序・検証で合意が必要なとき。Planner/Architect/Critic ループが変更前に計画を批評します。
4. **`ultragoal`** —— 持続的で監査可能な多段階ゴール。作業は `.gjc/` 配下に残るゴール台帳へ分割され、実装・修正・検証を通して進められます。
5. **`team`** —— *承認済み*の作業が並列 tmux ワーカーで利益を得るときのみ。

計画成果物は、実行パスを明示的に承認するまで `pending approval` のままです。`deep-interview` と `ralplan` は自ら実装を開始しません。仕様や計画を作って停止します。

| ワークフロー | いつ使うか |
| --- | --- |
| 直接編集 | 正確に説明できる、明確で低リスクの変更。 |
| `deep-interview` | 意図 / 範囲 / 受け入れ基準が曖昧。 |
| `ralplan` | 計画可能だが、アプローチ / 順序 / 検証に合意が必要。 |
| `ultragoal` | 持続的な台帳を持つ、持続的で監査可能なゴール。 |
| `team` | 並列 tmux ワーカーで利益を得る承認済みの作業。 |

## ユースケース

### バグ修正

典型的な完全ループです。明確化し、計画し、運び、検証します。各ステップは「コードを少し書いた」ではなく証拠で終わります。

- **いつ使うか:** コードに触れる前に原因や修正範囲を確定すべき実際の不具合。
- **スキル / エージェント:** `deep-interview` → `ralplan` → `ultragoal`。`critic` が計画を確認し、`executor` が範囲を限定した修正を行います。並列作業のときだけ `team` を加えます。

<Steps>

<Step>

#### `deep-interview` でバグを明確化する

計画に入る前に、バグ、影響を受ける挙動、非目標、受け入れチェックを確定します。

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
```

得られるのはコード変更ではなく、明確化されたバグレポート —— `pending approval` と記された仕様です。

</Step>

<Step>

#### `ralplan` でレビュー済みの計画に変える

`ralplan` は明確化されたレポートを Planner/Architect/Critic 合意ループに通し、コードに触れる前に弱い順序や薄い検証を捕まえます。

```text
/skill:ralplan turn the clarified bug report into a reviewed fix plan
```

レビュー済みの計画も `pending approval` のままです。アプローチを明示的に承認します。

</Step>

<Step>

#### `ultragoal` で持続的なゴールを作る

承認済みの計画を `ultragoal` に渡すと、作業を持続的なゴールとして追跡します。計画は `--brief-file` で渡します。

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
```

作業は `.gjc/ultragoal/` 配下の監査可能なゴール台帳になります。

</Step>

<Step>

#### (任意)`team` で作業を分割する

並列ワーカーが本当に役立つときだけ、実装と検証を tmux ベースの `executor` ワーカーに分割します。

```sh
gjc team 2:executor "split implementation and verification for this bug fix"
```

`team` は任意の実行モードであり、必須の引き継ぎではありません。シングルスレッドの修正では省きます。`team` は tmux セッション内（`$TMUX` が設定済み）でのみ動作します — まずシェルから `gjc` CLI を起動してください。例：`gjc --tmux`。

</Step>

<Step>

#### ゴールを完了し証拠を集める

ゴールを完了まで運びます。`ultragoal` は作業を最終的な証拠サマリーまで運びます。

```sh
gjc ultragoal complete-goals
```

何を変えたか、発見後に何を修正したか、どの検査を実行したか、そして `.gjc/` 配下に残る痕跡が返ってきます。

</Step>

</Steps>

参考までに、全体の流れを 1 ブロックにまとめると次のとおりです。

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# Optional only for parallel work:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

### 新機能の構築

- **いつ使うか:** 欲しいものは大まかに分かるが、要件にまだ表に出すべき前提が残っている新しい能力。
- **スキル / エージェント:** 範囲が曖昧なら `deep-interview` から始め、合意計画のために `ralplan`、続いて `ultragoal` で運びます。`executor` が実装し、`architect` と `critic` がレビューします。

機能が大きい、または要件が曖昧なときの推奨チェーンは、明確さ → 実現可能性 → 持続的なゴールです。

```text
/skill:deep-interview clarify the feature, scope, non-goals, and acceptance criteria
/skill:ralplan turn the clarified spec into a reviewed implementation plan
gjc ultragoal create-goals --brief-file <approved-plan>
gjc ultragoal complete-goals
```

要件がすでに直接計画できるほど明確なら、`deep-interview` を省いて `ralplan` から始めます。

### 安全なリファクタリングと整理

- **いつ使うか:** 検証まではメインのチェックアウトから切り離しておきたい構造的な変更。
- **スキル / エージェント:** 隔離された Git worktree に加え、アプローチ用の `ralplan` と変更追跡用の `ultragoal`。`architect` が読み取り専用でレビューします。

メインのチェックアウトを清潔に保つため、隔離された Git worktree 内でセッションを実行します。

```sh
gjc --tmux --worktree <path>
```

使い捨てやブランチ固有の作業には専用パスを使ってください。worktree 内で `ralplan` で計画し、`ultragoal` で変更を追跡します。

### 不慣れなコードベースの探索

- **いつ使うか:** 変更する前にシステムを理解する必要があるとき。
- **スキル / エージェント:** まず読み取り専用の `architect` と `planner`。`architect` はアーキテクチャとコードレビューの評価、`planner` は順序と受け入れ基準を担当します。どちらもファイルを変更しません。

読み取り専用ロールを先頭に立ててシステムを把握し、具体的な変更を説明できるようになったら `ralplan` へ移ります。承認に値する計画ができるまで、探索は読み取り専用に保ちます。

### 長時間・手放し・並列の作業

- **いつ使うか:** 1 回の推論を超えて存続すべき多段階ゴール、またはワーカー間できれいに分割できる作業。
- **スキル / エージェント:** 持続的でリーダー所有の台帳には `ultragoal`、可視の tmux 実行エンジンとして `team`。

`ultragoal` は `.gjc/ultragoal/goals.json` と `.gjc/ultragoal/ledger.jsonl` を所有します。`team` はタスクと検証の証拠をリーダーに返します。リーダーがその証拠でチェックポイントを打ち、ワーカーが自ら `ultragoal` をチェックポイントすることはありません。

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
gjc team 2:executor "<task description>"
gjc ultragoal complete-goals
```

### 手早い回答や小さな編集

- **いつ使うか:** 質問、1 行の修正、または正確に説明できるあらゆる変更。
- **スキル / エージェント:** なし。直接実行が正しいツールです。

ワークフローは不要です。ファイルパス、関数名、受け入れ基準といった明確なアンカーがあるリクエストなら、そのまま実行できます。曖昧さや持続性が本当に必要なときだけ、ワークフローを取り出してください。

## worktree 隔離作業

メインのチェックアウトから切り離すべき作業 —— リファクタリング、リスクのある実験、ブランチ固有のタスク —— は、隔離された Git worktree 内で実行します。

```sh
gjc --tmux --worktree <path>
```

メインのチェックアウトを清潔に保つため、使い捨てやブランチ固有の作業には専用パスを使ってください。
