Gajae-Code (`gjc`) keeps its public surface intentionally small: four workflow skills (`deep-interview`, `ralplan`, `team`, `ultragoal`) and four role agents (`executor`, `architect`, `planner`, `critic`). The skill is choosing the smallest workflow that fits, then letting it carry the work the rest of the way.

## Choose the smallest workflow that fits

Most tasks need less than the full loop. Climb the ladder only as far as the work demands:

1. **Direct edit** — clear, low-risk work with file paths, function names, or acceptance criteria. Just do it; no workflow needed.
2. **`deep-interview`** — intent, scope, or acceptance is fuzzy. Socratic questions remove ambiguity before any planning or code change.
3. **`ralplan`** — you can plan, but need to agree on approach, sequencing, or verification. The Planner/Architect/Critic loop critiques the plan before mutation.
4. **`ultragoal`** — durable, auditable multi-step goals. Work is split into a ledger of goals that persist under `.gjc/` and are driven through implementation, revision, and verification.
5. **`team`** — only when *approved* work benefits from parallel tmux workers.

Plan artifacts stay `pending approval` until you explicitly approve an execution path. `deep-interview` and `ralplan` do not self-start implementation — they end with a spec or plan and stop.

| Workflow | When to reach for it |
| --- | --- |
| Direct edit | Clear, low-risk change you can describe precisely. |
| `deep-interview` | Intent / scope / acceptance criteria are ambiguous. |
| `ralplan` | Planable, but approach / sequencing / verification needs consensus. |
| `ultragoal` | Durable, auditable goals with a persistent ledger. |
| `team` | Approved work that benefits from parallel tmux workers. |

## Use cases

### Fix a bug

This is the canonical full loop: clarify, plan, carry, verify. Each step ends with evidence, not just "wrote some code."

- **When to use:** a real defect where the cause or the fix boundary needs to be pinned down before touching code.
- **Skills / agents:** `deep-interview` → `ralplan` → `ultragoal`, with `critic` reviewing the plan and `executor` doing the bounded fix; add `team` only for parallel work.

<Steps>

<Step>

#### Clarify the bug with `deep-interview`

Pin down the bug, the affected behavior, the non-goals, and the acceptance checks before any planning.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
```

You leave with a clarified bug report — a spec marked `pending approval`, not a code change.

</Step>

<Step>

#### Turn it into a reviewed plan with `ralplan`

`ralplan` runs the clarified report through the Planner/Architect/Critic consensus loop so weak sequencing or thin verification gets caught before code is touched.

```text
/skill:ralplan turn the clarified bug report into a reviewed fix plan
```

The reviewed plan also stays `pending approval`. You approve the approach explicitly.

</Step>

<Step>

#### Create durable goals with `ultragoal`

Hand the approved plan to `ultragoal`, which tracks the work as durable goals. Pass the plan with `--brief-file`:

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
```

The work becomes an auditable ledger of goals under `.gjc/ultragoal/`.

</Step>

<Step>

#### (Optional) Split the work with `team`

Only when parallel workers genuinely help, split the implementation and verification across `executor` workers backed by tmux:

```sh
gjc team 2:executor "split implementation and verification for this bug fix"
```

`team` is an optional execution mode, not a required handoff. Skip it for single-threaded fixes.

</Step>

<Step>

#### Complete the goals and collect evidence

Drive the goals to completion. `ultragoal` carries the work through to a final evidence summary.

```sh
gjc ultragoal complete-goals
```

You get back what changed, what was revised after findings, which checks ran, and the persisted trail under `.gjc/`.

</Step>

</Steps>

The whole pass in one block, for reference:

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# Optional only for parallel work:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

### Build a new feature

- **When to use:** a new capability where you know roughly what you want, but the requirements still carry assumptions worth exposing.
- **Skills / agents:** start with `deep-interview` when scope is fuzzy, then `ralplan` for a consensus plan, then `ultragoal` to carry it; `executor` implements, `architect` and `critic` review.

When the feature is large or the requirements are vague, the recommended chain is clarity → feasibility → durable goals:

```text
/skill:deep-interview clarify the feature, scope, non-goals, and acceptance criteria
/skill:ralplan turn the clarified spec into a reviewed implementation plan
gjc ultragoal create-goals --brief-file <approved-plan>
gjc ultragoal complete-goals
```

If the requirements are already clear enough to plan directly, skip `deep-interview` and start at `ralplan`.

### Refactor or clean up safely

- **When to use:** structural changes you want kept off the main checkout until verified.
- **Skills / agents:** an isolated Git worktree plus `ralplan` for the approach and `ultragoal` to track the change; `architect` reviews read-only.

Run the session inside an isolated Git worktree so the main checkout stays clean:

```sh
gjc --tmux --worktree <path>
```

Use a dedicated path for throwaway or branch-specific work. From inside the worktree, plan with `ralplan`, then track the change with `ultragoal`.

### Explore an unfamiliar codebase

- **When to use:** you need to understand a system before changing it.
- **Skills / agents:** read-only `architect` and `planner` first — `architect` for architecture and code-review assessment, `planner` for sequencing and acceptance criteria. Neither mutates files.

Lead with the read-only roles to map the system, then move into `ralplan` once you can describe a concrete change. Keep exploration read-only until you have a plan worth approving.

### Long, hands-off, or parallel work

- **When to use:** a multi-step goal that should survive beyond one reasoning burst, or work that splits cleanly across workers.
- **Skills / agents:** `ultragoal` for the durable leader-owned ledger, with `team` as the visible tmux execution engine.

`ultragoal` owns `.gjc/ultragoal/goals.json` and `.gjc/ultragoal/ledger.jsonl`; `team` returns task and verification evidence to the leader. The leader checkpoints from that evidence — workers never checkpoint `ultragoal` themselves.

```sh
gjc ultragoal create-goals --brief-file <approved-plan>
gjc team 2:executor "<task description>"
gjc ultragoal complete-goals
```

### Quick answer or a small edit

- **When to use:** a question, a one-line fix, or any change you can state precisely.
- **Skills / agents:** none. Direct execution is the right tool.

No workflow is needed. A clear, anchored request — a file path, function name, or acceptance criterion — is enough to act on directly. Reach for a workflow only when ambiguity or durability genuinely warrant it.

## Worktree-isolated work

For anything that should stay off the main checkout — refactors, risky experiments, branch-specific tasks — run inside an isolated Git worktree:

```sh
gjc --tmux --worktree <path>
```

Use a dedicated path for throwaway or branch-specific work so the main checkout stays clean.
