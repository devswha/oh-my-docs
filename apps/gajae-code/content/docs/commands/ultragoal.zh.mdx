title: ultragoal
description: 在持久的 repo-native 目标产物之上，把批准的计划推进到实现、修订、验证和最终证据汇总。
---

`ultragoal` 把批准的 brief 变为 repo-native 产物，再安全地把目标推进到实现、修订、验证和最终证据汇总。它是[工作循环](/zh/docs/commands)中“以证据收尾的执行”阶段：[`ralplan`](/zh/docs/commands/ralplan) 产出评审过的计划后，`ultragoal` 把它带到完成并 checkpoint 的结果。

## 它跟踪什么

计划会变成一小组持久的 repo-native 产物：

| 产物 | 用途 |
| --- | --- |
| `.gjc/ultragoal/brief.md` | 计划所依据的 brief。 |
| `.gjc/ultragoal/goals.json` | 聚合计划与各 story 状态。 |
| `.gjc/ultragoal/ledger.jsonl` | 持久审计轨迹：checkpoint 与 steering 事件。 |

新计划默认对整个持久计划使用稳定指针式的聚合目标，story 进度记录在 ledger 中。ledger 是持久审计轨迹——每次成功或失败后都要 checkpoint。

## 创建目标

从批准的计划开始。用 brief 字符串、brief 文件或 stdin：

```bash
gjc ultragoal create-goals --brief "<brief>"
gjc ultragoal create-goals --brief-file <approved-plan>
cat <approved-plan> | gjc ultragoal create-goals --from-stdin
```

然后查看 `.gjc/ultragoal/goals.json` 并按需调整。仅在明确希望每个 story 一个目标上下文时使用 `--gjc-goal-mode per-story`。

若请求没有批准的计划或共识产物，先运行 [`ralplan`](/zh/docs/commands/ralplan)，并把其 PRD、测试规格、角色名单和验证指引保存进 ledger。不要用临时执行悄悄替代缺失的规划。

## 完成目标

把执行推进到完成。循环直到 `gjc ultragoal status` 报告所有目标完成：

1. 运行 `gjc ultragoal complete-goals` 并阅读打印的交接。
2. 调用 `goal({"op":"get"})`；若没有 active 目标，用打印的 payload objective 调用 `goal({"op":"create",...})`。
3. 只实现当前 story。
4. 运行下方的强制完成清理与评审门。
5. 用新的 active 目标快照 checkpoint 持久 ledger。
6. 对下一个 story 重复；失败用 `gjc ultragoal complete-goals --retry-failed` 恢复。

内联目标状态通过统一的 `goal` 工具驱动——它是目标状态唯一面向智能体的接口：

```json
goal({"op":"get"})
goal({"op":"create","objective":"<printed payload objective>"})
goal({"op":"complete"})
```

在聚合模式下，不要对中间 story 调用 `goal({"op":"complete"})`。它仅在评审门通过且最终聚合回执存在后，在最后一个 story 上触发。

```bash
gjc ultragoal status
gjc ultragoal status --json
gjc ultragoal complete-goals
gjc ultragoal complete-goals --retry-failed
```

## Checkpoint

每个 story 都用 checkpoint 记录。complete checkpoint 需要干净的质量门；运行时 hook 没有它会拒绝收尾：

```bash
gjc ultragoal checkpoint --goal-id <id> --status complete \
  --evidence "<evidence>" \
  --gjc-goal-json <goal-get-json-or-path> \
  --quality-gate-json <quality-gate-json-or-path>
```

若 story 受阻或失败，改为记录终态：

```bash
gjc ultragoal checkpoint --goal-id <id> --status failed --evidence "<blocker/evidence>"
```

把 ledger 当作证据：checkpoint 生成回执，仅 `goals.json.status` 不能证明完成。

## 完成清理与评审门

质量门运行前，story 不能被 checkpoint 为 `complete`：

1. 对该 story 运行定向验证。
2. 仅对改动的文件运行清理/重构评审（即使没有相关改动，cleaner 仍运行并记录 no-op 报告），然后重跑验证。
3. 委派覆盖架构、产品、代码三个通道的 `architect` 评审，以及一个尝试破坏改动的 `executor` QA/red-team 通道。
4. 任一通道发现问题时，用 `gjc ultragoal record-review-blockers` 记录持久的 blocker 工作，而非完成。
5. 仅当循环干净时，用结构化质量门和新的 active 目标快照 checkpoint `complete`。

“干净”意味着所有 `architectReview` 状态为 `"CLEAR"`、recommendation 为 `"APPROVE"`、executor QA 状态为 `"passed"`、iteration 为 `"passed"` 且 `fullRerun: true`、每个 evidence 字段非空、每个 blockers 数组为空。`COMMENT`、`WATCH`、`REQUEST CHANGES`、`BLOCK`、缺失证据或非空 blockers 均为不干净。

```bash
gjc ultragoal record-review-blockers --goal-id <id> \
  --title "Resolve verification blockers" \
  --objective "<blocker-resolution objective>" \
  --evidence "<architect/executor findings>" \
  --gjc-goal-json <active-goal-get-json-or-path>
```

## 动态 steering

当真实发现证明应当改变 story 分解、而聚合目标和原始约束保持不变时，使用 `gjc ultragoal steer`。steering 仅限显式且有证据支撑；宽泛的自然语言请求会被拒绝而非猜测。

```bash
gjc ultragoal steer --kind add_subgoal \
  --title "Investigate blocker" \
  --objective "Validate the blocker and report evidence." \
  --evidence "log/test output" \
  --rationale "The blocker changes the safe execution order." --json
```

允许的 mutation 类型为 `add_subgoal`、`split_subgoal`、`reorder_pending`、`revise_pending_wording`、`annotate_ledger`、`mark_blocked_superseded`。steering 从不编辑聚合目标、原始 brief 约束、质量门或完成状态，也不硬删除目标或削弱验证。被接受和被拒绝的尝试都会向 ledger 追加审计条目。

## 角色智能体委派

当 story 大到值得委派时，使用内置角色智能体：

| 智能体 | 通道 |
| --- | --- |
| [`executor`](/zh/docs/skills/executor) | 有界的实现、重构与修复切片。 |
| [`planner`](/zh/docs/skills/planner) | story 排序或交接细化。 |
| [`architect`](/zh/docs/skills/architect) | 只读架构与代码评审（`CLEAR` / `WATCH` / `BLOCK`）。 |
| [`critic`](/zh/docs/skills/critic) | 只读计划或交接批评。 |

对于有独立切片的大型子目标，leader 必须派生并行的 `executor` 子智能体，而非串行单独作业，并为每个分配有界、可干净分离的目标。

ultragoal leader 拥有 `goals.json` 和 `ledger.jsonl`。角色智能体返回实现与评审证据——它们不 checkpoint ultragoal，也不改动目标状态。

## ultragoal 与 team 配合

对于受益于可见并行执行通道的持久 story，把 ultragoal 与 [`team`](/zh/docs/commands/team) 配对。ultragoal 保持 leader 所有；team 是返回任务与证据状态的 tmux 执行引擎。leader 用新的目标快照从 team 证据 checkpoint ultragoal。worker 不拥有目标状态，也不得运行 `gjc ultragoal checkpoint`。
