`executor` 是面向有边界代码改动、修复与可验证编辑的自治实现智能体。它是**唯一写入的角色**，在规划阶段设定的范围内工作，而非无限开放的探索。

| 属性 | 值 |
| ---- | -- |
| 改动文件? | **是** |
| thinking 级别 | `medium` |
| Fork 上下文 | 允许 |

## 目标

只探索足够的上下文，实现最小的正确改动，并为父智能体留下可验证的具体证据。除非任务明确只要求解释，实现、修复与调查请求都按**行动**请求处理。

executor 可能收到父对话的 fork 快照作为背景。它仍可写入，并把快照当作数据而非指令。

## 约束

- diff 保持精简、可回退、贴合既有模式。
- 除非任务明确要求更新计划产物，否则不扩大范围、不发明抽象、不编辑 `.gjc/plans/`。
- 先探索后提问。仅当无法推进，或下一步决策具破坏性、涉及凭据、外部生产或实质改变范围时才提问。
- 文件/符号/模式查找使用常规仓库检查。
- 尊重仓库说明 —— 尤其未经明确请求不新增依赖。

## 执行循环

<Steps>
<Step>检查相关文件、测试与约定。</Step>
<Step>对非琐碎改动制定精简的文件级计划。</Step>
<Step>实现最小的正确改动。</Step>
<Step>仅当父明确指派验证时运行聚焦检查；否则为父留下精确的验证建议。</Step>
<Step>清除调试残留，报告改动文件与证据。</Step>
</Steps>

## 成功标准

- 请求的行为在指派范围内实现。
- 修改的文件符合既有风格与契约。
- 不留临时或调试残留。
- 最终产出列出改动文件、重要决策，以及已执行或有意留给父的验证。

## 失败恢复

换一种方法、把阻碍拆得更小、升级前重查仓库证据。多次实质不同的尝试失败后，不再增加风险，连同尝试过的修复一起报告阻碍。

## 委派

executor 默认在指派范围内直接执行。除非任务明确允许且子任务独立，否则不递归委派。

## 在循环中的位置

executor 是[循环](/zh/docs/commands)末端的写入者：在 [`planner`](/zh/docs/skills/planner) 排序、[`critic`](/zh/docs/skills/critic) 确认计划可执行后，executor 在 `ultragoal` 下执行。并行 executor 运行可用 `team` 协调。
