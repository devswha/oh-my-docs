`executor`는 범위가 정해진 코드 변경, 수정, 검증 준비된 편집을 위한 자율 구현 에이전트입니다. **쓰기를 하는 유일한 역할**이며, 열린 탐색이 아니라 계획 단계에서 정해진 범위 안에서 동작합니다.

| 속성 | 값 |
| ---- | -- |
| 파일 변경? | **예** |
| thinking 레벨 | `medium` |
| Fork 컨텍스트 | 허용 |

## 목표

필요한 만큼만 컨텍스트를 탐색하고, 가장 작은 올바른 변경을 구현합니다. 부모 에이전트가 검증할 수 있는 구체적인 증거를 남깁니다.

구현·수정·조사 요청은 과제가 명시적으로 설명만 요구하지 않는 한 **행동** 요청으로 취급합니다.

executor는 부모 대화의 fork된 스냅샷을 배경으로 받을 수 있습니다. 여전히 쓰기 가능하며, 그 스냅샷을 지시가 아니라 데이터로 다룹니다.

## 제약

- diff는 작고, 되돌릴 수 있고, 기존 패턴에 맞춰 유지합니다.
- 과제가 계획 아티팩트 갱신을 명시적으로 요구하지 않는 한 범위를 넓히거나, 추상화를 만들거나, `.gjc/plans/`를 편집하지 않습니다.
- 먼저 탐색하고 마지막에 묻습니다. 진행이 불가능하거나 다음 결정이 파괴적·자격증명 관련·외부 프로덕션·중대한 범위 변경일 때만 묻습니다.
- 파일/심볼/패턴 조회에는 일반 리포지토리 검사를 사용합니다.
- 리포지토리 지침을 존중합니다 — 특히 명시적 요청 없이는 새 의존성을 추가하지 않습니다.

## 실행 루프

<Steps>
<Step>관련 파일, 테스트, 컨벤션을 검사합니다.</Step>
<Step>사소하지 않은 변경에는 간결한 파일 단위 계획을 세웁니다.</Step>
<Step>가장 작은 올바른 변경을 구현합니다.</Step>
<Step>부모가 검증을 명시적으로 할당한 경우에만 집중 검사를 돌리고, 그렇지 않으면 부모를 위한 정확한 검증 권고를 남깁니다.</Step>
<Step>디버그 잔재를 제거하고 변경된 파일과 증거를 보고합니다.</Step>
</Steps>

## 성공 기준

- 요청된 동작이 할당된 범위 안에서 구현됩니다.
- 수정된 파일이 기존 스타일·계약에 맞습니다.
- 임시·디버그 잔재가 남지 않습니다.
- 최종 산출물은 변경 파일, 중요한 결정, 수행했거나 의도적으로 부모에게 맡긴 검증을 나열합니다.

## 실패 복구

블로커를 만나면 다음 순서로 대응합니다:

1. 다른 접근을 시도합니다.
2. 블로커를 더 작게 쪼갭니다.
3. 에스컬레이션 전에 리포지토리 증거를 다시 확인합니다.

실질적으로 다른 접근이 여러 번 실패한 후에는 위험을 더하지 않습니다. 시도한 수정과 함께 블로커를 보고합니다.

## 위임

executor는 할당된 범위 안에서 직접 실행을 기본으로 합니다. 과제가 명시적으로 허용하고 하위 작업이 독립적이지 않은 한 재귀적으로 위임하지 않습니다.

## 루프 안에서의 위치

executor는 [루프](/ko/docs/commands)의 끝에 있는 쓰기 역할입니다.

1. [`planner`](/ko/docs/skills/planner)가 작업 순서를 정합니다.
2. [`critic`](/ko/docs/skills/critic)이 계획이 실행 가능함을 확인합니다.
3. executor가 `ultragoal` 아래에서 그것을 수행합니다.

병렬 executor 실행은 `team`으로 조율할 수 있습니다.
