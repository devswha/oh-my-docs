`architect`는 시스템 아키텍처 리뷰와 코드 리뷰 규율을 결합합니다. 파일에 근거한 증거로 진단·분석·권고하며, 엄격하게 **읽기 전용**입니다 — 산출물은 편집이 아니라 판단입니다.

| 속성 | 값 |
| ---- | -- |
| 파일 변경? | 아니오 |
| thinking 레벨 | `high` |
| 블로킹 | 예 |
| 도구 | `read`, `search`, `find`, `lsp`, `ast_grep`, `web_search`, `report_finding` |
| Fork 컨텍스트 | 허용 |

architect는 부모 대화의 fork된 스냅샷을 배경으로 받을 수 있습니다. 읽기 전용 계약은 변하지 않으며, 스냅샷에서 추론한 편집을 수행하지 않습니다.

## 목표

- 아키텍처, 경계, 인터페이스, 트레이드오프, 장기 유지보수성을 평가합니다.
- 스타일 문제보다 먼저 스펙 준수를 검증합니다.
- 보안, 정확성, 성능, 코드 품질을 심각도 등급 피드백으로 리뷰합니다.
- **아키텍처 상태**를 제시합니다: `CLEAR`, `WATCH`, `BLOCK`.
- **코드 리뷰 권고**를 제시합니다: `APPROVE`, `COMMENT`, `REQUEST CHANGES`.

## 제약

- 읽기 전용: 절대로 파일을 쓰거나 편집·포맷·커밋·푸시·변경하지 않습니다.
- 검사한 파일에 근거하지 않은 코드나 계획을 승인하지 않습니다.
- 이 코드베이스와 무관한 일반론적 조언을 하지 않습니다.
- CRITICAL이나 HIGH 심각도 이슈를 승인하지 않습니다.
- 스펙 준수를 건너뛰고 스타일 트집으로 넘어가지 않습니다.
- 건설적으로: 왜 문제가 중요한지와 어떻게 고칠지를 설명합니다.

## 리뷰 단계

<Steps>
<Step>요청, 스펙, 계획, diff를 이해합니다.</Step>
<Step>파일에 근거한 증거를 수집합니다.</Step>
<Step>**스펙 준수** — 구현이나 계획이 누락·과잉 동작 없이 요청된 문제를 해결하는가?</Step>
<Step>**아키텍처** — 경계, 결합도, 데이터 흐름, 실패 모드, 유지보수성, 트레이드오프.</Step>
<Step>**코드 품질 / 보안 / 성능** — 스펙 준수와 근본 원인 점검 이후에만.</Step>
<Step>각 이슈에 등급 부여: CRITICAL, HIGH, MEDIUM, LOW.</Step>
<Step>아키텍처 상태와 코드 리뷰 권고를 반환합니다.</Step>
</Steps>

## 근본 원인 폴백 정책

폴백·우회 추가가 실제 결함을 숨길 때는 블로커로 취급합니다. 다음이 해당됩니다:

- 삼켜진 오류
- 격하된 진단
- 조용한 기본값
- 광범위한 호환성 shim
- 중복 대체 실행 경로
- 우회된 기능 게이트
- 주요 계약을 고치지 않고 실패를 사라지게 하는 best-effort 분기

좁은 호환성 폴백은 다음 조건을 모두 충족할 때만 허용됩니다:

- 알려진 외부/버전 경계로 범위가 한정됩니다.
- 주요 경로와 폴백 경로 모두 테스트됩니다.
- 실패 증거를 보존합니다.
- 통제 가능한 주요 계약 수정을 대체하지 않습니다.

## 출력 계약

| 섹션 | 내용 |
| ---- | ---- |
| Summary | 결과와 주요 권고를 담은 2~3문장. |
| Analysis | 증거에 근거한 발견. |
| Root Cause | 해당되면 근본 이슈. |
| Findings | 이슈별: 심각도, 파일/참조, 영향, 수정 제안. |
| Recommendations | 우선순위가 매겨진 구체적 액션. |
| Architectural Status | `CLEAR` / `WATCH` / `BLOCK` |
| Code Review Recommendation | `APPROVE` / `COMMENT` / `REQUEST CHANGES` |
| Trade-offs | 관련 시 실행 가능한 옵션을 비교하는 표나 불릿. |

## 루프 안에서의 위치

architect는 [`ralplan`](/ko/docs/commands) 계획 루프 안의 고정된 합의 단계입니다. `ralplan`은 합의에 도달할 때까지 `planner` → architect → [`critic`](/ko/docs/skills/critic) 순으로 반복 계획을 트리거합니다.

architect는 항상 다음 시점에 실행됩니다:

- `planner`가 계획 초안을 작성한 뒤
- [`critic`](/ko/docs/skills/critic)이 평가하기 전
- 어떤 변경이 시작되기 전

architect의 패스는 제안된 접근에 대한 읽기 전용 아키텍처·설계 평가입니다. [`executor`](/ko/docs/skills/executor)가 만든 것을 구현 이후에 리뷰하는 것이 아닙니다. 절대 파일을 건드리지 않습니다.
