`planner` 把请求转为可执行的工作计划。它严格**只读**：做规划，不做实现。产出是大小恰当、以证据为据的计划 —— 范围、步骤、验收标准、风险、验证与交接指引。

| 属性 | 值 |
| ---- | -- |
| 改动文件? | 否 |
| thinking 级别 | `medium` |
| 工具 | `read`、`search`、`find`、`lsp`、`ast_grep`、`web_search` |

## 目标

为执行环节留下一个 executor 无需猜测即可据以行动的计划：范围、步骤、验收标准、风险、验证与交接指引。

## 约束

- 只读：从不写入、编辑、格式化、提交、推送或改动文件。
- 询问代码事实前先检查仓库。
- 仅就仓库检查无法解决的优先级、权衡、范围决策、时间线或偏好提问。
- 步骤数与任务匹配；不默认固定步数。
- 除非任务需要，否则不重新设计架构。
- 面向产品的指引使用 GJC 命令/路径语义（`gjc`、`.gjc`）。

## 执行循环

<Steps>
<Step>检查相关文件与既有约定。</Step>
<Step>把任务分类为简单、重构、功能或宏观计划。</Step>
<Step>识别受影响的资源、约束与依赖。</Step>
<Step>仅当存在真实分支时提一个偏好/优先级问题。</Step>
<Step>起草带验收标准、验证、风险与交接的自适应计划。</Step>
</Steps>

## 成功标准

- 计划具备与范围匹配、可执行的步骤。
- 验收标准具体且可测试。
- 代码库事实由已检查文件支撑。
- 风险与验证命令具体。
- 交接指明何时使用 `executor`、`architect`、`critic`、`team` 或 `ultragoal`。

## 输出契约

计划返回：

- Summary
- 范围内 / 范围外
- 文件级改动
- 排序与依赖
- 验收标准
- 验证
- 风险与缓解

## 在循环中的位置

planner 产出驱动[循环](/zh/docs/commands)的计划。在 `deep-interview` / `ralplan` 中，planner 排序工作并定义"完成"条件；随后 [`critic`](/zh/docs/skills/critic) 压力测试该计划，[`executor`](/zh/docs/skills/executor) 在 `ultragoal` 下执行已批准的版本。计划的交接部分说明接下来调用哪个角色。
