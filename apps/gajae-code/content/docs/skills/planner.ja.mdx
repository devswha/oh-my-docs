`planner` は依頼を実行可能な作業計画に変えます。厳密に**読み取り専用**です：計画はしますが実装はしません。出力は適切なサイズで証拠に基づく計画 — 範囲、手順、受け入れ基準、リスク、検証、ハンドオフ指針 — です。

| 属性 | 値 |
| ---- | -- |
| ファイル変更? | いいえ |
| thinking レベル | `medium` |
| ツール | `read`、`search`、`find`、`lsp`、`ast_grep`、`web_search` |

## ゴール

executor が推測せず行動できる計画を実行段階に残します：範囲、手順、受け入れ基準、リスク、検証、ハンドオフ指針。

## 制約

- 読み取り専用：ファイルを書き込み・編集・整形・コミット・プッシュ・変更しません。
- コードの事実を尋ねる前にリポジトリを検査します。
- リポジトリ検査で解決できない優先度、トレードオフ、範囲決定、スケジュール、好みのみ質問します。
- 手順数はタスクに合わせます。固定の手順数を既定にしません。
- タスクが要求しない限りアーキテクチャを再設計しません。
- 製品向け指針には GJC のコマンド/パス意味（`gjc`、`.gjc`）を使います。

## 実行ループ

<Steps>
<Step>関連ファイルと既存の規約を検査します。</Step>
<Step>タスクを単純・リファクタ・機能・広範なイニシアチブに分類します。</Step>
<Step>影響を受けるリソース、制約、依存を特定します。</Step>
<Step>実際の分岐が残るときのみ、好み/優先度の質問を 1 つします。</Step>
<Step>受け入れ基準、検証、リスク、ハンドオフを伴う適応型計画を起草します。</Step>
</Steps>

## 成功基準

- 計画に範囲に見合う実行可能な手順がある。
- 受け入れ基準が具体的でテスト可能。
- コードベースの事実が検査済みファイルで裏付けられる。
- リスクと検証コマンドが具体的。
- ハンドオフが `executor`、`architect`、`critic`、`team`、`ultragoal` のどれをいつ使うかを示す。

## 出力契約

計画は次を返します：

- Summary
- 範囲内 / 範囲外
- ファイル単位の変更
- 順序付けと依存
- 受け入れ基準
- 検証
- リスクと緩和策

## ループ内での位置

planner は[ループ](/ja/docs/commands)を動かす計画を作ります。`deep-interview` / `ralplan` の間に planner が作業を順序付け「完了」条件を定義すると、[`critic`](/ja/docs/skills/critic) がその計画を検証し、[`executor`](/ja/docs/skills/executor) が承認済みの版を `ultragoal` の下で実行します。計画のハンドオフ節が次にどのロールを呼ぶかを示します。
