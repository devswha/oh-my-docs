---
title: 快速开始
description: 安装 Gajae-Code，启动 gjc 可执行文件，认证 provider，运行 deep-interview → ralplan → ultragoal 循环。
---

让 `gjc` 运行起来，登录模型 provider，跑通第一个循环。

<Steps>

<Step>

### 前提条件

- 可用 `bun` 执行的 Bun。
- `gjc` 要处理的 Git repository。
- 如果使用推荐的 `gjc --tmux` 体验，请安装 `tmux`。
- 至少一个可访问的 model provider。

OS 与 shell notes 见[推荐环境](/zh/docs/recommended-environment)。

</Step>

<Step>

### 安装

Gajae-Code 通过 npm 注册表以 `gajae-code` 发布。推荐使用 Bun：

```bash
bun install -g gajae-code
```

这会安装 `gjc` 可执行文件，也可使用 scoped 包 `@gajae-code/coding-agent`。

</Step>

<Step>

### 启动

启动推荐的 tmux 体验：

```bash
gjc --tmux
```

直接运行 `gjc` 则不创建或附加 tmux 会话：

```bash
gjc
```

需要更安全的分支级工作区时，在隔离的 Git worktree 中运行。`--worktree` 的值不是文件系统路径，而是 GJC 管理的 sibling worktree 的 branch-like 名称：

```bash
gjc --tmux --worktree my-task-branch
```

如果你已经创建了 worktree 目录，请进入该目录再启动：

```bash
cd ../my-task-worktree && gjc --tmux
```

</Step>

<Step>

### 传入图片

启动时，在本地图片路径前加 `@`：

```bash
gjc @screenshot.png "What should I change?"
```

在 TUI 中，把图片复制到系统剪贴板后使用 **Paste image from clipboard** 键（Linux/macOS 为 `Ctrl+V`，Windows 为 `Alt+V`），或输入 `#paste-image` 并选择 prompt action。剪贴板不可用时，粘贴或传入 `@path/to/image.png`。

</Step>

<Step>

### 认证 provider

Gajae-Code 通过模型 provider 工作，所以选择模型前至少要能连上一个 provider。

最简单的方式是 **API 密钥环境变量**。导出所用 provider 的密钥，`gjc` 会自动识别：

| 变量                   | Provider      |
| ---------------------- | ------------- |
| `ANTHROPIC_API_KEY`    | Anthropic     |
| `OPENAI_API_KEY`       | OpenAI        |
| `GEMINI_API_KEY`       | Google Gemini |
| `OPENROUTER_API_KEY`   | OpenRouter    |
| `ZAI_API_KEY`          | z.ai          |
| `MINIMAX_CODE_API_KEY` | MiniMax Code  |

完整列表见[环境变量](/zh/docs/reference/environment-variables)。

也可用预设**注册自定义 provider**。预设只写入引用文档中环境变量名的 `models.yml` 条目，不存储密钥：

```bash
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

同样的预设可在 TUI 中使用：

```text
/provider add --preset glm
/provider add zai
```

**OAuth provider** 通过浏览器登录而非环境变量。Anthropic、OpenAI Code、Google（Gemini CLI / Antigravity）、GitLab Duo 各自带内置浏览器回调端口；生成的 OAuth 凭据保存在本地 `agent.db`。

`gjc` 解析某 provider 的密钥时按此顺序：

1. CLI `--api-key`
2. 存储的 API 密钥（`agent.db`）
3. 存储的 OAuth 凭据（`agent.db`）
4. 环境变量
5. `models.yml` 的 `apiKey`

随后在 TUI 用 `/model`、在 CLI 用 `--model` / `--models` 选择模型。详见[模型与凭据](/zh/docs/reference/models-and-credentials)。

> 高级：要将 OAuth 刷新令牌从笔记本移到共享主机，参见[模型与凭据](/zh/docs/reference/models-and-credentials)中的 `gjc auth-broker`。

</Step>

<Step>

### 使用 research 模式

如果目标是探索研究而不是直接修改代码，请使用 `rlm`：

```bash
gjc rlm
```

`rlm` 通过持久 Python kernel 和受限工具集打开笔记本式 research/REPL 循环：`python`、`read`、`web_search`、`search_tool_bm25`、read-only `bash`、`goal`、`complete_research`。`notebook.ipynb`、`report.md`、metadata 与 agent-session 产物写入 `.gjc/_session-<gjcSessionId>/rlm/<rlmSessionId>/`。

</Step>

<Step>

### 运行循环

Gajae-Code 专注于一个有用的循环：

```text
deep-interview -> ralplan -> ultragoal
                         └─ 当并行 tmux worker 有帮助时可选 team 执行
```

- `deep-interview` — 厘清意图、影响范围、非目标与验收标准。
- `ralplan` — 在变更前构建并评审计划。
- `$ultragoal` — 推进实现、修订、验证与证据汇总。

只有在并行 tmux worker 有帮助时才加 `$team`。`team` 仅在 tmux 会话内（已设置 `$TMUX`）才工作 —— 请先从 shell 启动 `gjc` CLI，例如 `gjc --tmux`。一个具体的修 bug 流程：

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# 仅并行工作时可选：
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

</Step>

</Steps>
