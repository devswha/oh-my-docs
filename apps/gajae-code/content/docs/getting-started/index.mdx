Get the `gjc` binary running, sign in to a model provider, and drive your first loop.

<Steps>

<Step>

### Install

Gajae-Code is published through the npm registry as `gajae-code`. Use Bun for the recommended runtime workflow:

```bash
bun install -g gajae-code
```

This installs the `gjc` binary. The scoped package is also available as `@gajae-code/coding-agent`.

</Step>

<Step>

### Launch

Start the recommended tmux-backed experience:

```bash
gjc --tmux
```

Bare `gjc` launches directly without creating or attaching a tmux session:

```bash
gjc
```

Run inside an isolated Git worktree when you want a safer, branch-local workspace. Use a dedicated path for throwaway or branch-specific work so the main checkout stays clean:

```bash
gjc --tmux --worktree <path>
```

</Step>

<Step>

### Authenticate a provider

Gajae-Code routes through model providers, so at least one must be reachable before you can pick a model.

**API-key environment variables** are the simplest path. Export the key for the provider you want and `gjc` resolves it automatically:

| Variable             | Provider   |
| -------------------- | ---------- |
| `ANTHROPIC_API_KEY`  | Anthropic  |
| `OPENAI_API_KEY`     | OpenAI     |
| `GEMINI_API_KEY`     | Google Gemini |
| `OPENROUTER_API_KEY` | OpenRouter |
| `ZAI_API_KEY`        | z.ai       |
| `MINIMAX_CODE_API_KEY` | MiniMax Code |

See [Environment variables](/docs/reference/environment-variables) for the full list.

**Register a custom provider** with a preset, which writes a `models.yml` entry referencing a documented env var name (it does not store the secret):

```bash
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

The same presets are available inside the TUI:

```text
/provider add --preset glm
/provider add zai
```

**OAuth providers** sign in through your browser instead of an env var. Anthropic, OpenAI Code, Google (Gemini CLI / Antigravity), and GitLab Duo each have a built-in browser callback port; the resulting OAuth credentials are stored locally in `agent.db`.

When `gjc` resolves a key for a provider, it uses this order:

1. CLI `--api-key`
2. Stored API key (`agent.db`)
3. Stored OAuth credential (`agent.db`)
4. Environment variable
5. `models.yml` `apiKey`

Then pick a model with `/model` in the TUI, or `--model` / `--models` on the CLI. For details, see [Models and credentials](/docs/reference/models-and-credentials).

> Advanced: to move OAuth refresh tokens off your laptop onto a shared host, see `gjc auth-broker` in [Models and credentials](/docs/reference/models-and-credentials).

</Step>

<Step>

### Run the loop

Gajae-Code focuses on one useful loop:

```text
deep-interview -> ralplan -> ultragoal
                         â””â”€ optional team execution when parallel tmux workers help
```

- `deep-interview` â€” clarify intent, affected behavior, non-goals, and acceptance checks.
- `ralplan` â€” build and critique the plan before any mutation.
- `$ultragoal` â€” carry the work through implementation, revision, verification, and an evidence summary.

Add `$team` only when coordinated parallel tmux workers help. A concrete bug-fix pass might look like:

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# Optional only for parallel work:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

</Step>

## Skills

Gajae-Code ships exactly four workflow skills â€” no sprawling default skill zoo. Invoke each one inside a GJC session as `/skill:<name>`.

| Skill            | What it does                                                                        |
| ---------------- | ----------------------------------------------------------------------------------- |
| `deep-interview` | Removes ambiguity before planning or code changes.                                  |
| `ralplan`        | Builds and critiques a plan before mutation.                                        |
| `team`           | Optionally coordinates tmux-backed parallel execution when multiple workers help.   |
| `ultragoal`      | Tracks durable goals through implementation, revisions, verification, and evidence. |

The four bundled role agents â€” `executor`, `architect`, `planner`, and `critic` â€” carry these skills out. See [Workflows](/docs/workflows) for each skill and [Agents](/docs/agents) for the roles.

## Configuration

Provider retry budgets and other settings live in `~/.gjc/config.yml` (or the active project/user settings source):

```yaml
retry:
  # Similar to codex-cli request_max_retries. Counts retries, not the initial request.
  requestMaxRetries: 4
  # Similar to codex-cli stream_max_retries. Counts replay-safe stream retries.
  streamMaxRetries: 100
  # Session-level terminal-error retries remain separately configurable.
  maxRetries: 3
  maxDelayMs: 300000
```

Gajae-Code has two retry layers: session auto-retry (`retry.maxRetries`) retries a failed assistant turn after a terminal transient error, while provider retry budgets (`requestMaxRetries`, `streamMaxRetries`) control retries inside the provider transport before that terminal error reaches the session. Invalid auth, unsupported models/providers, malformed requests, context overflow, user aborts, and permanent quota failures stay fail-fast instead of being hidden by retry loops.
