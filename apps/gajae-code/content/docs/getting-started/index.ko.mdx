---
title: 시작하기
description: Gajae-Code를 설치하고, gjc 바이너리를 실행하고, 프로바이더에 인증한 뒤 deep-interview → ralplan → ultragoal 루프를 돌립니다.
---

`gjc` 바이너리를 띄우고, 모델 프로바이더에 로그인한 다음, 첫 루프를 돌려봅니다.

<Steps>

<Step>

### 설치

Gajae-Code는 npm 레지스트리에 `gajae-code`로 배포됩니다. 권장 런타임은 Bun입니다.

```bash
bun install -g gajae-code
```

`gjc` 바이너리가 설치됩니다. 스코프 패키지 `@gajae-code/coding-agent`로도 받을 수 있습니다.

</Step>

<Step>

### 실행

권장하는 tmux 기반 환경으로 시작합니다.

```bash
gjc --tmux
```

`gjc`만 실행하면 tmux 세션을 만들거나 붙지 않고 바로 켜집니다.

```bash
gjc
```

브랜치 단위로 더 안전하게 작업하려면 격리된 Git worktree에서 실행하세요. 일회성·브랜치 전용 작업엔 별도 경로를 써서 메인 체크아웃을 깨끗이 유지합니다.

```bash
gjc --tmux --worktree <path>
```

</Step>

<Step>

### 프로바이더 인증

Gajae-Code는 모델 프로바이더를 통해 동작하므로, 모델을 고르기 전에 적어도 하나의 프로바이더에 닿을 수 있어야 합니다.

가장 간단한 방법은 **API 키 환경 변수**입니다. 쓰려는 프로바이더의 키를 export 하면 `gjc`가 자동으로 인식합니다.

| 변수                   | 프로바이더    |
| ---------------------- | ------------- |
| `ANTHROPIC_API_KEY`    | Anthropic     |
| `OPENAI_API_KEY`       | OpenAI        |
| `GEMINI_API_KEY`       | Google Gemini |
| `OPENROUTER_API_KEY`   | OpenRouter    |
| `ZAI_API_KEY`          | z.ai          |
| `MINIMAX_CODE_API_KEY` | MiniMax Code  |

전체 목록은 [환경 변수](/ko/docs/reference/environment-variables)를 참고하세요.

프리셋으로 **커스텀 프로바이더를 등록**할 수도 있습니다. 프리셋은 문서화된 환경 변수 이름을 참조하는 `models.yml` 항목만 작성하며, 비밀 값을 저장하지는 않습니다.

```bash
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

같은 프리셋을 TUI 안에서도 쓸 수 있습니다.

```text
/provider add --preset glm
/provider add zai
```

**OAuth 프로바이더**는 환경 변수 대신 브라우저로 로그인합니다. Anthropic, OpenAI Code, Google(Gemini CLI / Antigravity), GitLab Duo는 각각 내장된 브라우저 콜백 포트를 가지며, 발급된 OAuth 자격 증명은 로컬 `agent.db`에 저장됩니다.

`gjc`가 프로바이더의 키를 해석할 때는 다음 순서를 따릅니다.

1. CLI `--api-key`
2. 저장된 API 키 (`agent.db`)
3. 저장된 OAuth 자격 증명 (`agent.db`)
4. 환경 변수
5. `models.yml`의 `apiKey`

그런 다음 TUI에서는 `/model`로, CLI에서는 `--model` / `--models`로 모델을 고릅니다. 자세한 내용은 [모델과 자격 증명](/ko/docs/reference/models-and-credentials)을 참고하세요.

> 고급: OAuth 리프레시 토큰을 노트북에서 공유 호스트로 옮기려면 [모델과 자격 증명](/ko/docs/reference/models-and-credentials)의 `gjc auth-broker`를 참고하세요.

</Step>

<Step>

### 루프 돌리기

Gajae-Code는 하나의 쓸모 있는 루프에 집중합니다.

```text
deep-interview -> ralplan -> ultragoal
                         └─ 병렬 tmux 워커가 도움이 될 때만 team 실행
```

- `deep-interview` — 의도, 영향 범위, 비목표, 수용 기준을 명확히 합니다.
- `ralplan` — 변경 전에 계획을 세우고 비평합니다.
- `$ultragoal` — 구현·수정·검증·증거 요약까지 작업을 끌고 갑니다.

병렬 tmux 워커가 도움이 될 때만 `$team`을 더하세요. 구체적인 버그 수정 흐름은 이런 모습입니다.

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# 병렬 작업일 때만 선택:
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

</Step>

</Steps>

## 스킬

Gajae-Code는 정확히 네 개의 워크플로 스킬만 제공합니다. 방대한 기본 스킬 동물원은 없습니다. 각 스킬은 GJC 세션 안에서 `/skill:<name>`으로 호출합니다.

| 스킬             | 하는 일                                                              |
| ---------------- | -------------------------------------------------------------------- |
| `deep-interview` | 계획이나 코드 변경 전에 모호함을 제거합니다.                         |
| `ralplan`        | 변경 전에 계획을 세우고 비평합니다.                                  |
| `team`           | 여러 워커가 도움이 될 때 tmux 기반 병렬 실행을 선택적으로 조율합니다. |
| `ultragoal`      | 구현, 수정, 검증, 증거를 거치며 지속 가능한 goal을 추적합니다.       |

번들된 네 개의 역할 에이전트 `executor`, `architect`, `planner`, `critic`이 이 스킬들을 수행합니다. 각 스킬은 [워크플로우](/ko/docs/workflows)에서, 역할은 [에이전트](/ko/docs/agents)에서 확인하세요.

## 설정

provider 재시도 예산 등 설정은 `~/.gjc/config.yml`(또는 활성 프로젝트/사용자 설정 소스)에 둡니다.

```yaml
retry:
  # codex-cli의 request_max_retries와 비슷. 최초 요청이 아닌 재시도 횟수를 셉니다.
  requestMaxRetries: 4
  # codex-cli의 stream_max_retries와 비슷. replay 가능한 스트림 재시도 횟수를 셉니다.
  streamMaxRetries: 100
  # 세션 레벨 터미널 오류 재시도는 별도로 설정합니다.
  maxRetries: 3
  maxDelayMs: 300000
```

Gajae-Code에는 재시도 계층이 둘 있습니다. 세션 자동 재시도(`retry.maxRetries`)는 터미널 일시 오류 이후 실패한 어시스턴트 턴을 다시 시도하고, provider 재시도 예산(`requestMaxRetries`, `streamMaxRetries`)은 그 터미널 오류가 세션에 도달하기 전 provider 전송 계층 내부의 재시도를 제어합니다. 잘못된 인증, 미지원 모델/provider, 잘못된 요청, 컨텍스트 초과, 사용자 중단, 영구 쿼터 실패는 재시도 루프에 가려지지 않고 즉시 실패(fail-fast)합니다.
