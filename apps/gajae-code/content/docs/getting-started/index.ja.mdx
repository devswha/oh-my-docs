`gjc` バイナリを動かし、最初のループを回します。

<Steps>

<Step>

### インストール

Gajae-Code は npm レジストリに `gajae-code` として公開されています。推奨ランタイムは Bun です：

```bash
bun install -g gajae-code
```

`gjc` バイナリがインストールされます。scoped パッケージ `@gajae-code/coding-agent` も利用できます。

</Step>

<Step>

### 起動

推奨の tmux ベース環境で起動します：

```bash
gjc --tmux
```

`gjc` だけなら tmux セッションを作成・接続せずに直接起動します：

```bash
gjc
```

ブランチ単位で安全に作業したい場合は、隔離された Git worktree で実行します。使い捨てやブランチ専用の作業には専用のパスを使い、メインのチェックアウトをきれいに保ちます：

```bash
gjc --tmux --worktree <path>
```

</Step>

<Step>

### ループを回す

Gajae-Code は 1 つの有用なループに集中します：

```text
deep-interview -> ralplan -> ultragoal
                         └─ 並列 tmux ワーカーが役立つ場合は team 実行を選択
```

- `deep-interview` — 意図、影響範囲、非目標、受け入れ基準を明確化。
- `ralplan` — 変更前に計画を立て批評。
- `$ultragoal` — 実装・改訂・検証・証拠のまとめまで進める。

並列 tmux ワーカーが役立つ場合のみ `$team` を加えます。具体的なバグ修正の流れは次のとおりです：

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# 並列作業のときのみ任意：
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

</Step>

</Steps>

## 設定

provider のリトライ予算などの設定は `~/.gjc/config.yml`（または有効なプロジェクト/ユーザー設定ソース）にあります：

```yaml
retry:
  # codex-cli の request_max_retries に類似。初回リクエストではなくリトライ回数を数える。
  requestMaxRetries: 4
  # codex-cli の stream_max_retries に類似。リプレイ可能なストリームリトライを数える。
  streamMaxRetries: 100
  # セッションレベルの終端エラーリトライは別途設定可能。
  maxRetries: 3
  maxDelayMs: 300000
```

Gajae-Code には 2 つのリトライ層があります。セッション自動リトライ（`retry.maxRetries`）は終端的な一時エラー後に失敗したアシスタントターンを再試行し、provider リトライ予算（`requestMaxRetries`、`streamMaxRetries`）はその終端エラーがセッションに到達する前に provider 転送層内部のリトライを制御します。無効な認証、未対応のモデル/provider、不正なリクエスト、コンテキスト超過、ユーザーによる中断、恒久的なクォータ失敗は、リトライループに隠されず即座に失敗（fail-fast）します。
