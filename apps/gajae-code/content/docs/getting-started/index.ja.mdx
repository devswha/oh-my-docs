`gjc` バイナリを動かし、モデル provider にサインインして、最初のループを回します。

<Steps>

<Step>

### 前提

- `bun` として実行できる Bun。
- `gjc` が作業する Git repository。
- 推奨の `gjc --tmux` 体験を使う場合は `tmux`。
- 到達可能な model provider が 1 つ以上。

OS と shell notes は[推奨環境](/ja/docs/recommended-environment)を参照してください。

</Step>

<Step>

### インストール

Gajae-Code は npm レジストリに `gajae-code` として公開されています。推奨ランタイムは Bun です：

```bash
bun install -g gajae-code
```

`gjc` バイナリがインストールされます。scoped パッケージ `@gajae-code/coding-agent` も利用できます。

</Step>

<Step>

### 起動

推奨の tmux ベース環境で起動します：

```bash
gjc --tmux
```

`gjc` だけなら tmux セッションを作成・接続せずに直接起動します：

```bash
gjc
```

ブランチ単位で安全に作業したい場合は、隔離された Git worktree で実行します。`--worktree` の値はファイルシステムパスではなく、GJC が管理する sibling worktree の branch-like 名です：

```bash
gjc --tmux --worktree my-task-branch
```

既存の worktree ディレクトリがある場合は、そのディレクトリに移動して起動します：

```bash
cd ../my-task-worktree && gjc --tmux
```

</Step>

<Step>

### 画像を渡す

起動時はローカル画像パスの前に `@` を付けます：

```bash
gjc @screenshot.png "What should I change?"
```

TUI 内では画像をシステムクリップボードへコピーし、**Paste image from clipboard** キー（Linux/macOS は `Ctrl+V`、Windows は `Alt+V`）を使うか、`#paste-image` と入力して prompt action を選びます。クリップボードが使えない場合は `@path/to/image.png` を貼り付けるか渡してください。

</Step>

<Step>

### provider を認証する

Gajae-Code はモデル provider 経由で動作するため、モデルを選ぶ前に少なくとも 1 つの provider に到達できる必要があります。

最も簡単なのは **API キーの環境変数**です。使いたい provider のキーを export すれば `gjc` が自動的に解決します：

| 変数                   | Provider      |
| ---------------------- | ------------- |
| `ANTHROPIC_API_KEY`    | Anthropic     |
| `OPENAI_API_KEY`       | OpenAI        |
| `GEMINI_API_KEY`       | Google Gemini |
| `OPENROUTER_API_KEY`   | OpenRouter    |
| `ZAI_API_KEY`          | z.ai          |
| `MINIMAX_CODE_API_KEY` | MiniMax Code  |

全リストは[環境変数](/ja/docs/reference/environment-variables)を参照してください。

プリセットで**カスタム provider を登録**することもできます。プリセットは文書化された環境変数名を参照する `models.yml` エントリを書き込むだけで、シークレットは保存しません：

```bash
gjc setup provider --preset minimax
gjc setup provider --preset minimax-cn
gjc setup provider --preset glm
```

同じプリセットは TUI でも使えます：

```text
/provider add --preset glm
/provider add zai
```

**OAuth provider** は環境変数ではなくブラウザでサインインします。Anthropic、OpenAI Code、Google（Gemini CLI / Antigravity）、GitLab Duo はそれぞれ組み込みのブラウザコールバックポートを持ち、発行された OAuth 認証情報はローカルの `agent.db` に保存されます。

`gjc` が provider のキーを解決するときは次の順序です：

1. CLI `--api-key`
2. 保存された API キー（`agent.db`）
3. 保存された OAuth 認証情報（`agent.db`）
4. 環境変数
5. `models.yml` の `apiKey`

その後、TUI では `/model`、CLI では `--model` / `--models` でモデルを選びます。詳細は[モデルと認証情報](/ja/docs/reference/models-and-credentials)を参照してください。

> 上級者向け：OAuth リフレッシュトークンをラップトップから共有ホストへ移すには、[モデルと認証情報](/ja/docs/reference/models-and-credentials)の `gjc auth-broker` を参照してください。

</Step>

<Step>

### research モードを使う

直接のコード変更ではなく探索調査が目的なら `rlm` を使います：

```bash
gjc rlm
```

`rlm` は永続 Python カーネルと、`python`、`read`、`web_search`、`search_tool_bm25`、read-only `bash`、`goal`、`complete_research` に制限されたツールセットでノートブック形式の research/REPL ループを開きます。`notebook.ipynb`、`report.md`、metadata、agent-session アーティファクトは `.gjc/_session-<gjcSessionId>/rlm/<rlmSessionId>/` 配下に記録されます。

</Step>

<Step>

### ループを回す

Gajae-Code は 1 つの有用なループに集中します：

```text
deep-interview -> ralplan -> ultragoal
                         └─ 並列 tmux ワーカーが役立つ場合は team 実行を選択
```

- `deep-interview` — 意図、影響範囲、非目標、受け入れ基準を明確化。
- `ralplan` — 変更前に計画を立て批評。
- `$ultragoal` — 実装・改訂・検証・証拠のまとめまで進める。

並列 tmux ワーカーが役立つ場合のみ `$team` を加えます。`team` は tmux セッション内（`$TMUX` が設定済み）でのみ動作します — まずシェルから `gjc` CLI を起動してください。例：`gjc --tmux`。具体的なバグ修正の流れは次のとおりです：

```text
/skill:deep-interview clarify the bug, affected behavior, non-goals, and acceptance checks
/skill:ralplan turn the clarified bug report into a reviewed fix plan
gjc ultragoal create-goals --brief-file <approved-plan>
# 並列作業のときのみ任意：
gjc team 2:executor "split implementation and verification for this bug fix"
gjc ultragoal complete-goals
```

</Step>

</Steps>
