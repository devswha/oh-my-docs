Gajae-Code は境界が明確なロールエージェントをちょうど 4 つ同梱します。ワークフロー skill ではなく、委譲用のタスクロールエージェントです。2 つのルールがこれらを律します：**1 つのロールだけがファイルを変更し、残り 3 つは厳密に読み取り専用です。**

## 4 つのロール

| エージェント | 役割                                | 変更?  | thinking |
| ----------- | ----------------------------------- | ------ | -------- |
| `executor`  | 範囲の定まった実装・修正・リファクタ。 | **はい** | medium   |
| `architect` | 読み取り専用のアーキテクチャ・レビュー。| いいえ | high     |
| `planner`   | 読み取り専用の順序付け・受け入れ基準。 | いいえ | medium   |
| `critic`    | 読み取り専用の計画批評・実行可能性審査。| いいえ | high     |

読み取り専用ロールはファイルを書き込み・編集・整形・コミット・プッシュ・変更しません。その出力は判断 — 評価、計画、裁定 — です。`executor` のみがファイルを変更し、それも無制限ではなく計画段階で設定された範囲内で動作します。

## ループとの対応

ロールエージェントは[ワークフローループ](/ja/docs/workflows)に対応します：

- **planner** は `deep-interview` / `ralplan` で意図を明示的な受け入れ基準を持つ順序付き計画に変えます。
- **critic** は変更が始まる前にその計画を検証し、`OKAY`、`ITERATE`、`REJECT` のいずれかを返します。
- **executor** は `ultragoal` の下で承認済みの作業を実行し、diff を小さく保ち証拠を残します。
- **architect** は設計や変更に評価が必要なとき、独立した読み取り専用レビュー（`CLEAR` / `WATCH` / `BLOCK` と `APPROVE` / `COMMENT` / `REQUEST CHANGES`）を提供します。

これらのロールの並列実行は `team` で調整できます。

## 読み取り専用と書き込み

この分離こそ要点です。計画・批評・レビューを変更から切り離すため、ファイルが変わる前に計画が検証され、レビュアーが「ついでに直す」ことなく変更がレビューされます。委譲時は最も適した最も狭いロールを選びます：評価や計画には読み取り専用ロールを、実際に何かを変える必要があるときだけ `executor` を。
