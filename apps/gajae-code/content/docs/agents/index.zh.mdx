Gajae-Code 恰好内置四个边界清晰的角色智能体。它们是用于委派的任务角色智能体，不是工作流 skill。两条规则约束它们：**只有一个角色改动文件，其余三个严格只读。**

## 四个角色

| 智能体      | 角色                            | 修改?  | thinking |
| ----------- | ------------------------------- | ------ | -------- |
| `executor`  | 有边界的实现、修复与重构。       | **是** | medium   |
| `architect` | 只读的架构与代码评审。           | 否     | high     |
| `planner`   | 只读的排序与验收标准。           | 否     | medium   |
| `critic`    | 只读的计划评审与可执行性审查。   | 否     | high     |

只读角色从不写入、编辑、格式化、提交、推送或改动文件。它们的产出是判断——评估、计划与裁决。只有 `executor` 改动文件，且在规划阶段设定的范围内进行，而非无限开放。

## 在循环中的对应

角色智能体对应[工作流循环](/zh/docs/workflows)：

- **planner** 在 `deep-interview` / `ralplan` 中把意图转为带明确验收标准的有序计划。
- **critic** 在任何变更前压力测试该计划，返回 `OKAY`、`ITERATE` 或 `REJECT`。
- **executor** 在 `ultragoal` 下执行已批准的工作，保持 diff 精简并留下证据。
- **architect** 在设计或变更需要评估时提供独立只读评审（`CLEAR` / `WATCH` / `BLOCK` 及 `APPROVE` / `COMMENT` / `REQUEST CHANGES`）。

这些角色的并行运行可用 `team` 协调。

## 只读与写入

这种分离正是要点。计划、评审与复审与变更隔离，因此文件变更前计划已被压力测试，变更被复审时复审者无法"顺手就改"。委派时选最贴合的最窄角色：评估或计划用只读角色，只有确实要改动时才用 `executor`。
