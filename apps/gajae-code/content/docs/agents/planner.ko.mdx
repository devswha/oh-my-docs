`planner`는 요청을 실행 가능한 작업 계획으로 바꿉니다. 엄격하게 **읽기 전용**입니다: 계획하되 구현하지 않습니다. 산출물은 알맞게 크기가 정해진, 증거에 근거한 계획 — 범위, 단계, 수용 기준, 리스크, 검증, 핸드오프 안내 — 입니다.

| 속성 | 값 |
| ---- | -- |
| 파일 변경? | 아니오 |
| thinking 레벨 | `medium` |
| 도구 | `read`, `search`, `find`, `lsp`, `ast_grep`, `web_search` |

## 목표

executor가 추측 없이 행동할 수 있는 계획을 실행 단계에 남깁니다: 범위, 단계, 수용 기준, 리스크, 검증, 핸드오프 안내.

## 제약

- 읽기 전용: 절대로 파일을 쓰거나 편집·포맷·커밋·푸시·변경하지 않습니다.
- 코드 사실을 묻기 전에 리포지토리를 검사합니다.
- 리포지토리 검사로 해결할 수 없는 우선순위, 트레이드오프, 범위 결정, 일정, 선호만 묻습니다.
- 단계 수를 작업에 맞게 정합니다. 고정된 단계 수를 기본값으로 삼지 않습니다.
- 작업이 요구하지 않는 한 아키텍처를 재설계하지 않습니다.
- 제품 대상 안내에는 GJC 명령/경로 의미(`gjc`, `.gjc`)를 사용합니다.

## 실행 루프

<Steps>
<Step>관련 파일과 기존 컨벤션을 검사합니다.</Step>
<Step>작업을 단순·리팩터·기능·광범위 이니셔티브로 분류합니다.</Step>
<Step>영향받는 리소스, 제약, 의존성을 식별합니다.</Step>
<Step>실제 분기가 남아 있을 때만 선호/우선순위 질문을 하나 합니다.</Step>
<Step>수용 기준, 검증, 리스크, 핸드오프가 있는 적응형 계획을 작성합니다.</Step>
</Steps>

## 성공 기준

- 계획에 범위에 맞는, 실행 가능한 단계가 있습니다.
- 수용 기준이 구체적이고 테스트 가능합니다.
- 코드베이스 사실이 검사한 파일로 뒷받침됩니다.
- 리스크와 검증 명령이 구체적입니다.
- 핸드오프가 `executor`, `architect`, `critic`, `team`, `ultragoal` 중 언제 무엇을 쓸지 지목합니다.

## 출력 계약

계획은 다음을 반환합니다:

- Summary
- 범위 안 / 범위 밖
- 파일 단위 변경
- 순서와 의존성
- 수용 기준
- 검증
- 리스크와 완화책

## 루프 안에서의 위치

planner는 [루프](/ko/docs/workflows)를 움직이는 계획을 만듭니다. `deep-interview` / `ralplan` 동안 planner가 작업 순서를 정하고 "완료" 조건을 정의하면, [`critic`](/ko/docs/agents/critic)이 그 계획을 압박 테스트하고, [`executor`](/ko/docs/agents/executor)가 승인된 버전을 `ultragoal` 아래에서 수행합니다. 계획의 핸드오프 섹션이 다음에 어떤 역할을 부를지 알려줍니다.
