Gajae-Code는 경계가 분명한 역할 에이전트 네 가지를 번들합니다. 둘은 쓰고, 둘은 읽기 전용입니다.

| 에이전트    | 역할                                     | 변경? |
| ----------- | ---------------------------------------- | ----- |
| `executor`  | 범위가 정해진 구현·수정·리팩터링.        | 예    |
| `architect` | 읽기 전용 아키텍처·코드 리뷰 평가.       | 아니오 |
| `planner`   | 읽기 전용 순서 설계·수용 기준 정의.      | 아니오 |
| `critic`    | 읽기 전용 계획 비평·실행 가능성 검토.    | 아니오 |

## executor

범위가 정해진 구현을 합니다 — 코드를 쓰고, 수정을 적용하고, 리팩터링을 돌립니다. 파일을 바꾸는 유일한 역할이며, 무제한이 아니라 계획 단계에서 정한 범위 안에서 일합니다.

## architect

아키텍처와 코드 리뷰를 위한 읽기 전용 평가 역할입니다. 파일을 건드리지 않고 설계나 기존 코드베이스를 평가합니다 — 산출물은 편집이 아니라 판단입니다.

## planner

작업 순서를 설계하고 수용 기준을 정의하는 읽기 전용 역할입니다. 의도를 명시적 "완료" 조건이 있는 정렬된 계획으로 바꾸고, 그 계획을 `ralplan`이 비평합니다.

## critic

변경 전에 계획의 타당성과 실행 가능성을 비평하는 읽기 전용 역할입니다. 루프 안의 회의론자로서 빈틈, 위험, 적힌 대로는 실제로 실행되지 않을 단계를 찾습니다.

## 루프 안에서의 위치

역할 에이전트는 [워크플로우 루프](/ko/docs/workflows)에 매핑됩니다: planner와 critic은 `deep-interview` / `ralplan`에서 계획을 다듬고 압박 검증하고, executor는 `ultragoal`에서 그것을 수행하고, architect는 설계나 변경에 독립적 평가가 필요할 때 읽기 전용 리뷰를 제공합니다. 이 역할들의 병렬 실행은 `team`으로 조율할 수 있습니다.
