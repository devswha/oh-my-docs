メモリは Gajae-Code が手作業なしにセッションをまたいで知識を引き継ぐための機能です。有効にすると、エージェントは過去のセッションから永続的なシグナルを **保持（retain）** し、プロジェクト範囲のコンパクトな要約を新しいセッションごとに **呼び戻し（recall）** ます — 技術的決定、繰り返すワークフロー、落とし穴など、毎回再発見せずに済みます。

メモリは **デフォルト無効** です。`/settings` または `config.yml` で有効にします：

```yaml
memories:
  enabled: true
```

## メモリのループ上の位置

メモリはセッションを駆動するのではなく、その前後を挟みます：

- **呼び戻し（セッション開始）** — 現在のプロジェクトのメモリ要約があれば、システムプロンプトに **Memory Guidance** ブロックとして注入されます。
- **保持（バックグラウンド）** — 起動時（または手動）にパイプラインが走り、過去のセッションから知識を抽出し、次回のためにディスクへ統合します。

メモリは権威ある状態ではなく **ヒューリスティックなコンテキスト** として扱われます。エージェントは次のように指示されます：

- メモリは現在のリポジトリ状態の真実源ではなく、プロセスや過去の決定に使う、
- メモリが計画を変えるときはメモリ成果物のパスを引用し、行動前に現在のリポジトリ証拠と組み合わせる、
- リポジトリ状態やユーザー指示と矛盾するときは後者を優先し、矛盾するメモリは古いものとみなす。

## メモリ成果物の読み取り

エージェントは `read` ツールと `memory://` URL でメモリファイルを直接読めます：

| URL                                    | 内容                       |
| -------------------------------------- | -------------------------- |
| `memory://root`                        | 起動時に注入される要約      |
| `memory://root/MEMORY.md`              | 長期メモリの全文書          |
| `memory://root/skills/<name>/SKILL.md` | 生成された skill プレイブック |

## `/memory` スラッシュコマンド

| サブコマンド          | 効果                                |
| --------------------- | ----------------------------------- |
| `view`                | 現在のメモリ注入ペイロードを表示      |
| `clear` / `reset`     | すべてのメモリデータと生成成果物を削除 |
| `enqueue` / `rebuild` | 次回起動時に統合を強制実行            |

## 保持の仕組み

保持パイプラインは起動時、またはスラッシュコマンドで 2 フェーズで実行されます。

<Steps>
<Step>
**セッションごとの抽出。** 前回処理以降に変更された各過去セッションについて、モデルがセッション履歴を読み、永続的シグナル（技術的決定、制約、解決済みの失敗、繰り返すワークフロー）を抽出します。新しすぎる・古すぎる・現在アクティブなセッションはスキップされます。各抽出は生のメモリブロックと短い概要を生成します。
</Step>
<Step>
**統合。** 2 回目のモデルパスがすべてのセッション別抽出を読み、3 つの出力をディスクに書き込みます：`MEMORY.md`（厳選された長期メモリ文書）、`memory_summary.md`（セッション開始時に注入されるコンパクトなテキスト）、`skills/`（再利用可能な手順プレイブック、各々独立したサブディレクトリ）。このフェーズは同時プロセスの二重実行を防ぐ lease を用い、過去実行の古い skill ディレクトリは自動的に整理されます。
</Step>
</Steps>

すべての出力はディスクへの書き込み前に秘密情報（secret）のスキャンを受けます。

## メモリを実行するモデル

メモリは[モデルロールシステム](/ja/docs/concepts/models-and-providers)に相乗りします：

| フェーズ            | ロール    | 目的                |
| ------------------- | --------- | ------------------- |
| 抽出（フェーズ 1）  | `default` | セッション別知識抽出 |
| 統合（フェーズ 2）  | `smol`    | セッション横断の統合 |

要求したロールが未設定なら、メモリモデルの解決は `default`、アクティブなセッションモデル、registry の最初のモデルの順にフォールバックします。

## 設定

| 設定                                   | デフォルト | 説明                                       |
| -------------------------------------- | --------- | ------------------------------------------ |
| `memories.enabled`                     | `false`   | マスタースイッチ                            |
| `memories.maxRolloutAgeDays`           | `30`      | これより古いセッションは処理しない           |
| `memories.minRolloutIdleHours`         | `12`      | これより最近アクティブなセッションはスキップ |
| `memories.maxRolloutsPerStartup`       | `64`      | 1 回の起動で処理するセッション数の上限       |
| `memories.summaryInjectionTokenLimit`  | `5000`    | システムプロンプトに注入する要約の最大トークン数 |

並行性、lease 期間、トークン予算などの追加チューニングノブが上級者向けに config にあります。全体は[設定](/ja/docs/reference/configuration)を参照してください。
