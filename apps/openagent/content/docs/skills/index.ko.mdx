Skill은 재사용 가능한 instruction pack입니다. 모든 에이전트 프롬프트를 영구히 부풀리지 않고 워크플로우, 제약, 예시, MCP 도구 접근을 더해줍니다.

## Skill이 하는 일

<Cards>
  <Card title="Built-in skill">browser QA, git discipline, review, AI-slop 제거 같은 일반 워크플로우.</Card>
  <Card title="User skill">OpenCode config 아래에 둔 개인 convention.</Card>
  <Card title="Project skill">`.opencode/skills/` 아래의 repo 전용 워크플로우.</Card>
  <Card title="MCP 포함 skill">필요할 때만 로드되는 도구 서버와 함께 들어오는 instruction.</Card>
</Cards>

## Category + Skill 패턴

Category는 모델과 프로필을 고릅니다. Skill은 도메인 playbook을 더합니다.

```ts
task({
  category: "writing",
  load_skills: ["sync-openagent-docs"],
  run_in_background: false,
  prompt: "Refresh docs from upstream README and docs/reference facts.",
});
```

원칙은 단순합니다.

- UI 작업은 diff가 작아 보여도 `visual-engineering`을 쓰세요.
- task domain과 겹치는 skill은 모두 load하세요.
- project/user skill이 있으면 built-in보다 먼저 보세요.
- skill 프롬프트는 구체적으로: context, goal, downstream use, request.

## 문서 동기화 Skill

이 docs site에는 `sync-openagent-docs`가 들어 있습니다. upstream `code-yeongyu/oh-my-openagent`와 컨텐츠를 맞추는 project skill입니다.

이 skill은 다음 에이전트에게 이렇게 요구합니다.

- 수정 전에 upstream README, docs, package metadata, 관련 `.opencode/skills` 파일을 읽기.
- 패키지 이름 전환을 그대로 보존: 기본은 `oh-my-openagent`를 쓰고, `oh-my-opencode`는 legacy 호환 alias로만 언급합니다.
- 아키텍처 경계를 짚는 다이어그램이 아니라면 사용을 줄이기.
- 영어와 한국어를 먼저 갱신하고, 일본어와 중국어 라우트는 깨지지 않게 유지.
- lint, build, route smoke check로 검증.

## 좋은 Project Skill 쓰는 법

| Section | 이유 |
|---|---|
| Purpose | skill이 언제 load되어야 하는지 정합니다. |
| Required sources | stale docs와 추측을 막습니다. |
| Workflow | 에이전트가 같은 길을 반복할 수 있게 합니다. |
| Guardrails | naming, safety, verification 규칙을 잡습니다. |
| Output expectations | 완료 여부를 관찰 가능하게 만듭니다. |
