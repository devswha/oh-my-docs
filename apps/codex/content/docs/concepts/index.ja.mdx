## OMX の構成要素

OMX は 4 つのシステムが噛み合って動きます。

<Mermaid chart={`
graph TD
    A["ユーザー入力"] --> B["フック (Hooks)\nイベント検知とルーティング"]
    B --> C["スキル (Skills)\n挙動の注入とワークフロー合成"]
    C --> D["エージェント (Agents)\n専門タスクの実行"]
    D --> E["ステート (State)\n進捗と知識の保持"]
`} />

### 各コンセプトの要約

#### エージェント (Agents)

OMX は 33 個の役割別エージェントを提供します。各エージェントは 1 つの責務に集中します。

例: `explore`(探索)、`planner`(計画)、`architect`(設計)、`executor`(実装)、`verifier`(検証)

#### スキル (Skills)

スキルは、エージェントの挙動を組み合わせて拡張するワークフローの入口です。`$autopilot`、`$ralph`、`$team` のように呼び出します。

例: `$deep-interview`(要件の明確化)、`$ralplan`(合意ベースの計画)、`$ralph`(完了まで継続実行)

#### フック (Hooks)

フックは Codex CLI のライフサイクルイベントに反応してスキルとランタイム挙動を起動します。セッション開始、ユーザー入力、ツール実行、停止時点のそれぞれで介入できます。

#### ステート (State)

作業進捗とセッション間の記憶は `.omx/` ディレクトリ以下に保存されます。そのおかげで context compaction の後でも重要な情報が残ります。

## エージェントとは?

エージェントは特定の役割に特化した作業 surface です。OMX はこれらを 4 レーンに分類します。

<Mermaid chart={`
graph LR
    A["explore\nコードベース探索"] --> B["analyst\n要件分析"]
    B --> C["planner / architect\n計画と設計"]
    C --> D["critic\n計画レビュー"]
    D --> E["executor\n実装"]
    E --> F["verifier\n完了検証"]
`} />

### Build / Analysis レーン

要件の理解から実装、検証まで、開発の中心フローを担当します。

| エージェント | 役割 |
|----------|------|
| `explore` | コードベース探索、ファイル/パターン/関係の把握 |
| `analyst` | 要件分析、隠れた制約の発見 |
| `planner` | タスクの順序と計画立案 |
| `architect` | システム設計、インターフェース、トレードオフ分析 |
| `debugger` | 根本原因分析 |
| `executor` | コード実装、リファクタリング |
| `verifier` | 完了検証、テスト妥当性の確認 |

### Review レーン

実装結果を引き渡す前に品質とリスクを点検します。

| エージェント | 役割 |
|----------|------|
| `quality-reviewer` | ロジック欠陥、保守性、アンチパターンのレビュー |
| `security-reviewer` | 脆弱性、信頼境界、auth のレビュー |
| `code-reviewer` | 総合コードレビュー |
| `style-reviewer` | スタイル、命名、フォーマットのレビュー |
| `api-reviewer` | API 契約と後方互換性の確認 |
| `performance-reviewer` | 性能ボトルネックとコスト確認 |

### Domain レーン

特定領域に強いスペシャリスト。必要なときに呼び出されます。

例:

- `test-engineer` — テスト戦略
- `designer` — UI/UX 設計
- `writer` — ドキュメントとユーザーガイダンス
- `researcher` — 外部ドキュメントとリファレンス調査
- `build-fixer` — ビルド/ツールチェーンの修正
- `git-master` — Git 操作と履歴管理

### Coordination レーン

他のエージェントが作った計画と結果をレビュー・調整します。

例:

- `critic` — 計画/設計の抜けをレビュー
- `team-executor` — チーム実行レーンの補助
- `quality-strategist` — 品質戦略の判断

### エージェント選択の勘どころ

シンプルに言うとこうなります。

| タスクの種類 | まず思い浮かべるエージェント |
|-----------|----------------------|
| コードベースを素早く読みたい | `explore` |
| 要件が曖昧 | `analyst` |
| まず計画を立てたい | `planner`、`architect`、`critic` |
| 実装そのものが必要 | `executor` |
| 完了かどうかを確認したい | `verifier` |
| セキュリティ/品質/性能が気になる | review レーン |

全リストは [エージェントカタログ](/ja/docs/agents) にあります。

## スキルとは?

スキルは、エージェントの挙動を組み合わせるワークフローシステムです。エージェントを置き換えるのではなく、どの順序でどの役割を呼ぶかを整理して提供します。

<Mermaid chart={`
graph TD
    G["入口スキル\n$deep-interview / $ralplan / $ralph / $team"] --> H["役割選択\nどのエージェントをいつ呼ぶか決定"]
    H --> I["実行\nエージェント呼び出し、ステート更新、検証ループ"]
    I --> J["完了\n検証できたら終了"]
`} />

### スキルはどう呼ぶのか?

OMX では通常、`$` プレフィックスで呼び出します。

```bash
$deep-interview "認証要件を整理して"
$ralplan "もっとも安全な実装計画を立てて"
$ralph "承認済みの計画を完了まで進めて"
```

マジックキーワードが検出されると、対応するスキルが自動で起動することもあります。

### 主要なワークフロースキル

| スキル | 役割 |
|------|------|
| `$deep-interview` | 要件が曖昧なときに質問で明確化 |
| `$ralplan` | planner/architect/critic の合意ベース計画 |
| `$ralph` | 完了が検証されるまで走り続ける |
| `$team` | 並列実行が必要な作業を調整 |
| `$autopilot` | アイデアから実装/QA/検証まで一括 |
| `$ultraqa` | テスト、検証、修正の反復 |

### スキルはいつ使うのか?

- **何を作るか曖昧** → `$deep-interview`
- **実装前に計画を立てたい** → `$ralplan`
- **一人のオーナーが最後まで押し切る** → `$ralph`
- **並列作業が必要** → `$team`
- **全工程を一度に任せたい** → `$autopilot`

全リストは [スキルカタログ](/ja/docs/skills) にあります。

## フックとは?

フック (Hook) は Codex CLI のライフサイクルイベントに反応するランタイムコードです。ユーザーがプロンプトを入力したとき、ツールが実行されたとき、セッションが開始/終了するときに自動で介入します。

<Mermaid chart={`
graph TD
    A["SessionStart"] --> B["ステート復元 / コンテキスト注入"]
    C["UserPromptSubmit"] --> D["マジックキーワード検出 / スキル起動"]
    E["PreToolUse / PostToolUse"] --> F["ガード / 後処理 / ヒント表出"]
    G["Stop"] --> H["未完了の実行を継続するかの判断"]
`} />

OMX はフックを通じて、キーワード検出、スキル起動、ステート復元、停止ブロックといった挙動を実装します。

### 主なイベントと役割

| イベント | OMX の処理 |
|--------|---------------|
| `SessionStart` | ステート復元、開発者コンテキスト注入、`.omx/` 初期点検 |
| `UserPromptSubmit` | マジックキーワード検出、対応スキルの起動 |
| `PreToolUse` | 危険コマンドの警告、実行前ガード |
| `PostToolUse` | 失敗ヒントの表出、後処理 |
| `Stop` | 長時間モードの停止/継続の制御 |

つまりフックはユーザーが直接呼ぶ機能ではなく、**OMX が裏で自然に動作するためのイベント層**と考えれば分かりやすいです。

詳細は [Hooks](/ja/docs/hooks) セクションを参照してください。

## ステート管理の概要

OMX は実行中のステートとセッション間の記憶を `.omx/` 以下に保存します。

<Mermaid chart={`
graph TD
    A["現在のセッション"] --> B[".omx/state\nアクティブなモード状態"]
    A --> C[".omx/notepad.md\n作業メモ"]
    A --> D[".omx/project-memory.json\n永続知識"]
    A --> E[".omx/wiki\n蓄積されるドキュメント知識"]
`} />

| パス | 役割 |
|------|------|
| `.omx/state/` | アクティブモードの状態 |
| `.omx/plans/` | `$ralplan` などの計画結果 |
| `.omx/logs/` | セッションと実行ログ |
| `.omx/notepad.md` | 自由形式メモ |
| `.omx/project-memory.json` | 長期プロジェクト指示/ノート |
| `.omx/wiki/` | 蓄積される wiki 知識 |

### なぜ重要なのか?

ステート管理があるおかげで OMX は:

- context compaction のあとも進捗を失わず
- 長時間ワークフローを再開でき
- セッションをまたいでプロジェクト知識を引き継げる

ようになります。

つまり、フックとスキルが「挙動」を担うのに対して、ステート管理はその挙動が途切れないようにする基盤です。

詳細は [`.omx/` ディレクトリ リファレンス](/ja/docs/reference/omx-directory) と [State ツール](/ja/docs/tools/state) を参照してください。
