## 概要

`product-manager` エージェントは何を作るかを決める段階を担当します。実装前に要件を明確にし、優先順位を設定し、スコープを確定します。開発エージェントが誤った方向に進まないよう、製品観点のコンテキストを提供します。

## いつ呼ばれるか

- 要件が曖昧でスコープを先に定める必要があるとき
- 機能の優先順位についてトレードオフを議論しなければならないとき
- `$deep-interview` や `$ralplan` の前に製品コンテキストを整理しなければならないとき
- 複数のステークホルダーの要求を一つの実行可能なスコープにまとめなければならないとき

## 使用例

```text
"この機能リストから MVP スコープを決めてほしい"
"認証システム変更の製品要件を整理してほしい"
"現在のロードマップで何を先にすべきか判断してほしい"
```

## 作業範囲

| 領域 | 内容 |
|---|---|
| 要件整理 | ビジネス目標を実行可能な基準へ変換 |
| スコープ確定 | 含めるものと除外するものの境界設定 |
| 優先順位判断 | インパクトと複雑度を基準に順序決定 |
| 受け入れ基準作成 | `analyst` が使える検証可能な条件を明示 |

## 作業プロセス

1. 現状と目標を把握します。
2. 要件の曖昧な部分を質問で絞り込みます。
3. 実行可能なスコープと受け入れ基準を整理します。
4. 優先順位の根拠を明示し、`analyst` や `planner` に引き渡します。

## 入力

- ユーザーが提示した機能アイデアやビジネス目標
- 既存のロードマップ、仕様書、イシューリスト
- ステークホルダーの要求や制約条件

## 出力

- スコープが確定した要件ドキュメント
- 受け入れ基準を含む機能定義
- 優先順位の根拠が付いたロードマップ

## 範囲の限界

- 実際の実装は `executor` が担当します。
- 技術アーキテクチャの決定は `architect` に任せます。
- 要件分析の詳細な受け入れ基準の導出は `analyst` と共に進めます。

## 関連エージェント

- [analyst](/ja/docs/agents/build-analysis/analyst) — product-manager が整理した要件を受け入れ基準として具体化します。
- [planner](/ja/docs/agents/build-analysis/planner) — 確定したスコープを基に実行順序を計画します。
- [architect](/ja/docs/agents/build-analysis/architect) — 製品要件を技術設計に移します。
