## 概要

`writer` エージェントはコードではなく文章を書きます。ユーザーガイド、API ドキュメント、マイグレーションノート、リリース変更点のまとめが主な作業です。実装が終わった後、それをどう説明するかがこのエージェントの領域です。

## いつ呼ばれるか

- 機能実装後にユーザードキュメントや API リファレンスを作成しなければならないとき
- breaking change を含むリリースにマイグレーションガイドが必要なとき
- 変更点をユーザーが理解できる言葉で整理しなければならないとき
- `$release` スキルでリリースノートの作成が必要なとき

## 使用例

```text
"この API 変更点をマイグレーションガイドとして書いてほしい"
"新しい認証機能のユーザーガイドを作成してほしい"
"v2 から v3 へのアップグレードのマイグレーションノートを作ってほしい"
```

## 作業範囲

| 項目 | 内容 |
|---|---|
| ユーザーガイド | 機能説明、使用例、設定方法 |
| API ドキュメント | エンドポイント、パラメータ、レスポンス例 |
| マイグレーションノート | breaking change リスト、アップグレード手順 |
| リリースノート | 変更点まとめ、主要機能の案内 |

## 作業プロセス

1. ドキュメント化する機能や変更点を把握します。
2. 読者が誰か（開発者、エンドユーザーなど）を確認します。
3. 使用例と共に明確で実用的な内容を作成します。
4. 技術用語は正確に、説明は具体的に書きます。

## 入力

- ドキュメント化する機能や変更点の説明
- `executor` が実装したコードや API スペック
- 対象読者とドキュメントの目的

## 出力

- ユーザーガイドや API リファレンスドキュメント
- マイグレーションガイドやリリースノート
- コード例を含む実用的な説明

## 範囲の限界

- コード実装は `executor` が担当します。
- 外部ドキュメントの調査は `researcher` に任せます。
- セキュリティやパフォーマンスに関する判断は該当するレビューエージェントを使います。

## 関連エージェント

- [executor](/ja/docs/agents/build-analysis/executor) — writer がドキュメント化する実装を担当します。
- [researcher](/ja/docs/agents/domain/researcher) — 外部リファレンスと公式ドキュメントを調査して writer をサポートします。
- [product-manager](/ja/docs/agents/domain/product-manager) — ドキュメントの範囲と方向を決定します。
