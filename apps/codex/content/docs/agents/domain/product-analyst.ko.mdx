`product-analyst`는 모든 제품 결정이 측정 가능하도록 합니다. 지표를 정밀하게 정의하고(분자, 분모, 시간 윈도우, 세그먼트), 이벤트 스키마를 설계하고, 표본 크기 계산이 포함된 A/B 실험 계획을 수립하고, 모든 KPI를 실제 사용자 결과에 연결합니다. 시스템 활동을 측정하는 허상 지표가 아닌 실제 의미 있는 지표를 목표로 합니다.

## 담당

- 제품 지표를 정확하게 정의 — 카운트 기준, 모집단, 측정 윈도우, 세그먼트 분류
- 계측을 위한 이벤트 스키마 설계 — 이벤트 이름, 트리거 조건, 필수 속성, 예시 페이로드
- 실험 측정 계획 수립 — 주요 지표, 가드레일 지표, 최소 감지 효과, 필요 표본 크기
- 기존 KPI가 사용자 결과에 연결되어 있는지 또는 단순히 시스템 활동만 측정하는지 감사

## 호출되는 시점

- 기능 계획 중 코드 한 줄 쓰기 전에 "성공"의 정의를 확립할 때
- 엄격한 측정 계획이 필요한 A/B 테스트나 실험을 준비할 때
- 기능 출시 후 사후 리뷰 — 결과가 예상대로 움직였는가 확인
- 팀이 지표가 올바른 방향으로 움직이고 있는지 의견 불일치가 있을 때

## 입력

지표가 지원할 제품 결정, 성공을 나타내는 사용자 행동, 기존 추적 인프라나 이벤트 로그를 제공합니다. 실험 설계라면 예상 효과 크기와 변경 사항 배포를 정당화할 비즈니스 최소값도 포함합니다.

## 출력

지표 정의(필수 구성 요소 모두 포함), 엔지니어링 인계용으로 형식화된 이벤트 스키마 제안, 실험 측정 체크리스트, 각 선행 지표가 후행 비즈니스 결과와 어떻게 연결되는지 보여주는 지표-결과 매핑.

## 범위 제한

- 데이터 파이프라인 구축이나 이벤트 추적 코드 구현은 하지 않습니다. 계측 코드는 `executor`에 넘깁니다.
- 심층 통계 분석이나 인과 분석은 하지 않습니다. 고급 통계가 필요하면 `researcher`에 넘깁니다.
- 어떤 기능을 만들지 결정하지 않습니다. `product-manager`가 우선순위를 정할 수 있도록 측정 근거를 제공하는 역할입니다.

## 관련 에이전트

| 에이전트 | 관계 |
|---|---|
| [`analyst`](/ko/docs/agents/build-analysis/analyst) | 지표가 구현하는 요구사항과 수락 기준 명확화 |
| [`product-manager`](/ko/docs/agents/domain/product-manager) | 지표 프레임워크를 사용해 범위와 우선순위 결정 |
| [`researcher`](/ko/docs/agents/domain/researcher) | 실험 결과의 인과 추론이 필요할 때 심층 통계 분석 제공 |
