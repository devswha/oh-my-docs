## 概要

`qa-tester` エージェントは、自動化テストが捕捉できないランタイム動作を直接実行して検証します。CLI コマンドを実際に実行したり、サービスエンドポイントを呼び出したり、エッジケースを手動で試すのがこのエージェントの役割です。`verifier` が証拠を収集する役割なら、`qa-tester` はその証拠を生み出す側に近いです。

## いつ呼ばれるか

- `executor` の実装後、実際の CLI/サービスが期待通りに動作するか手動で確認する必要があるとき
- 自動化テストに含まれていないユーザーシナリオを検証しなければならないとき
- `$ultraqa` サイクルで fix 後に動作が実際に復旧したか確認する必要があるとき
- フレーキーなテストを再現したり、エッジケースを直接試してみる必要があるとき

## 使用例

```text
"CLI コマンドを実際に実行して output が正しいか確認してほしい"
"認証フローのエッジケースを手動で検証してほしい"
"この API エンドポイントが実際に期待通りにレスポンスするか確認してほしい"
```

## 検証範囲

| 項目 | 内容 |
|---|---|
| CLI 実行 | 実際のコマンド実行と stdout/stderr 確認 |
| サービス呼び出し | HTTP エンドポイントの直接呼び出しとレスポンス検証 |
| ユーザーシナリオ | 実際の使用フローをステップごとに実行 |
| エッジケース | 境界値、不正な入力、失敗パスの検証 |

## 作業プロセス

1. 検証すべきシナリオと基準を確認します。
2. 実際の環境で CLI やサービスを実行します。
3. 出力と状態を期待値と比較します。
4. 異常があれば再現手順を記録し、`executor` や `debugger` に引き渡します。

## 入力

- 検証する機能やシナリオのリスト
- CLI コマンドや API スペック
- `executor` が完了した実装結果

## 出力

- 実行したコマンドと実際の出力の記録
- PASS / FAIL / PARTIAL 判定と根拠
- 再現可能なエッジケースの記録
- 失敗がある場合は再現手順を含むバグレポート

## 範囲の限界

- 自動化テストの作成は `test-engineer` が担当します。
- 完了証拠の公式判定は `verifier` が行います。
- 根本原因分析は `debugger` に任せます。

## 関連エージェント

- [verifier](/ja/docs/agents/build-analysis/verifier) — qa-tester が収集した証拠を基に公式判定を下します。
- [test-engineer](/ja/docs/agents/domain/test-engineer) — qa-tester が発見したケースを自動化テストとして定着させます。
- [debugger](/ja/docs/agents/build-analysis/debugger) — qa-tester が再現した失敗の根本原因を分析します。
- [executor](/ja/docs/agents/build-analysis/executor) — qa-tester の結果を基に実際の修正を行います。
