## 概述

`product-manager` agent 负责决定要构建什么的阶段。在实现之前明确需求、设定优先级、确定范围。为开发 agent 提供产品视角的上下文，防止它们朝错误方向前进。

## 何时被调用

- 需求模糊、需要先确定范围时
- 需要就功能优先级讨论权衡取舍时
- 在 `$deep-interview` 或 `$ralplan` 之前需要整理产品上下文时
- 需要将多个利益相关者的要求整合为一个可执行范围时

## 使用示例

```text
"从这个功能列表中确定 MVP 范围"
"整理认证系统变更的产品需求"
"判断当前路线图中应该先做什么"
```

## 工作范围

| 领域 | 内容 |
|---|---|
| 需求整理 | 将业务目标转换为可执行的标准 |
| 范围确定 | 设定包含与排除内容的边界 |
| 优先级判断 | 基于影响力和复杂度确定顺序 |
| 验收标准编写 | 明确 `analyst` 可用的可验证条件 |

## 工作流程

1. 了解当前状况和目标。
2. 通过提问缩小需求中的模糊部分。
3. 整理可执行的范围和验收标准。
4. 明示优先级依据，移交给 `analyst` 或 `planner`。

## 输入

- 用户提出的功能创意或业务目标
- 现有路线图、规格文档、issue 列表
- 利益相关者的要求或约束条件

## 输出

- 范围已确定的需求文档
- 包含验收标准的功能定义
- 附有优先级依据的路线图

## 范围限制

- 实际实现由 `executor` 负责。
- 技术架构决策交给 `architect`。
- 需求分析中详细验收标准的推导与 `analyst` 共同进行。

## 相关 agent

- [analyst](/zh/docs/agents/build-analysis/analyst) — 将 product-manager 整理的需求具体化为验收标准。
- [planner](/zh/docs/agents/build-analysis/planner) — 基于确定的范围规划执行顺序。
- [architect](/zh/docs/agents/build-analysis/architect) — 将产品需求转化为技术设计。
