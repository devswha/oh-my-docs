## 概要

`ux-researcher` エージェントはユーザー観点からインターフェースとインタラクションを分析します。使いやすさの問題を見つけ、ユーザーシナリオを整理し、改善方向の根拠を作る役割です。`designer` が視覚的な設計を担当するなら、`ux-researcher` はその設計判断の背景となるユーザーデータとシナリオを提供します。

## いつ呼ばれるか

- UI 変更前に現在のインターフェースの使いやすさの問題を把握しなければならないとき
- ユーザーシナリオとペルソナを整理して設計方向を決めなければならないとき
- 機能変更が実際の使用フローに与える影響を分析しなければならないとき
- `designer` に引き渡す前に UX 要件を整理しなければならないとき

## 使用例

```text
"現在のオンボーディングフローで使いやすさの問題を見つけてほしい"
"この機能の主要なユーザーシナリオを整理してほしい"
"変更された設定画面が既存の使用フローにどんな影響を与えるか分析してほしい"
```

## 分析範囲

| 項目 | 内容 |
|---|---|
| 使いやすさ分析 | インターフェースの摩擦ポイント、混乱要素の把握 |
| ユーザーシナリオ | 主要な使用フローとエッジケースの整理 |
| ペルソナ | 代表的なユーザータイプと目標の定義 |
| 影響分析 | 変更が既存の使用パターンに与える影響 |

## 作業プロセス

1. 分析対象の画面やフローを把握します。
2. ユーザー観点から主要なシナリオと問題点を整理します。
3. 改善方向の根拠を作り、`designer` や `product-manager` に引き渡します。
4. 変更後に使用フローへの影響を再確認します。

## 入力

- 分析対象の UI や機能の説明
- 既存のユーザーフィードバックや指標（ある場合）
- 製品目標と対象ユーザー情報

## 出力

- 使いやすさの問題リストと優先順位
- 主要なユーザーシナリオおよびペルソナドキュメント
- 改善方向の根拠まとめ
- `designer` や `product-manager` が使える UX 要件

## 範囲の限界

- 視覚設計とコンポーネントレイアウトは `designer` が担当します。
- 実際のユーザーデータ収集は ux-researcher の範囲外です。
- 実装は `executor` が行います。

## 関連エージェント

- [designer](/ja/docs/agents/domain/designer) — ux-researcher が整理した UX 要件を視覚設計に移します。
- [product-manager](/ja/docs/agents/domain/product-manager) — UX 調査結果を製品意思決定に反映します。
- [analyst](/ja/docs/agents/build-analysis/analyst) — UX 要件を具体的な受け入れ基準へ変換します。
