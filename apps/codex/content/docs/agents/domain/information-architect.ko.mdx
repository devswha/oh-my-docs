`information-architect`는 구조와 찾기 쉬움을 담당합니다. 현재 콘텐츠를 분석하고, 분류 체계가 사용자 멘탈 모델을 어디서 깨는지 파악하고, 마이그레이션 경로와 함께 중복 없는 카테고리 구조를 제안하고, 모든 제안된 변경이 실제 사용자 작업 기준으로 검증됐는지 확인합니다. 미적 선호가 아니라 사용자가 실제로 찾을 수 있는지를 기준으로 합니다.

## 담당

- 기존 문서 구조의 네이밍 불일치, 고아 콘텐츠, 탐색 실패 감사
- 얕고(최대 3단계), 넓으며, 구현 모듈이 아닌 사용자 작업 중심으로 내비게이션 계층 설계
- CLI 도움말, 문서, 에러 메시지, UI 레이블 전반에서 일관된 네이밍 규칙 제안
- 변경 권장 전에 작업 기반 탐색 가능성 매핑으로 분류 체계 제안 검증

## 호출되는 시점

- 콘텐츠가 원래 분류 체계를 넘어서 성장했을 때 문서 재구성이나 사이트 구조 개편 중
- 새 주요 기능을 기존 내비게이션에 배치해야 하는데 탐색 가능성을 유지해야 할 때
- `ux-researcher`가 휴리스틱 감사에서 탐색 실패를 식별한 후
- 제품 영역 전반(CLI, 문서, UI)에서 동일한 개념에 다른 이름을 쓰는 불일치가 누적됐을 때

## 입력

현재 문서 또는 내비게이션 트리, 구조가 지원해야 하는 사용자 작업, 탐색 불만이나 지원 티켓, 재구성 범위를 제공합니다. 네이밍 규칙이 초점이라면 해당 용어가 나타나는 모든 영역(CLI, 문서, UI, 에러 메시지)을 포함합니다.

## 출력

현재 구조 대 제안 구조를 보여주는 IA 맵, 탐색 가능성 점수(정확히 찾음/근접/미발견)가 포함된 작업-위치 매핑, 기존 사용자를 끊지 않는 현재에서 제안 구조로의 마이그레이션 경로, 반복되는 불일치에 대한 네이밍 규칙 가이드.

## 범위 제한

- 실제 문서 콘텐츠를 작성하거나 업데이트하지 않습니다. `writer`에 넘깁니다.
- 시각적 스타일링이나 내비게이션 UI 컴포넌트를 구현하지 않습니다. `designer`에 넘깁니다.
- 구조적 변경의 비즈니스 우선순위를 결정하지 않습니다. `product-manager`에 넘깁니다.

## 관련 에이전트

| 에이전트 | 관계 |
|---|---|
| [`writer`](/ko/docs/agents/domain/writer) | 구조가 확정된 후 콘텐츠 작성 |
| [`designer`](/ko/docs/agents/domain/designer) | 시각적 내비게이션 컴포넌트와 레이아웃 구현 |
| [`ux-researcher`](/ko/docs/agents/domain/ux-researcher) | 구조적 결정을 이끄는 사용자 증거와 탐색 가능성 조사 제공 |
