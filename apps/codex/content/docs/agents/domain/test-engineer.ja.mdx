## 概要

`test-engineer` エージェントはテストをどのように書くかを設計する役割です。単体テストから統合テスト、e2e まで戦略を立て、フレーキーなテストの原因を見つけて安定化させます。テストを直接書くことも含まれますが、主な役割は何をどのようにテストするかを判断する側です。

## いつ呼ばれるか

- 新機能に合ったテスト戦略を最初から設計しなければならないとき
- フレーキーなテストが繰り返し失敗し、原因を把握しなければならないとき
- 現在のテストカバレッジの穴を見つけて補強しなければならないとき
- `$tdd` モードで実装前にテストを先に書かなければならないとき
- `verifier` が依存するテスト基盤を強化しなければならないとき

## 使用例

```text
"この API エンドポイントのテスト戦略を立ててほしい"
"フレーキーに失敗するこのテストの原因を調べてほしい"
"認証モジュールのテストカバレッジの穴を把握して埋めてほしい"
```

## 作業範囲

| 項目 | 内容 |
|---|---|
| テスト戦略 | 単体/統合/e2e の境界設定、何をどのレベルでテストするか決定 |
| カバレッジ分析 | 既存テストの穴の把握と補強計画 |
| フレーキー解消 | 非決定的な失敗の原因分析と安定化 |
| テスト作成 | 戦略に基づく実際のテストコード作成 |

## 作業プロセス

1. 対象機能やモジュールの動作境界を把握します。
2. どのレベルでどのケースを検証するか戦略を決めます。
3. フレーキーテストなら失敗パターンを分析して根本原因を絞り込みます。
4. テストを作成または修正し、一貫して通過するか確認します。

## 入力

- テスト対象の機能やモジュールの説明
- 既存のテストコードと失敗ログ
- `planner` や `executor` が作った計画や実装結果
- カバレッジレポート

## 出力

- レベル別テスト戦略ドキュメント
- 実際のテストコード
- フレーキーテストの原因分析と修正結果
- カバレッジ改善計画

## 範囲の限界

- ランタイム動作の手動検証は `qa-tester` が担当します。
- 完了証拠の判定は `verifier` に任せます。
- ビルド失敗のようなテストと無関係なビルド問題は `build-fixer` が処理します。

## 関連エージェント

- [verifier](/ja/docs/agents/build-analysis/verifier) — test-engineer が設計したテスト基盤で完了可否を判定します。
- [qa-tester](/ja/docs/agents/domain/qa-tester) — test-engineer が扱わないインタラクティブなランタイム検証を担当します。
- [executor](/ja/docs/agents/build-analysis/executor) — test-engineer が作成したテストと共に実装を進めます。
- [debugger](/ja/docs/agents/build-analysis/debugger) — フレーキーテストの複雑な根本原因分析を支援します。
