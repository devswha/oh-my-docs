## 概要

`vision` エージェントは画像とスクリーンショットを読み取り、視覚的な結果を分析します。レンダリング結果が期待と異なるか、レイアウトが崩れていないか、デザインスペックと実際の実装が一致しているかを確認する役割です。テキストベースのコードレビューが捕捉できない視覚的な回帰を見つけるために使います。

## いつ呼ばれるか

- UI 変更後に実際のレンダリング結果をスクリーンショットで検証しなければならないとき
- デザインモックアップと実装結果を比較しなければならないとき
- レイアウト崩れ、色の不一致、テキスト切れのような視覚的バグを見つけなければならないとき
- `$visual-verdict` スキルで視覚 QA 判定が必要なとき

## 使用例

```text
"このスクリーンショットでレイアウト問題を見つけてほしい"
"デザインモックアップと実際の実装結果を比較してほしい"
"ダークモード切り替え後のレンダリング結果が正しいか確認してほしい"
```

## 分析範囲

| 項目 | 内容 |
|---|---|
| レイアウト検証 | コンポーネントの位置、間隔、整列の確認 |
| 視覚的回帰 | 変更前後のレンダリング差異の検出 |
| デザイン一致 | モックアップ vs 実際の実装の比較 |
| UI バグ | テキスト切れ、色の不一致、重なりなど |

## 作業プロセス

1. 分析する画像やスクリーンショットを受け取ります。
2. 期待値（デザインモックアップ、以前のスクリーンショット、スペック）と比較します。
3. 差異と問題を具体的に記録します。
4. 修正が必要な場合は `designer` や `executor` に引き渡します。

## 入力

- 分析するスクリーンショットや画像ファイル
- 比較基準となるデザインモックアップや以前のスクリーンショット
- 確認すべき具体的な項目（ある場合）

## 出力

- 発見した視覚的問題のリストと位置
- 期待値と実際の結果の比較まとめ
- PASS / FAIL / PARTIAL 判定
- 修正が必要な項目の具体的な説明

## 範囲の限界

- コード変更は `executor` が担当します。
- UX 戦略の判断は `ux-researcher` と `designer` に任せます。
- パフォーマンス測定は `performance-reviewer` が処理します。

## 関連エージェント

- [designer](/ja/docs/agents/domain/designer) — vision が発見した視覚的問題を設計観点で解決します。
- [qa-tester](/ja/docs/agents/domain/qa-tester) — vision と共に UI のランタイム動作を検証します。
- [executor](/ja/docs/agents/build-analysis/executor) — vision が見つけた視覚的バグを実際に修正します。
