`explore` 에이전트는 다른 에이전트가 코드를 건드리기 전에 "X가 어디에 있나?"를 먼저 답합니다. 저장소 전체를 넓게 훑고 좁혀 가는 병렬 검색으로 파일 경로, 심볼 위치, 호출 관계를 구조화해서 반환합니다. 완전히 읽기 전용이라 워크플로우 어느 시점에서든 부작용 없이 호출할 수 있습니다.

## 역할

- 질문과 관련된 파일, 디렉터리, 모듈 경계를 파악합니다.
- 함수, 클래스, 타입 같은 심볼 위치와 파일 간 관계를 찾습니다.
- 진입점, dependency chain, import graph를 식별합니다.
- Glob, Grep, AST search, LSP 결과를 교차 검증해 false positive를 줄입니다.

## 언제 호출되는가

| 상황 | 호출 경로 |
|---|---|
| `$autopilot`, `$deep-interview`, `$deepsearch` 시작 시 | 자동 |
| "X가 어디 있는지 찾아줘", "Y를 건드리는 파일이 뭐야" | 직접 요청 |
| `planner`나 `executor`가 실제 파일 경로를 받아야 할 때 | 상위 에이전트 요청 |
| 디버깅 중인데 증상 위치를 아직 모를 때 | `debugger`에서 위임 |

## 입력

- 저장소 루트 경로
- 검색 질의 또는 범위 힌트 (심볼 이름, 기능 영역, 파일 패턴, 자유 형식 설명)
- 선택적으로 에러 메시지, 이미 알고 있는 파일 경로

## 출력

- 절대 파일 경로, 매칭된 심볼, 관계 요약이 담긴 in-session 보고서
- deep-search 실행에서는 `.omx/research/<topic>.md`에 결과를 남겨 다른 에이전트가 참고할 수 있습니다.
- 이어서 어떤 에이전트가 작업해야 하는지 알려주는 `next_steps` 섹션

## 제한 사항

- 파일을 생성, 수정, 삭제하지 않습니다.
- 테스트 실행, 빌드, 코드 품질 평가는 담당하지 않습니다.
- 아키텍처 권고는 하지 않습니다. 그 역할은 `architect`입니다.

## 관련 에이전트

- [analyst](/ko/docs/agents/build-analysis/analyst) — `explore` 결과를 바탕으로 요구사항을 정리합니다.
- [planner](/ko/docs/agents/build-analysis/planner) — `explore`가 찾은 파일·모듈 지도를 바탕으로 작업 순서를 세웁니다.
- [debugger](/ko/docs/agents/build-analysis/debugger) — 증상 위치를 찾을 때 `explore`를 활용합니다.
- [executor](/ko/docs/agents/build-analysis/executor) — 수정 전에 `explore`와 `planner`가 정리한 파일 목록을 입력으로 받습니다.
