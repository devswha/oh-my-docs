`debugger` エージェントは症状を隠す代わりに、失敗を root cause まで追跡します。調査前に再現可能性をまず確保し、エラーメッセージと stack trace をすべて読み、一度に一つの仮説だけを立てて検証します。同じアプローチが3回失敗したら、変形を繰り返さず上位レーンにエスカレーションします。

## いつ呼ばれるか

| 状況 | 呼び出し経路 |
|---|---|
| テストスイートが失敗しているが root cause がすぐに見えないとき | 直接要求 |
| 「debug X」の要求だがバグの場所がまだわからないとき | 直接要求 |
| `ralph` や `$autopilot` loop が同じ失敗で繰り返し止まるとき | 自動 |
| regression が入ってきて offending change を絞り込む必要があるとき | 直接要求 |

## 使用例

```text
"このエラーがなぜ起きるか root cause を見つけて"
"このテストがなぜ突然壊れたか追跡して"
"ビルドエラーを症状ではなく原因の観点で絞り込んで"
```

## ランタイムバグ調査プロトコル

1. 失敗を安定して再現します。
2. stack trace、エラーメッセージ、最近の変更履歴を集めます。
3. 一度に一つの仮説だけを立てます。
4. 最小の修正でその仮説を検証します。
5. 合わなければ次の仮説に移ります。

複数の修正案を一度に入れません。一仮説、一変更の原則を維持します。

## ビルド / コンパイルエラー解決プロトコル

- エラーの種類をまず分類します。
- import/export の問題、型推論の問題、設定の問題を区別します。
- 最小の修正で一つずつ減らします。
- 修正後は diagnostics / build を再実行して実際に解決されたか確認します。

エラーメッセージを読むだけで止まらず、**分類 → 仮説 → 最小修正 → 再検証**のループを繰り返します。

## 入力

- エラーメッセージ、stack trace、失敗したテストの出力
- Grep、Read、git log/blame でアクセスしたコードベース全体
- オプションで再現手順、環境情報、以前の仮説メモ

## 出力

- root cause、`file:line` 根拠、最小再現手順、最小修正の勧告、検証方法を含む bug report
- 同じパターンが存在し得る他のコード箇所のリスト

## 範囲の限界

- 3回の仮説失敗後は circuit breaker を適用し、evidence とともにエスカレーションします。
- 複数の修正案を一度にまとめません。
- `probably`、`seems like` などの推測表現で診断を終わらせません。`file:line` の根拠が必要です。

## 関連エージェント

- [explore](/ja/docs/agents/build-analysis/explore) — バグの場所がまだわからないとき、軽量な read-only lookup を実行します。
- [verifier](/ja/docs/agents/build-analysis/verifier) — 修正が実際に問題を解決したか全体の証拠で確認します。
- [test-engineer](/ja/docs/agents/domain/test-engineer) — root cause が確認された後に regression test を設計します。
- [explore-harness](/ja/docs/agents/build-analysis/explore-harness) — 複数仮説の因果調査に使用されます。
