title: explore
description: "ファイル、シンボル、エントリポイント、依存関係を読み取り専用で把握するコードベース探索エージェント。"
---

`explore` エージェントは、他のエージェントがコードに触れる前に「X はどこにあるか？」をまず答えます。リポジトリ全体を広く走査してから絞り込む並列検索で、ファイルパス、シンボル位置、呼び出し関係を構造化して返します。完全に読み取り専用なので、ワークフローのどの時点でも副作用なしに呼び出せます。

## 役割

- 質問に関連するファイル、ディレクトリ、モジュール境界を把握します。
- 関数、クラス、型といったシンボルの位置とファイル間の関係を見つけます。
- エントリポイント、dependency chain、import graph を識別します。
- Glob、Grep、AST search、LSP の結果を交差検証して false positive を減らします。

## いつ呼ばれるか

| 状況 | 呼び出し経路 |
|---|---|
| `$autopilot`、`$deep-interview`、`$deepsearch` 開始時 | 自動 |
| 「X がどこにあるか探して」「Y を触っているファイルは？」 | 直接要求 |
| `planner` や `executor` が実際のファイルパスを受け取る必要があるとき | 上位エージェント要求 |
| デバッグ中だが症状の場所がまだわからないとき | `debugger` から委譲 |

## 入力

- リポジトリのルートパス
- 検索クエリまたは範囲ヒント（シンボル名、機能領域、ファイルパターン、自由記述）
- オプションでエラーメッセージ、すでに判明しているファイルパス

## 出力

- 絶対ファイルパス、マッチしたシンボル、関係サマリーを含む in-session レポート
- deep-search 実行では `.omx/research/<topic>.md` に結果を残し、他のエージェントが参照できます。
- 次にどのエージェントが作業すべきかを示す `next_steps` セクション

## 範囲の限界

- ファイルの作成、修正、削除は行いません。
- テスト実行、ビルド、コード品質の評価は担当しません。
- アーキテクチャの勧告はしません。その役割は `architect` です。

## 関連エージェント

- [analyst](/ja/docs/agents/build-analysis/analyst) — `explore` の結果をもとに要件を整理します。
- [planner](/ja/docs/agents/build-analysis/planner) — `explore` が見つけたファイル・モジュールのマップをもとに作業順序を組み立てます。
- [debugger](/ja/docs/agents/build-analysis/debugger) — 症状の場所を探すときに `explore` を活用します。
- [executor](/ja/docs/agents/build-analysis/executor) — 修正前に `explore` と `planner` が整理したファイルリストを入力として受け取ります。
