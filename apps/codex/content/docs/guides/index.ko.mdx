## 어떤 흐름을 고를지

OMX는 워크플로우가 여러 개라 처음 쓰면 고르는 것부터 막힙니다. 이 페이지는 스킬 이름을 나열하는 문서가 아니라, **지금 어떤 상황이면 어떤 흐름을 꺼내야 하는지** 결정을 돕는 쪽입니다.

## 한 눈에 보는 결정표

| 상황 | 첫 선택 | 이유 |
|---|---|---|
| 요구사항 자체가 흐릿함 | `$deep-interview` | 구현보다 범위 확정이 먼저 |
| 방향은 알지만 계획은 없음 | `$ralplan` | 트레이드오프를 먼저 못 박아야 |
| 계획 있고 한 명이 끝까지 밀어야 함 | `$ralph` | 지속 소유자 + 검증 루프 |
| 큰 계획을 여러 milestone으로 오래 추적해야 함 | `$ultragoal` | `.omx/ultragoal` ledger + Codex goal handoff |
| 계획을 여러 lane으로 쪼갤 수 있음 | `$team` | 병렬화 이득이 실제로 있을 때만 |
| 프로토타입 하나 빠르게 보고 싶음 | `$autopilot` | 제어를 내려놓는 대신 속도 |

거의 모든 선택이 이 표 하나로 끝납니다. 아래는 각 항목이 왜 그런 자리에 있는지 풀어 쓴 내용입니다.

---

## 기본 경로

OMX를 처음 쓴다면 표준 경로는 셋입니다.

```text
$deep-interview "인증 변경 스코프를 정리해줘"
$ralplan "가장 안전한 구현 경로를 승인해줘"
$ralph "승인된 계획을 끝까지 완료해줘"
```

이 셋이 왜 분리되어 있는지부터 이해하면 OMX가 어떻게 움직이는지 보입니다.

- **`$deep-interview`** — 요구가 모호할 때 질문으로 먼저 정리. 코드보다 "무엇을 만들지" 확정이 먼저.
- **`$ralplan`** — planner / architect / critic이 합의할 때까지 반복. 구현 직전에 트레이드오프를 고정.
- **`$ralph`** — 승인된 계획을 잡고 완료 증거가 나올 때까지 반복 실행.

`$autopilot` 하나로 퉁칠 수도 있지만, **한 번은 이 세 단계를 손으로 타봐야 OMX가 뭘 하는지 감이 옵니다.** 그 다음에 언제 단축해도 되는지가 보입니다.

---

## `$autopilot`은 언제 쓰나

아이디어 하나를 구현 → QA → 검증까지 한 번에 맡길 때.

```bash
$autopilot "build me a todo app in next.js"
```

| 항목 | 상태 |
|---|---|
| 제어 수준 | 낮음 — OMX가 흐름 대부분을 결정 |
| 속도 | 빠름 — 파이프라인 전체를 한 번에 관찰 |
| 잘 맞는 경우 | 프로토타입, 작은 기능, end-to-end 확인 |
| 피할 경우 | 요구가 흐릴 때, 계획 합의가 먼저인 작업 |

짧게 말하면 `$autopilot`은 "빨리 전체를 본다", 계획 기반 흐름은 "제어하면서 간다".

## 계획 기반 흐름을 먼저 타야 할 때

범위가 넓거나, 잘못된 방향이 비싼 작업이면 실행을 늦추는 쪽이 오히려 빠릅니다.

```text
$deep-interview "무엇을 만들어야 하는지 정리해줘"
$ralplan "가장 안전한 계획을 세워줘"
$ralph "그 계획대로 끝까지 진행해줘"
```

이 경로가 특히 유용한 경우:

- 아키텍처 · 테스트 · 운영 영향을 같이 봐야 함
- 요구가 여러 이해관계자를 거쳐 아직 정리 안 됨
- 중간에 방향 합의가 필요한 큰 작업

---

## `$ultragoal`은 durable goal이 필요할 때

`$ultragoal`은 큰 brief나 승인된 plan을 durable Codex/OMX goal로 바꾸는 흐름입니다. 장기 계획은 `.omx/ultragoal/` 아래에 남기고, story 진행은 ledger로 기록하며, 활성 thread는 Codex goal mode에 명시적으로 handoff합니다.

![Ultragoal이 승인된 계획을 durable milestone, ledger, Codex goal handoff, 최종 review evidence로 이어주는 흐름](/images/docs/guides-ultragoal-flow.svg)

```bash
$ultragoal "이 launch plan을 durable goal로 나누고 끝까지 완료해줘"
```

| 항목 | 상태 |
|---|---|
| 제어 수준 | 높음 — plan과 evidence가 파일로 남음 |
| 지속성 | 높음 — compaction, session 변경, handoff 이후에도 이어가기 쉬움 |
| 잘 맞는 경우 | 여러 milestone, release train, 큰 리팩터링, 최종 검증이 중요한 작업 |
| 피할 경우 | 작은 one-shot 수정, 요구가 아직 모호한 작업, 병렬 lane이 더 자연스러운 작업 |

명령 세부사항과 completion gate는 [`$ultragoal`](/ko/docs/skills/workflow/ultragoal) 문서를 보세요.

대표 흐름은 이렇습니다.

```text
$deep-interview "제품 범위를 정리해줘"
$ralplan "milestone 기준으로 안전한 계획을 승인해줘"
$ultragoal "승인된 계획을 durable goal로 바꾸고 완료까지 진행해줘"
```

선택 기준은 간단합니다.

- **계획은 크지만 순차 milestone으로 추적해야 함** → `$ultragoal`
- **한 명이 한 맥락으로 끝까지 밀면 충분함** → `$ralph`
- **frontend / backend / test처럼 동시에 나눌 수 있음** → `$team`
- **요구가 아직 흐림** → `$deep-interview` 또는 `$ralplan` 먼저

---

## `$team`은 병렬화 가치가 있을 때만

`$team`은 "항상 더 빠른 모드"가 아닙니다. **실제로 나눌 수 있는 작업이 있을 때**만 가치가 나옵니다.

```bash
$team 3:executor "승인된 계획을 병렬로 실행해줘"
```

| 항목 | 상태 |
|---|---|
| 제어 수준 | 중간~높음 — 분할을 미리 설계해야 함 |
| 병렬성 | 높음 — 여러 lane 동시 진행 |
| 잘 맞는 경우 | frontend / backend / test처럼 자연스럽게 나뉘는 작업 |
| 피할 경우 | 범위가 흐림, 병렬화가 충돌만 늘림 |

피해야 하는 신호:

- 요구가 모호 → `$deep-interview` 먼저
- 한 맥락을 끝까지 붙잡고 가야 함 → `$ralph`
- lane끼리 충돌할 것 같음 → 순차가 더 빠름
