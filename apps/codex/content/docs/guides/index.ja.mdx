## どのフローを選ぶか

OMX はワークフローが複数あるため、初めて使うときはどれを選べばいいか迷います。このページはスキル名を列挙するドキュメントではなく、**今どんな状況ならどのフローを選ぶべきか**の判断を助けるものです。

## 一目でわかる決定表

| 状況 | 最初の選択 | 理由 |
|---|---|---|
| 要件自体が曖昧 | `$deep-interview` | 実装より先にスコープを確定すべき |
| 方向はわかるが計画がない | `$ralplan` | トレードオフを先に固める必要がある |
| 計画があり一人で最後まで押し進める | `$ralph` | 継続所有者 + 検証ループ |
| 計画を複数レーンに分割できる | `$team` | 並列化の恩恵が実際にある場合のみ |
| プロトタイプを素早く見たい | `$autopilot` | 制御を手放す代わりにスピード重視 |

ほぼすべての選択がこの表一つで完結します。以下は各項目がなぜそこにあるかを解説した内容です。

## 基本パス

OMX を初めて使うなら、標準パスは三つです。

```text
$deep-interview "認証変更のスコープを整理してくれ"
$ralplan "最も安全な実装パスを承認してくれ"
$ralph "承認された計画を最後まで完了してくれ"
```

この三つがなぜ分かれているかを理解すると、OMX がどのように動くか見えてきます。

- **`$deep-interview`** — 要件が曖昧なときに質問で先に整理。コードより「何を作るか」の確定が先。
- **`$ralplan`** — planner / architect / critic が合意するまで反復。実装直前にトレードオフを固定。
- **`$ralph`** — 承認された計画を持ち、完了証拠が出るまで反復実行。

`$autopilot` 一つでまとめることもできますが、**一度はこの三段階を手で体験してこそ OMX が何をしているか感覚がつかめます。** そのあと、いつ短縮してよいかが見えてきます。

---

## `$autopilot` はいつ使うか

アイデア一つを実装 → QA → 検証まで一括で任せるとき。

```bash
$autopilot "build me a todo app in next.js"
```

| 項目 | 状態 |
|---|---|
| 制御レベル | 低い — OMX がフローの大半を決定 |
| スピード | 速い — パイプライン全体を一度に観察 |
| 向いているケース | プロトタイプ、小さな機能、end-to-end 確認 |
| 避けるべきケース | 要件が曖昧なとき、計画合意が先の作業 |

簡単に言えば `$autopilot` は「素早く全体を見る」、計画ベースのフローは「制御しながら進む」。

---

## 計画ベースのフローを先に通すべきとき

スコープが広い、または間違った方向が高くつく作業なら、実行を遅らせる方がかえって速いです。

```text
$deep-interview "何を作るべきか整理してくれ"
$ralplan "最も安全な計画を立ててくれ"
$ralph "その計画通りに最後まで進めてくれ"
```

このパスが特に有効なケース:

- アーキテクチャ · テスト · 運用への影響を同時に見る必要がある
- 要件が複数のステークホルダーを経てまだ整理されていない
- 途中で方向合意が必要な大きな作業

---

## `$team` は並列化に価値があるときだけ

`$team` は「常に速いモード」ではありません。**実際に分割できる作業があるとき**だけ価値があります。

```bash
$team 3:executor "承認された計画を並列で実行してくれ"
```

| 項目 | 状態 |
|---|---|
| 制御レベル | 中〜高 — 分割を事前に設計する必要がある |
| 並列性 | 高い — 複数レーンを同時進行 |
| 向いているケース | frontend / backend / test のように自然に分かれる作業 |
| 避けるべきケース | スコープが曖昧、並列化が衝突を増やすだけ |

避けるべきシグナル:

- 要件が曖昧 → `$deep-interview` を先に
- 一つのコンテキストを最後まで持ち続ける必要がある → `$ralph`
- レーン間で衝突しそう → 順次処理の方が速い
