## Use cases を探す

作業の種類は分かっているが、どの OMX flow が担当すべきか迷うときにこのページを使います。

短い基準はこうです。タスクが曖昧なら先に明確化し、間違えたときのコストが高いなら先に計画し、承認済み plan を長く残す必要があるなら `$ultragoal` に渡し、実際に並列性があるときだけチームに分けます。

## 判断表

| 状況 | 最初に使うもの | 理由 |
|---|---|---|
| 要件が曖昧 | `$deep-interview` | 実装前に scope を明確にする必要がある |
| 方向は分かるが計画がない | `$ralplan` | tradeoff と test を先に固定するため |
| UI/UX、フロントエンド、design-system の方向が必要 | `$design` | 実装前に repo-local `DESIGN.md` で基準を固定するため |
| 結合度が高く一つの context が必要 | `$ralph` | 単一 owner が一つの検証ループを維持するため |
| 大きな計画が milestone にまたがる | `$ultragoal` | goal、ledger、handoff evidence を永続化するため |
| 独立 lane に分割できる | `$team` | 並列化の価値が実際にあるため |
| 素早く prototype を見たい | `$autopilot` | 一部の制御を手放して速度を得るため |
| 検証と修正が中心 | `$ultraqa` | test、diagnosis、repair を繰り返すため |

## 標準 guided flow

OMX が初めてなら、shortcut を使う前にこの 3 段階 route を一度通るのがおすすめです。

```text
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ultragoal "turn the approved plan into durable goals and complete them"
```

| 段階 | 役割 |
|------|------|
| `$deep-interview` | 要件が曖昧なときにタスクを整理 |
| `$ralplan` | 実装前に計画と tradeoff を合意 |
| `$ultragoal` | 承認済み計画を durable goal、ledger evidence、completion checkpoint に変換する |

後で `$autopilot` に full-pipeline execution を任せることもできますが、この route を先に通ると各段階が見えます。

## `$autopilot` ですばやくプロトタイプを作る

```bash
$autopilot "build me a todo app in next.js"
```

よく合う場合:

- 小さな機能または throwaway prototype
- 運用リスクが低い
- 全体の pipeline を素早く見たい

要件が不明確、architecture risk が大きい、または明示的な plan approval が重要なら別の flow を選びます。

## 実装前に計画する

```text
$deep-interview "clarify what we need to build"
$ralplan "produce the safest implementation plan"
$ultragoal "turn that plan into durable goals and complete them"
```

architecture、testing、data migration、security、operations への影響が重要なときの安全な経路です。承認済み plan が一度の実行ループだけに残らず、durable goal として保存されるためです。

## `$design` で UI 作業前に設計基準を作る

プロダクト、UI/UX、フロントエンド、design-system の決定を実装前に repo の durable source of truth として残す必要がある場合は、[`$design`](/ja/docs/skills/workflow/design) を使います。既存 UI の根拠を確認し、設計上重要な不足文脈だけを質問し、repo root の `DESIGN.md` を作成または更新します。

```bash
$design "create a design contract for the dashboard refresh"
```

よい後続フロー:

- frontend、backend、test をきれいに分割できる場合は、承認済み design contract を `$team` に渡す
- design plan が複数 milestone にまたがる場合は `$ultragoal` に渡す
- 承認済み screenshot、mockup、image、live URL を視覚的に合わせる必要がある場合だけ `$visual-ralph` を使う


## `$ultragoal` で durable milestone を追跡する

`$ultragoal` は広い brief や承認済み plan を durable Codex/OMX goal に変換します。長期計画を `.omx/ultragoal/` に書き、ledger で進捗を追跡し、アクティブな thread を Codex goal mode に明示的に handoff します。

![Ultragoal が brief を durable state に変換し、Codex goal へ handoff し、checkpoint evidence を ledger に戻して final gate へ進む流れ](/images/docs/guides-ultragoal-flow.svg)

```bash
$ultragoal "split this launch plan into durable goals and complete them"
```

context compaction や handoff に耐える必要がある multi-milestone launch、release train、大きな refactor に使います。

## `$team` で並列作業を分ける

```bash
$team 3:executor "execute the approved plan in parallel"
```

![$team が shared state を作成し、worker lane を起動し、task claim と ACK/heartbeat を集めて完了時に shutdown する流れ](/images/docs/guides-team-flow.svg)

よく合う場合:

- frontend、backend、test が独立して進められる
- file ownership を明確に分けられる
- 計画がすでに承認されている

要件が曖昧、一つの context を維持すべき、または並列 lane がほとんど merge conflict を生むだけなら team mode は避けます。

## `$ultraqa` で検証と修正を繰り返す

green build だけでは不十分で、hostile/edge condition での実際の動作まで確認する必要がある場合は、[`$ultraqa`](/ja/docs/skills/workflow/ultraqa) を使います。baseline check と adversarial dynamic e2e scenario を組み合わせ、失敗を診断し、修正を適用し、temporary harness cleanup と evidence report まで残します。

```bash
$ultraqa --tests
$ultraqa --custom "the report export rejects stale state"
```

向いているケース:

- prompt injection、cancel/resume、stale state、misleading success output が重要な CLI または workflow 変更
- failure evidence と cleanup proof が必要な release-readiness 確認
- unit test は通ったが dynamic behavior probe がまだ必要な機能
