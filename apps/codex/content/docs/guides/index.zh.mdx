## 探索 use cases

当你知道任务类型，但不确定应该由哪个 OMX flow 负责时，请使用本页。

最短规则是：任务模糊时先澄清，错误代价高时先规划，已批准计划需要持久化时交给 `$ultragoal`，只有真正存在并行性时才拆成团队。

## 决策表

| 情况 | 起点 | 原因 |
|---|---|---|
| 需求模糊 | `$deep-interview` | 实现前必须澄清 scope |
| 方向明确但没有计划 | `$ralplan` | 先固定 tradeoff 和 test |
| 需要 UI/UX、前端或 design-system 方向 | `$design` | 实现前用 repo-local `DESIGN.md` 固定基准 |
| 紧密耦合的工作需要一个 context | `$ralph` | 让单一 owner 维持一个验证循环 |
| 大计划跨多个 milestone | `$ultragoal` | 持久化 goal、ledger 和 handoff evidence |
| 可以拆成独立 lane | `$team` | 并行化确实有价值 |
| 想快速看到 prototype | `$autopilot` | 用较少控制换取速度 |
| 重点是验证和修复 | `$ultraqa` | 循环 test、diagnosis 和 repair |

## 默认 guided flow

如果你刚开始使用 OMX，建议先走一次三步 route，再使用 shortcut。

```text
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ultragoal "turn the approved plan into durable goals and complete them"
```

| 步骤 | 作用 |
|------|------|
| `$deep-interview` | 需求模糊时整理任务 |
| `$ralplan` | 实现前对计划和 tradeoff 达成一致 |
| `$ultragoal` | 将已批准计划转成 durable goal、ledger evidence 和 completion checkpoint |

之后你可以用 `$autopilot` 快速交给 full-pipeline execution，但先走这条 route 可以看清各阶段。

## 用 `$autopilot` 快速制作原型

```bash
$autopilot "build me a todo app in next.js"
```

适合：

- 小功能或 throwaway prototype
- 运维风险低
- 想快速观察完整 pipeline

如果需求不清楚、architecture risk 较大，或必须显式 plan approval，请选择其他 flow。

## 实现前先规划

```text
$deep-interview "clarify what we need to build"
$ralplan "produce the safest implementation plan"
$ultragoal "turn that plan into durable goals and complete them"
```

当 architecture、testing、data migration、security、operations 影响重要时，这是更安全的路径，因为已批准计划会作为 durable goal 保留下来，而不是只存在于一次执行循环中。

## 用 `$design` 在 UI 工作前确定设计基准

当产品、UI/UX、前端或 design-system 决策需要在实现前成为 repo 的 durable source of truth 时，使用 [`$design`](/zh/docs/skills/workflow/design)。它会检查现有 UI 证据，只询问设计关键的缺失上下文，并创建或刷新 repo root 的 `DESIGN.md`。

```bash
$design "create a design contract for the dashboard refresh"
```

合适的后续流程：

- frontend、backend、test 可以清晰拆分时，把已批准 design contract 交给 `$team`
- design plan 跨多个 milestone 时，交给 `$ultragoal`
- 只有在需要匹配已批准 screenshot、mockup、image 或 live URL 时，才使用 `$visual-ralph`


## 用 `$ultragoal` 跟踪 durable milestone

`$ultragoal` 会把宽泛 brief 或已批准 plan 转成 durable Codex/OMX goal。它把长期计划写入 `.omx/ultragoal/`，用 ledger 跟踪进度，并明确把当前 thread handoff 到 Codex goal mode。

![Ultragoal 将 brief 转成 durable state，handoff 到 Codex goal，把 checkpoint evidence 写回 ledger，并进入 final gate 的流程](/images/docs/guides-ultragoal-flow.svg)

```bash
$ultragoal "split this launch plan into durable goals and complete them"
```

适用于需要经受 context compaction 或 handoff 的 multi-milestone launch、release train 和大型 refactor。

## 用 `$team` 拆分并行工作

```bash
$team 3:executor "execute the approved plan in parallel"
```

![$team 创建 shared state，启动 worker lane，收集 task claim 与 ACK/heartbeat，并在完成后 shutdown 的流程](/images/docs/guides-team-flow.svg)

适合：

- frontend、backend、test 可以独立推进
- file ownership 可以明确分配
- 计划已经批准

如果需求模糊、必须保持一个 context，或并行 lane 大多只会制造 merge conflict，请避免 team mode。

## 用 `$ultraqa` 验证并修复

当 green build 还不够、还需要确认 hostile/edge condition 下的真实行为时，使用 [`$ultraqa`](/zh/docs/skills/workflow/ultraqa)。它会把 baseline check 与 adversarial dynamic e2e scenario 结合起来，诊断失败、应用修复、清理 temporary harness，并输出 evidence report。

```bash
$ultraqa --tests
$ultraqa --custom "the report export rejects stale state"
```

适合：

- prompt injection、cancel/resume、stale state、misleading success output 很重要的 CLI 或 workflow 变更
- 需要 failure evidence 与 cleanup proof 的 release-readiness 检查
- unit test 已通过但仍需要 dynamic behavior probe 的功能
