## 如何选择流程

OMX 有多个工作流，初次使用时光是选择就令人犯难。本页不是罗列 skill 名称的文档，而是帮你判断**当前是什么情况应该选哪个流程**。

## 一目了然的决策表

| 情况 | 首选 | 原因 |
|---|---|---|
| 需求本身模糊 | `$deep-interview` | 应先确定范围，再开始实现 |
| 有方向但没有计划 | `$ralplan` | 需要先锁定权衡取舍 |
| 有计划且要一人推进到底 | `$ralph` | 持续所有者 + 验证循环 |
| 大型计划需要按多个 milestone 长期追踪 | `$ultragoal` | `.omx/ultragoal` ledger + Codex goal handoff |
| 计划可以拆分到多个 lane | `$team` | 仅在并行化真正有收益时使用 |
| 想快速看到原型 | `$autopilot` | 放弃控制换取速度 |

几乎所有选择都可以用这张表解决。以下是各项目为何在那个位置的详细说明。

## 基本路径

初次使用 OMX，标准路径有三条。

```text
$deep-interview "帮我整理认证变更的范围"
$ralplan "请批准最安全的实现路径"
$ralph "请把批准的计划执行完毕"
```

理解这三条为何分开，就能看清 OMX 是怎么运作的。

- **`$deep-interview`** — 需求模糊时先用问题梳理清楚。比起代码，先确定"做什么"更重要。
- **`$ralplan`** — planner / architect / critic 反复迭代直到达成共识。在实现前锁定权衡取舍。
- **`$ralph`** — 拿着批准的计划，循环执行直到出现完成证据。

也可以用 `$autopilot` 一步到位，但**至少手动走一遍这三个阶段，才能真正感受 OMX 在做什么。** 之后才能判断什么时候可以走捷径。

---

## `$autopilot` 什么时候用

把一个想法从实现 → QA → 验证一次性全部委托时。

```bash
$autopilot "build me a todo app in next.js"
```

| 项目 | 状态 |
|---|---|
| 控制程度 | 低 — OMX 决定大部分流程 |
| 速度 | 快 — 一次观察整个管道 |
| 适合的情况 | 原型、小功能、end-to-end 验证 |
| 应避免的情况 | 需求模糊时、需要先做计划共识的任务 |

简单来说，`$autopilot` 是"快速看全局"，基于计划的流程是"有控制地推进"。

---

## 应该先走计划流程的时候

范围宽泛，或走错方向代价高昂的任务，延迟执行反而更快。

```text
$deep-interview "帮我整理需要做什么"
$ralplan "帮我制定最安全的计划"
$ralph "按那个计划执行到底"
```

特别适合这个路径的情况：

- 需要同时考虑架构 · 测试 · 运营影响
- 需求经过多个利益相关方尚未整理清楚
- 需要中途达成方向共识的大型任务

---

## `$ultragoal` 适合需要 durable goal 时

`$ultragoal` 会把大型 brief 或已批准 plan 转成 durable Codex/OMX goals。长期计划保存在 `.omx/ultragoal/` 下，story progress 写入 ledger，active thread 则明确 handoff 给 Codex goal mode。

![Ultragoal 将已批准 plan 连接到 durable milestones、ledger、Codex goal handoff 与 final review evidence 的流程](/images/docs/skills-workflow-ultragoal-learning-v2.png)

```bash
$ultragoal "turn this launch plan into durable goals and complete them"
```

| 项目 | 状态 |
|---|---|
| 控制程度 | 高 — plan 与 evidence 都保存在文件中 |
| 持续性 | 高 — compaction、session change、handoff 后更容易继续 |
| 适合的情况 | 多个 milestone、release train、大型 refactor、最终验证很重要的任务 |
| 应避免的情况 | 小型 one-shot edit、需求仍模糊的任务、更适合 parallel lane 的任务 |

命令细节与 completion gate 请参见 [`$ultragoal`](/zh/docs/skills/workflow/ultragoal)。

典型流程如下：

```text
$deep-interview "帮我整理产品范围"
$ralplan "按 milestone 制定安全计划并批准"
$ultragoal "把批准的计划转成 durable goals 并完成"
```

判断方式很简单：

- **计划很大，但需要按 sequential milestone 追踪** → `$ultragoal`
- **一个人用同一上下文推进到底就够了** → `$ralph`
- **可以像 frontend / backend / test 那样并行拆分** → `$team`
- **需求还不清楚** → 先用 `$deep-interview` 或 `$ralplan`

---

## `$team` 只在并行化有价值时使用

`$team` 不是"永远更快的模式"。只有**实际上可以拆分的任务**时才有价值。

```bash
$team 3:executor "请并行执行批准的计划"
```

| 项目 | 状态 |
|---|---|
| 控制程度 | 中～高 — 需要提前设计拆分方式 |
| 并行性 | 高 — 多个 lane 同时进行 |
| 适合的情况 | frontend / backend / test 这类自然分开的任务 |
| 应避免的情况 | 范围模糊、并行化只会增加冲突 |

应该避免的信号：

- 需求模糊 → 先用 `$deep-interview`
- 需要一个上下文贯穿到底 → `$ralph`
- lane 之间可能产生冲突 → 顺序处理更快
