## クイックスタート

クリーンな環境から動作する OMX セッションまで、最短で確実に進みたいときはこのページに従います。

OMX は [OpenAI Codex CLI](https://github.com/openai/codex) の上で動作します。まず Codex をインストールし、その上に要件整理、計画、実行、チーム、検証のための orchestration layer として OMX を追加します。

## 前提条件

| 項目 | 要件 |
|------|------|
| Node.js | 22+ 推奨、20+ 最小 |
| Codex CLI | インストール済み、認証済み |
| OS | macOS または Linux 推奨 |
| tmux | durable team session に必要 |

<Callout type="info">
現在の OMX ランタイムは native Windows では一貫しない場合があります。可能であれば macOS、Linux、WSL、または remote Linux 開発ホストを推奨します。
</Callout>

## Setup

最初にどの方法で OMX を使いたいかに合わせて setup path を選びます。

<Tabs items={["ローカル", "チーム", "検証"]}>
  <Tab>
    ### ローカルセッション推奨

    リポジトリで最初の実作業を行う場合はこの経路を使います。

    <Steps>
      <Step>
        #### Codex CLI をインストール

        ```bash
        npm install -g @openai/codex
        codex login
        ```
      </Step>
      <Step>
        #### OMX をインストール

        ```bash
        npm install -g oh-my-codex
        omx setup
        ```
      </Step>
      <Step>
        #### インストールを確認

        ```bash
        omx doctor
        codex login status
        ```
      </Step>
      <Step>
        #### リーダーセッションを起動

        ```bash
        omx --madmax --high
        ```
      </Step>
    </Steps>
  </Tab>
  <Tab>
    ### チームランタイム

    計画に複数 lane が必要な場合、または tmux pane で作業を維持したい場合に使います。

    ```bash
    npm install -g @openai/codex oh-my-codex
    codex login
    omx setup
    omx --tmux --madmax --high
    ```

    タスクを分割できるほど明確になってから並列実行を始めます。

    ```text
    $team 3:executor "execute the approved plan in parallel"
    ```
  </Tab>
  <Tab>
    ### 検証用スモークテスト

    CLI、認証、モデル呼び出し経路が動作するかだけを確認する場合に使います。

    ```bash
    omx doctor
    codex login status
    omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
    ```

    期待される応答は正確に `OMX-EXEC-OK` です。
  </Tab>
</Tabs>

<Callout type="warn">
`--madmax` は Codex の承認と sandboxing を迂回します。その実行境界が意図されているリポジトリと環境でのみ使ってください。
</Callout>

## 初回セッション

最初の有用なセッションでは、小さくても現実的なタスクを送り、標準の guided flow を一度通るのがおすすめです。

```text
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ultragoal "turn the approved plan into durable goals and complete them"
```

各コマンドは別々の段階を担当します。

| 段階 | 何をするか |
|------|------------|
| `$deep-interview` | コード変更前に曖昧な要件を整理 |
| `$ralplan` | より安全な実装とテスト計画を作成 |
| `$ultragoal` | 承認済み計画を durable goal、ledger evidence、completion checkpoint に変換 |

承認済み計画が自然に並列化できる場合は、最後のコマンドを次に置き換えます。

```text
$team 3:executor "execute the approved plan in parallel"
```

### 最初のプロンプト例

```text
$deep-interview "help me scope a safer signup flow change"
$ralplan "plan the least risky way to add this report export"
$ultragoal "split the approved report export plan into durable goals and complete them"
```
