## Getting started

If this is your first time with oh-my-codex (OMX), follow the steps below.

1. [Install](#install) — install Codex CLI and OMX, then run `omx setup`
2. [First session](#first-session) — check execution boundaries and run the recommended workflow

OMX is a workflow layer that runs on top of [OpenAI Codex CLI](https://github.com/openai/codex). It keeps Codex as the execution engine and adds role-specific agent prompts, skills, and hooks on top, so a vague idea can move to working code in one coherent loop.

### Prerequisites

| Item | Requirement |
|------|----------|
| Node.js | 22+ recommended (20+ minimum) |
| [Codex CLI](https://github.com/openai/codex) | Installed and authenticated |
| OS | macOS or Linux recommended |
| tmux | Required for the durable team runtime |

<Callout type="info">
Native Windows may behave inconsistently or break. Prefer macOS or Linux when possible.
</Callout>

## Install

### Install Codex CLI

First install Codex CLI.

```bash
npm install -g @openai/codex
```

Then authenticate.

```bash
codex login
```

### Install OMX

Then install OMX.

```bash
npm install -g oh-my-codex
```

### Run setup

Once both are installed, run setup.

```bash
omx setup
```

`omx setup` handles the following automatically:

- Writes `.codex/config.toml` and enables `codex_hooks = true`
- Registers OMX hook commands in `.codex/hooks.json`
- Creates the `.omx/` project state directory
- Wires OMX agent context into the session prompt

### Verify the install

Confirm the install worked.

```bash
omx doctor
```

`omx doctor` checks for missing files, hooks, and local runtime prerequisites.

## First session

With the install verified, confirm the execution boundaries and start your first session.

### Check execution boundaries

Confirm the current shell and config can actually make a model call.

```bash
codex login status
omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
```

- `codex login status` shows the current auth state.
- The `omx exec` smoke test proves the active Codex profile can send a real request and receive the full response.

### Recommended launch

OMX recommends starting this way.

```bash
omx --madmax --high
```

If you explicitly want the leader session inside tmux:

```bash
omx --tmux --madmax --high
```

<Callout type="warn">
`--madmax` bypasses Codex approvals and sandboxing. Use it intentionally.
</Callout>

### Run the default workflow

For the first session, this is the recommended sequence.

```bash
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ralph "carry the approved plan to completion"
```

Each step means:

| Step | Role |
|------|------|
| `$deep-interview` | Clarify the task through questioning when requirements are fuzzy |
| `$ralplan` | Agree on the safest plan before implementation |
| `$ralph` | Push the approved plan through to completion |

If the approved plan needs parallel execution, swap the last step for:

```bash
$team 3:executor "execute the approved plan in parallel"
```

- Parallel work → `$team`
- One persistent owner carrying it all the way through → `$ralph`
