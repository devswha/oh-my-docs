## 快速开始

如果是第一次用 oh-my-codex (OMX)，按下面的顺序走:

1. [安装](#install) — 安装 Codex CLI 与 OMX，然后运行 `omx setup`
2. [首次会话](#first-session) — 确认执行边界并跑通推荐的工作流

OMX 是运行在 [OpenAI Codex CLI](https://github.com/openai/codex) 之上的工作流层。它把 Codex 保留为执行引擎，叠加角色化的 agent prompt、skill 与 hook，让一个模糊想法可以在同一个循环里跑到可运行的代码。

### 前置条件

| 项目 | 要求 |
|------|----------|
| Node.js | 建议 22+ (最低 20+) |
| [Codex CLI](https://github.com/openai/codex) | 必须已安装并完成认证 |
| 操作系统 | 建议 macOS 或 Linux |
| tmux | 使用持久化 team runtime 时需要 |

<Callout type="info">
原生 Windows 可能出现不一致或异常行为。建议尽量使用 macOS 或 Linux。
</Callout>

## 安装

### 安装 Codex CLI

先安装 Codex CLI。

```bash
npm install -g @openai/codex
```

然后完成认证。

```bash
codex login
```

### 安装 OMX

接着安装 OMX。

```bash
npm install -g oh-my-codex
```

### 运行 setup

两边都装好以后运行 setup。

```bash
omx setup
```

`omx setup` 会自动完成以下工作:

- 写入 `.codex/config.toml` 并启用 `codex_hooks = true`
- 在 `.codex/hooks.json` 注册 OMX 的 hook 命令
- 创建 `.omx/` 项目状态目录
- 把 OMX agent 上下文接入会话 prompt

### 验证安装

确认安装正常。

```bash
omx doctor
```

`omx doctor` 会检查缺失的文件、hook 以及本地运行时前置条件。

## 首次会话

安装通过后，先确认执行边界，再开始第一次会话。

### 确认执行边界

确认当前 shell 与配置真的能发起模型调用。

```bash
codex login status
omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
```

- `codex login status` 显示当前认证状态。
- `omx exec` 这一次冒烟测试证明当前活跃的 Codex profile 确实可以发出请求并完整收回响应。

### 推荐启动方式

OMX 推荐用下面的方式启动。

```bash
omx --madmax --high
```

如果要在 tmux 内明确拉起 leader 会话:

```bash
omx --tmux --madmax --high
```

<Callout type="warn">
`--madmax` 会绕过 Codex 的审批和沙箱。请有意识地使用。
</Callout>

### 运行标准工作流

首次会话建议按下面的顺序跑。

```bash
$deep-interview "澄清认证改动"
$ralplan "批准最安全的实现路径"
$ralph "把已批准的计划推进到完成"
```

每一步的意义:

| 步骤 | 作用 |
|------|------|
| `$deep-interview` | 需求模糊时通过提问澄清 |
| `$ralplan` | 实现前敲定最安全的计划 |
| `$ralph` | 持续推动已批准的计划，直到完成 |

已批准的计划需要并行执行时，把最后一步换成:

```bash
$team 3:executor "并行执行已批准的计划"
```

- 需要并行工作时 → `$team`
- 需要一个 owner 把事情扛到底时 → `$ralph`
