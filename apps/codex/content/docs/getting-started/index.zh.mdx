## 快速开始

如果你想从干净环境最短路径进入一个可用的 OMX 会话，请按照本页操作。

OMX 运行在 [OpenAI Codex CLI](https://github.com/openai/codex) 之上。先安装 Codex，再把 OMX 作为需求澄清、规划、执行、团队协作和验证的 orchestration layer 加上去。

## 前提条件

| 项目 | 要求 |
|------|------|
| Node.js | 推荐 22+，最低 20+ |
| Codex CLI | 已安装并完成认证 |
| OS | 推荐 macOS 或 Linux |
| tmux | durable team session 需要 |

<Callout type="info">
当前 OMX 运行时在 native Windows 上可能不一致。建议优先使用 macOS、Linux、WSL 或 remote Linux 开发主机。
</Callout>

## Setup

根据第一次想如何使用 OMX 来选择 setup path。

<Tabs items={["本地", "团队", "验证"]}>
  <Tab>
    ### 推荐本地会话

    在仓库中第一次处理真实任务时使用这条路径。

    <Steps>
      <Step>
        #### 安装 Codex CLI

        ```bash
        npm install -g @openai/codex
        codex login
        ```
      </Step>
      <Step>
        #### 安装 OMX

        ```bash
        npm install -g oh-my-codex
        omx setup
        ```
      </Step>
      <Step>
        #### 检查安装

        ```bash
        omx doctor
        codex login status
        ```
      </Step>
      <Step>
        #### 启动 leader 会话

        ```bash
        omx --madmax --high
        ```
      </Step>
    </Steps>
  </Tab>
  <Tab>
    ### 团队运行时

    当计划需要多个 lane，或需要通过 tmux pane 保持工作时使用。

    ```bash
    npm install -g @openai/codex oh-my-codex
    codex login
    omx setup
    omx --tmux --madmax --high
    ```

    只有当任务已经清晰到可以拆分时，再启动并行执行。

    ```text
    $team 3:executor "execute the approved plan in parallel"
    ```
  </Tab>
  <Tab>
    ### 仅验证的 smoke test

    只想证明 CLI、认证和模型调用路径可用时使用。

    ```bash
    omx doctor
    codex login status
    omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
    ```

    预期响应必须准确为 `OMX-EXEC-OK`。
  </Tab>
</Tabs>

<Callout type="warn">
`--madmax` 会绕过 Codex approvals 和 sandboxing。只在你明确接受该执行边界的仓库和环境中使用。
</Callout>

## 首次会话

第一次有用的会话，建议发送一个小但真实的任务，并完整走一次默认 guided flow。

```text
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ultragoal "turn the approved plan into durable goals and complete them"
```

每个命令负责不同阶段。

| 阶段 | 作用 |
|------|------|
| `$deep-interview` | 在代码变更前澄清模糊需求 |
| `$ralplan` | 生成更安全的实现和测试计划 |
| `$ultragoal` | 将已批准计划转成 durable goal、ledger evidence 与 completion checkpoint |

如果已批准计划天然适合并行，把最后一个命令替换为：

```text
$team 3:executor "execute the approved plan in parallel"
```

### 第一个提示示例

```text
$deep-interview "help me scope a safer signup flow change"
$ralplan "plan the least risky way to add this report export"
$ultragoal "split the approved report export plan into durable goals and complete them"
```
