## 시작하기

oh-my-codex(OMX)를 처음 쓴다면, 아래 순서대로 따라가면 됩니다.

1. [설치](#설치-방법) - Codex CLI와 OMX 설치, `omx setup` 실행
2. [첫 세션](#첫-세션-시작하기) - 실행 경계를 확인하고 권장 워크플로우 실행

OMX는 [OpenAI Codex CLI](https://github.com/openai/codex) 위에서 동작하는 워크플로우 레이어입니다. Codex를 실행 엔진으로 유지하면서 역할별 에이전트 프롬프트, 스킬, 훅을 얹어 주기 때문에, 막연한 아이디어에서 동작하는 코드까지 하나의 일관된 루프로 가져갈 수 있습니다.

### 사전 요구사항

| 항목 | 요구사항 |
|------|----------|
| Node.js | 22+ 권장 (최소 20+) |
| [Codex CLI](https://github.com/openai/codex) | 설치 및 인증 필수 |
| 운영체제 | macOS 또는 Linux 권장 |
| tmux | durable team runtime을 쓸 경우 필요 |

<Callout type="info">
네이티브 Windows에서는 동작이 깨지거나 일관되지 않을 수 있습니다. 가능하면 macOS 또는 Linux 환경에서 시작하는 것을 권장합니다.
</Callout>

## 설치 방법

### Codex CLI 설치

먼저 Codex CLI를 설치합니다.

```bash
npm install -g @openai/codex
```

설치 후 인증합니다.

```bash
codex login
```

### OMX 설치

그다음 OMX를 설치합니다.

```bash
npm install -g oh-my-codex
```

### setup 실행

설치가 끝났다면 setup을 실행합니다.

```bash
omx setup
```

`omx setup`은 다음을 자동으로 처리합니다.

- `.codex/config.toml` 작성 및 `codex_hooks = true` 활성화
- `.codex/hooks.json`에 OMX 훅 명령 등록
- `.omx/` 프로젝트 상태 디렉터리 생성
- 세션 프롬프트에 OMX 에이전트 컨텍스트 연결

### 설치 확인

설치가 제대로 되었는지 먼저 확인합니다.

```bash
omx doctor
```

`omx doctor`는 누락된 파일, 훅, 로컬 런타임 전제 조건을 점검합니다.

## 첫 세션 시작하기

설치 확인이 끝났다면, 이제 실행 경계를 확인하고 첫 세션을 시작해 보면 됩니다.

### 실행 경계 확인

아래 명령으로 현재 셸과 설정에서 실제 모델 호출이 가능한지 확인합니다.

```bash
codex login status
omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
```

- `codex login status`는 현재 인증 상태를 확인합니다.
- `omx exec` 스모크 테스트는 활성화된 Codex 프로필이 실제로 요청을 보내고 응답을 끝까지 받을 수 있는지 확인합니다.

### 권장 실행 방식

OMX는 아래 방식으로 시작하는 것을 권장합니다.

```bash
omx --madmax --high
```

리더 세션을 tmux 안에서 명시적으로 띄우고 싶다면:

```bash
omx --tmux --madmax --high
```

<Callout type="warn">
`--madmax`는 Codex 승인과 샌드박싱을 우회합니다. 의도적으로 사용할 때만 켜세요.
</Callout>

### 기본 워크플로우 실행

첫 세션에서는 아래 순서를 추천합니다.

```bash
$deep-interview "인증 변경 사항을 명확히 해줘"
$ralplan "가장 안전한 구현 경로를 승인해줘"
$ralph "승인된 계획을 끝까지 완료해줘"
```

각 단계의 의미는 다음과 같습니다.

| 단계 | 역할 |
|------|------|
| `$deep-interview` | 요구사항이 모호할 때 질문으로 생각을 정리 |
| `$ralplan` | 구현 전에 가장 안전한 계획을 합의 |
| `$ralph` | 승인된 계획을 계속 밀고 가며 완료까지 수행 |

승인된 계획에 병렬 실행이 필요하다면 마지막 단계를 아래처럼 바꾸면 됩니다.

```bash
$team 3:executor "승인된 계획을 병렬로 실행해줘"
```

- 병렬 작업이 필요하면 `$team`
- 한 명의 지속적인 소유자가 끝까지 밀어붙이는 흐름이면 `$ralph`
