## 빠른 시작

깨끗한 환경에서 동작하는 OMX 세션까지 가장 짧고 안전하게 가고 싶을 때 이 페이지를 따라가면 됩니다.

OMX는 [OpenAI Codex CLI](https://github.com/openai/codex) 위에서 실행됩니다. 먼저 Codex를 설치하고, 그 위에 요구사항 정리, 계획, 실행, 팀, 검증을 위한 orchestration layer로 OMX를 추가합니다.

## 사전 요구사항

| 항목 | 요구사항 |
|------|----------|
| Node.js | 22+ 권장, 최소 20+ |
| Codex CLI | 설치 및 인증 완료 |
| 운영체제 | macOS 또는 Linux 권장 |
| tmux | durable team session에 필요 |

<Callout type="info">
현재 OMX 런타임은 native Windows에서 일관되지 않을 수 있습니다. 가능하면 macOS, Linux, WSL, 또는 remote Linux 개발 호스트를 권장합니다.
</Callout>

## Setup

처음 어떤 방식으로 OMX를 사용할지에 맞춰 setup path를 고릅니다.

<Tabs items={["로컬", "팀", "검증"]}>
  <Tab>
    ### 로컬 세션 추천

    저장소에서 첫 실제 작업을 할 때 이 경로를 사용합니다.

    <Steps>
      <Step>
        #### Codex CLI 설치

        ```bash
        npm install -g @openai/codex
        codex login
        ```
      </Step>
      <Step>
        #### OMX 설치

        ```bash
        npm install -g oh-my-codex
        omx setup
        ```
      </Step>
      <Step>
        #### 설치 확인

        ```bash
        omx doctor
        codex login status
        ```
      </Step>
      <Step>
        #### 리더 세션 실행

        ```bash
        omx --madmax --high
        ```
      </Step>
    </Steps>
  </Tab>
  <Tab>
    ### 팀 런타임

    계획에 여러 lane이 필요하거나 tmux pane을 통해 작업을 오래 유지해야 할 때 사용합니다.

    ```bash
    npm install -g @openai/codex oh-my-codex
    codex login
    omx setup
    omx --tmux --madmax --high
    ```

    작업을 나눌 만큼 명확해진 뒤에만 병렬 실행을 시작합니다.

    ```text
    $team 3:executor "execute the approved plan in parallel"
    ```
  </Tab>
  <Tab>
    ### 검증용 스모크 테스트

    CLI, 인증, 모델 호출 경로가 동작하는지만 확인할 때 사용합니다.

    ```bash
    omx doctor
    codex login status
    omx exec --skip-git-repo-check -C . "Reply with exactly OMX-EXEC-OK"
    ```

    기대 응답은 정확히 `OMX-EXEC-OK`입니다.
  </Tab>
</Tabs>

<Callout type="warn">
`--madmax`는 Codex 승인과 sandboxing을 우회합니다. 해당 실행 경계가 의도된 저장소와 환경에서만 사용하세요.
</Callout>

## 첫 세션

첫 유용한 세션에서는 작지만 실제적인 작업을 보내고 기본 guided flow를 한 번 타보는 것이 좋습니다.

```text
$deep-interview "clarify the authentication change"
$ralplan "approve the safest implementation path"
$ultragoal "turn the approved plan into durable goals and complete them"
```

각 명령은 서로 다른 단계를 맡습니다.

| 단계 | 하는 일 |
|------|---------|
| `$deep-interview` | 코드 변경 전에 모호한 요구사항을 정리 |
| `$ralplan` | 더 안전한 구현 및 테스트 계획 작성 |
| `$ultragoal` | 승인된 계획을 durable goal, ledger evidence, completion checkpoint로 전환 |

승인된 계획이 자연스럽게 병렬화된다면 마지막 명령을 아래처럼 바꿉니다.

```text
$team 3:executor "execute the approved plan in parallel"
```

### 첫 프롬프트 예시

```text
$deep-interview "help me scope a safer signup flow change"
$ralplan "plan the least risky way to add this report export"
$ultragoal "split the approved report export plan into durable goals and complete them"
```
