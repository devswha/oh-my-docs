## 개요

스킬은 OMX에서 작업을 시작하는 입구입니다. 사용자는 보통 `$name` 또는 매직 키워드로 호출하고, 스킬이 안에서 필요한 에이전트 · 훅 · 도구를 순서대로 꺼내 씁니다.

스킬은 명령이 아니라 **"어떤 역할을 어떤 순서로 부를지" 정해둔 워크플로우 단위**입니다.

## 두 종류

### 워크플로우 스킬

실제 작업을 진행하는 실행 모드.

| 스킬 | 용도 |
|---|---|
| [`$deep-interview`](/ko/docs/skills/workflow/deep-interview) | 요구가 모호할 때 질문으로 정리 |
| [`$ralplan`](/ko/docs/skills/workflow/ralplan) | planner · architect · critic 합의 계획 |
| [`$ralph`](/ko/docs/skills/workflow/ralph) | 완료 증거가 나올 때까지 반복 실행 |
| [`$team`](/ko/docs/skills/workflow/team) | 병렬 lane으로 쪼개서 동시 실행 |
| [`$autopilot`](/ko/docs/skills/workflow/autopilot) | 아이디어 → 구현 → QA → 검증 일괄 |
| [`$ultraqa`](/ko/docs/skills/workflow/ultraqa) | 테스트 · 검증 · 수정 사이클 |

### 유틸리티 스킬

실행 흐름을 보조하는 도구형 스킬.

- [`$analyze`](/ko/docs/skills/utility/analyze) — 읽기 중심 분석
- [`$ask-claude`](/ko/docs/skills/utility/ask-claude) / [`$ask-gemini`](/ko/docs/skills/utility/ask-gemini) — 다른 모델에 자문
- [`$doctor`](/ko/docs/skills/utility/doctor) — 설치와 환경 진단
- [`$hud`](/ko/docs/skills/utility/hud) — HUD 확인과 설정
- [`$note`](/ko/docs/skills/utility/note) — 세션 메모 저장
- [`$cancel`](/ko/docs/skills/utility/cancel) — 활성 모드 종료

## 어디서부터 익힐까

전부 외우려 하지 말고 기본 흐름 하나만 몸에 붙이면 됩니다.

```text
$deep-interview "인증 변경 사항을 명확히 해줘"
$ralplan "가장 안전한 구현 경로를 승인해줘"
$ralph "승인된 계획을 끝까지 완료해줘"
```

이 셋이 익숙해지면 나머지는 상황에 따라 선택하면 됩니다.

| 상황 | 스킬 |
|---|---|
| 무엇을 만들지 모호 | `$deep-interview` |
| 계획부터 합의 | `$ralplan` |
| 한 명이 끝까지 밀어야 함 | `$ralph` |
| 병렬화 가능 | `$team` |
| 전 과정 일괄 | `$autopilot` |

## `$autopilot` vs 계획 기반 흐름

가장 자주 혼동되는 구분입니다.

**`$autopilot`** — 속도 우선, 제어 내려놓기.

```bash
$autopilot "build me a todo app in next.js"
```

잘 맞는 경우: 프로토타입, 작은 기능, end-to-end 확인.

**`$deep-interview → $ralplan → $ralph`** — 제어 우선, 단계 분리.

잘 맞는 경우: 범위가 넓은 작업, 방향 합의가 먼저인 작업, 잘못 가면 비용이 큰 작업.

## `$team`은 분할이 실제로 가능할 때만

```bash
$team 3:executor "승인된 계획을 병렬로 실행해줘"
```

잘 맞는 경우:

- frontend / backend / test처럼 자연스럽게 나뉘는 작업
- 순차 처리로는 속도가 안 나오는 큰 작업
- 계획이 이미 있고 lane으로 분배 가능한 경우

반대로 요구가 흐리거나 lane끼리 충돌할 것 같으면 `$ralph`가 더 안전합니다.
