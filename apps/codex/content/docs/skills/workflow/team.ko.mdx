`$team`은 여러 워커 세션을 tmux 창에 병렬로 띄우고, 공유 상태 디렉토리를 통해 조율합니다. 각 워커는 태스크 목록에서 작업을 가져와 브랜치에 커밋하고 mailbox로 보고합니다. 리더는 진행 상황을 모니터링하다가 모든 태스크가 완료되면 팀을 종료합니다.

한 명이 순차적으로 처리하는 것보다 lane을 나눠 병렬 실행하는 편이 빠른 작업에 씁니다.

## 언제 쓰나

- docs, tests, 구현, 리뷰처럼 독립적으로 나눌 수 있는 lane이 있을 때
- `$ralplan`에서 승인된 계획이 병렬 실행을 권장할 때
- 단일 reasoning burst를 넘어서는 지속 상태가 필요할 때
- 시작 → 모니터링 → 종료 → 정리의 명시적 lifecycle이 필요할 때

트리거 키워드: "team", "coordinated workers", "parallel agents", "spawn workers"

## 호출 방법

```
codex
> team 3:executor "implement the three modules from the approved plan"
```

```bash
omx team 2:executor "add validation and write tests in parallel"
```

Claude 워커를 쓰고 싶으면:

```bash
OMX_TEAM_WORKER_CLI=claude omx team 2:executor "update docs and report"
```

## 동작 방식

워커 실행 전에 `.omx/context/{slug}-{timestamp}.md`에 컨텍스트 스냅샷이 먼저 만들어져야 합니다.

오케스트레이터는 워커 수와 역할을 파싱하고, `.omx/state/team/{team}/` 아래에 팀 상태 파일을 초기화합니다. `AGENTS.md`를 수정하지 않고 워커 지시사항을 구성한 뒤 tmux 창을 분할하고 각 워커를 실행합니다. 각 워커는 독립된 환경 변수로 공유 상태 루트를 가리킵니다.

워커는 `omx team api claim-task`로 태스크를 가져와 작업한 뒤 `git add -A && git commit`으로 커밋하고 리더 mailbox에 ACK을 보냅니다. 리더는 `omx team status {team}`으로 진행을 모니터링하다가 `pending=0`, `in_progress=0`이 되면 `omx team shutdown {team}`을 실행합니다.

파이프라인은 team-plan → team-exec → team-verify → team-fix 순서로 진행하며, 모든 lane이 green이 되거나 상한에 도달할 때까지 fix 루프를 반복합니다.

## 출력

- 각 워커가 브랜치에 커밋한 소스 코드, 테스트, 문서
- `.omx/state/team/{team}/` — 태스크 상태, 워커 heartbeat, mailbox, manifest
- `.omx/context/{slug}-{timestamp}.md` — 전체 워커가 공유하는 컨텍스트 스냅샷
- 종료 전 태스크 완료를 확인하는 `omx team status` 증거
