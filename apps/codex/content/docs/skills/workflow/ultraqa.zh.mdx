title: $ultraqa

---

`$ultraqa` 在 v0.17 中升级为对抗式 QA 工作流。它仍会运行常规验证命令，但不再因为 build/lint/typecheck/test 变绿就宣告完成。只要目标可以安全运行、模拟或用 harness 覆盖，`$ultraqa` 就必须执行 dynamic end-to-end scenario、hostile user modeling、cleanup check 和 structured evidence report。

## 何时使用

- 你需要比“测试通过”更强的证据。
- 功能涉及 CLI、workflow state、MCP tool、agent、prompt、setup、hook 或 user-facing flow。
- 你希望诊断失败、精确修复，并 rerun 到目标达成或 bounded stop condition。
- 你想在 review 前捕捉 stale state、prompt injection、cancel/resume、misleading success 或 dirty worktree regression。

触发关键词：`ultraqa`、`fix until tests pass`、`qa cycle`、`make the build pass`。

## 调用方式

```text
codex
> $ultraqa --tests
```

```text
codex
> $ultraqa --build
```

```text
codex
> $ultraqa --custom "the CLI rejects stale session state"
```

可用 goal flag：`--tests`、`--build`、`--lint`、`--typecheck`、`--custom "pattern"`、`--interactive`。

## v0.17 contract

在声明成功前，`$ultraqa` 会创建并维护包含以下列的 scenario matrix。

| Column | 含义 |
|---|---|
| Scenario ID | 稳定 ID，例如 `ADV-E2E-003` |
| Intent | 要证明的风险或行为 |
| User/attacker model | normal user、careless operator、malicious prompt、stale runtime、flaky environment |
| Setup | 所需 fixture、state、service、branch 或 harness |
| Command/harness | 精确 command、script、browser step 或 generated harness |
| Expected signal | 证明成功的 exit code、output、UI state、artifact 或 state transition |
| Actual result | 观察到的 output 与 exit status |
| Fixes applied | 关联修复或 `none` |
| Evidence | log、test output、screenshot、artifact 或 transcript excerpt |
| Cleanup | 已删除、故意保留，或因原因 blocked |

## 必需 scenario class

包含正常路径，以及相关且安全的对抗类场景。

1. Malformed input：invalid JSON、missing field、bad flag、oversized string、unusual Unicode、path traversal-like value、corrupted state。
2. Repeated interruption：重复 `continue`、stop/cancel/abort 表达、中断的 command output、partial progress 后 retry。
3. Prompt injection：尝试覆盖 instruction、跳过验证、泄露 secret、删除 state 或宣称 false success。
4. Cancel/resume：active-state cleanup、resume detection、stale in-progress state、cancellation 后 fresh run。
5. Stale state：旧 `.omx/state` file、mismatched session、missing timestamp、contradictory phase metadata。
6. Dirty worktree：已有修改、untracked generated file，以及未覆盖或隐藏 unrelated work 的证据。
7. Hung command：explicit timeout、killed child process、recovery note。
8. Flaky test：rerun strategy、failure clustering，避免一次幸运通过造成 false green。
9. Misleading success output：success-looking text 但 non-zero exit、skipped test、hidden failure 或 truncated log。

## Dynamic harness 规则

- 当现有测试无法覆盖行为时，生成 temporary test、script、fixture 或 harness。
- 优先使用 project-native test tool 和小型 throwaway harness。
- 将每个生成 artifact 记录到 scenario matrix。
- 对可能 hang 的 command 使用 bounded timeout。
- 同时验证 exit code 与 output semantics；不要只相信看起来成功的文本。
- 不要删除、重写或掩盖 unrelated user work。

## Cycle flow

1. 规划 adversarial QA：说明 goal、success criteria、safety bounds、stop condition、runnable surface 和 scenario matrix。
2. 运行 baseline verification：tests、build、lint、typecheck 或 custom command。
3. 运行 matrix 中的 dynamic e2e scenario。
4. 用 architecture-level root cause 和 safety impact 诊断失败。
5. 应用精确 fix。
6. 清理 temporary harness、state、log 和 process，除非它们被有意保留。
7. 重复直到目标达成、5 个 cycle 用尽、同一失败重复 3 次，或被 safety boundary 阻止。

## Completion report

最终 `$ultraqa` report 应包含：

- Goal and success criteria
- Scenario matrix
- 带 exit code 和关键 evidence 的 command 列表
- 发现的 failure 与 root cause
- 已应用 fix 与 regression evidence
- Cleanup and rollback status
- Residual risk 或 blocked scenario
- 必要时包含 evidence link、log、screenshot、transcript、artifact
