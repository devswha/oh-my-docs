# OMC 문서 톤 레퍼런스 (언어별)

이 문서는 sibling repo인 **`oh-my-claudecode-docs`**의 문장 톤과 문서 구조를 언어별로 정리한 maintainer용 참고 문서입니다.

목적은 OMC의 **독자 경험(reader experience)** 을 OMX 문서 작업에 재사용하는 것입니다.  
단, OMC 전용 기능/명령/사실을 복사하라는 뜻은 아닙니다.

## 이 문서가 필요한 이유

OMX 문서를 OMC 기준으로 정리할 때, 단순 번역이나 단순 구조 복사만으로는 톤이 맞지 않는 경우가 많습니다.

예를 들어:

- 어떤 언어는 더 **온보딩형 / 안내형** 문장 흐름이 강하고
- 어떤 언어는 더 **압축된 기술 문서형**에 가깝고
- 어떤 언어는 sidebar나 제목에서 **영어 product label을 더 적극적으로 유지**합니다.

그래서 OMC를 참조할 때는 다음을 구분해야 합니다.

- **가져와야 할 것**: 문장 톤, 섹션 순서, 독자 안내 방식, 표/예시/다이어그램 사용 감각
- **가져오면 안 되는 것**: OMC 전용 명령, OMC 전용 사실, OMC 고유 런타임 가정

## 사용 규칙

OMX 문서를 OMC와 맞출 때는 다음 순서를 권장합니다.

1. 대상 로케일의 OMC 문서를 먼저 읽는다.
2. 해당 로케일에서 보이는 **톤 / 독자 안내 방식 / 구조**를 파악한다.
3. 한국어 기준 리뷰가 필요한 경우, **한국어 페이지를 먼저 수정하고 승인받는다.**
4. 승인된 구조를 영어에 반영한다.
5. 그 뒤 일본어/중국어에 각 언어에 맞는 톤으로 반영한다.

## 공통적으로 보이는 OMC 문서 특성

영어, 한국어, 일본어, 중국어 페이지를 공통으로 보면 OMC 문서는 대체로 다음 성향을 가집니다.

- **가이드형 / 온보딩형** 문장으로 시작한다.
- 독자가 현재 어떤 단계에 있는지 먼저 잡아 준다.
- 섹션 초반에 **"왜 이 문서를 읽어야 하는가"** 를 설명한다.
- 예시, 표, 단계형 설명을 비교적 빠르게 배치한다.
- 단순 카탈로그 요약보다 **실전 적용 흐름**을 우선한다.
- 명령은 그냥 던지지 않고, **언제/왜 쓰는지** 를 먼저 설명한다.

즉, OMC 문서는 전반적으로 **"참조용 목록"보다 "바로 따라갈 수 있는 안내서"** 쪽에 더 가깝습니다.

## 영어 톤

### 전반 인상
OMC 영어 문서는 **product-facing onboarding guide**에 가깝습니다.

### 특징
- 짧고 직접적인 문장을 많이 씁니다.
- 독자에게 지금 무엇을 하면 되는지 바로 말합니다.
- 비슷한 surface를 초반에 비교해서 혼동을 줄입니다.
- "That's it. Everything else is automatic." 같은 짧은 확신형 문장을 잘 씁니다.
- 구조는 보통 **install → setup → run → inspect** 순서로 자연스럽게 이어집니다.

### 문장 감각
- 지나치게 딱딱하지 않음
- 짧고 명령형에 가까운 안내 문장
- 처음 읽는 사람도 바로 이해할 수 있게 함

### 활용 가이드
OMX 영어 문서에 반영할 때는:
- intro를 짧고 명확하게 쓴다.
- CLI vs in-session 같은 헷갈리는 surface는 초반에 표나 비교 문장으로 분리한다.
- page title 아래 문단이 너무 reference-like 하지 않게 한다.

### 관찰 근거 예시
- `oh-my-claudecode-docs/content/docs/index.mdx`
- `oh-my-claudecode-docs/content/docs/getting-started/index.mdx`
- `oh-my-claudecode-docs/content/docs/guides/index.mdx`

## 한국어 톤

### 전반 인상
OMC 한국어 문서는 **실전 안내형 기술 문서**에 가깝습니다.

### 특징
- 독자가 지금 어느 학습 단계에 있는지 먼저 잡아 줍니다.
  - 예: "기본 개념을 이해했다면…"
  - 예: "이제 실전에서 써봅시다"
- 완전한 직역보다, 읽히는 한국어 안내 문장을 우선합니다.
- product name, 명령어, mode 이름은 영어를 자연스럽게 유지합니다.
- 영어 label과 한국어 설명을 섞는 데 비교적 적극적입니다.
- 표와 단계형 목록을 자주 사용합니다.

### 문장 감각
- 한국어 기술 문서치고는 다소 친절한 편
- 선생님처럼 흐름을 잡아 주는 문장
- 너무 딱딱한 reference 문장보다 안내형 문장을 선호

### 구조적 특징
- `언제 쓰는가`, `사용 예시`, `프로세스`, `추천 순서` 같은 제목이 잘 맞습니다.
- sidebar에서도 `Guides`, `Skills`, `Tools` 같은 영어 라벨을 일부 유지하는 것이 어색하지 않을 수 있습니다.
- intro 문단은 "이 문서가 독자에게 어떤 도움을 주는가"를 먼저 말할수록 OMC에 가까워집니다.

### 활용 가이드
OMX 한국어 문서에 반영할 때는:
- 한국어를 **첫 승인 surface**로 삼는다.
- 직역보다 읽히는 문장을 우선한다.
- product surface와 명령어는 영어 유지에 관대해도 된다.
- "무엇을 설명하는 문서인가"보다 **"지금 읽으면 무엇이 풀리는가"** 를 먼저 말한다.

### 관찰 근거 예시
- `oh-my-claudecode-docs/content/docs/index.ko.mdx`
- `oh-my-claudecode-docs/content/docs/getting-started/index.ko.mdx`
- `oh-my-claudecode-docs/content/docs/guides/index.ko.mdx`

### 보충 메모
- 한국어 문서는 유지보수자/운영자 시점의 기술 문서 톤이 강합니다.
- 짧은 요약형 문장, mixed-language technical Korean, 그리고 표/코드/다이어그램 중심 설명이 잘 맞습니다.
- sidebar에서도 일부 product-facing label을 영어로 남기는 패턴이 자연스럽습니다.

## 일본어 톤

### 전반 인상
OMC 일본어 문서는 **정돈된 기술 가이드형**에 가깝습니다.

### 특징
- product term은 영어를 유지하는 편입니다.
- 설명은 한국어보다 덜 코칭형이고, 더 정리된 매뉴얼 느낌이 있습니다.
- 과하게 친근하지 않고, 적당히 절제된 기술 문체를 유지합니다.
- section progression이 중요하고, 과한 장황함은 피하는 편입니다.

### 문장 감각
- 한국어보다 다소 차분하고 절제됨
- 구조를 따라 읽기 쉽게 정리됨
- onboarding 성격은 있지만 노골적인 코칭 톤은 덜함

### 활용 가이드
OMX 일본어 문서에 반영할 때는:
- 불필요한 감탄/강한 확신형 문장은 줄인다.
- English identifier는 product recognition을 위해 유지한다.
- intro는 짧게, 구조는 또렷하게 잡는다.

### 관찰 근거 예시
- `oh-my-claudecode-docs/content/docs/index.ja.mdx`
- `oh-my-claudecode-docs/content/docs/getting-started/index.ja.mdx`

## 중국어 톤 (간체)

### 전반 인상
OMC 중국어 문서는 **압축적이고 실용적인 제품 문서형**에 가깝습니다.

### 특징
- 정보 밀도가 높고 문장이 비교적 짧습니다.
- 설명은 빠르게 핵심으로 들어가고, 표/예시로 구조를 정리합니다.
- product term과 identifier는 영어를 유지하는 경우가 많습니다.
- 한국어보다 덜 설명적이고, 더 빠르게 actionable structure로 진입합니다.

### 문장 감각
- 간결하고 밀도 높음
- 군더더기 적은 매뉴얼 느낌
- 안내형이지만 대화체보다는 정보형에 가까움

### 활용 가이드
OMX 중국어 문서에 반영할 때는:
- 장황한 intro를 피한다.
- 표와 구조화된 bullet을 적극적으로 사용한다.
- 한 문단에 너무 많은 설명을 싣기보다 압축된 문장을 유지한다.

### 관찰 근거 예시
- `oh-my-claudecode-docs/content/docs/index.zh.mdx`
- `oh-my-claudecode-docs/content/docs/getting-started/index.zh.mdx`

### 보충 메모
- 현재 zh 문서는 부분 번역 상태가 강하고, English label / identifier가 광범위하게 섞여 있습니다.
- 따라서 zh parity 작업에서는 "자연스러운 중국어 문장"보다도 먼저, **운영 문서로서의 압축성과 혼용 규칙**을 맞추는 것이 중요합니다.

## 영어 라벨 / 혼용 규칙

OMC localized docs를 보면 다음 경향이 있습니다.

- sidebar title 일부는 영어를 유지합니다. (`Guides`, `Skills`, `Tools` 등)
- workflow name, command, agent ID는 영어를 유지합니다.
- 설명 문장은 현지어로 충분히 풀어 쓰되, product-facing label은 영어를 남기는 경우가 많습니다.

### OMX 적용 규칙
추천 규칙은 다음과 같습니다.

- product-facing identifier는 영어 유지
- 설명 문장은 현지 언어로 적극 변환
- 영어를 남길 때는 우연이 아니라 **의도된 제품 인지성**으로 남긴다
- 어색한 반쪽 번역보다, 명확한 혼용 패턴을 택한다

## OMC에서 절대 복사하면 안 되는 것

다음은 OMC tone reference를 보더라도 그대로 가져오면 안 됩니다.

- OMC 전용 설치 흐름
- OMC 전용 명령
- OMC 에이전트 / 스킬 수치
- OMC 런타임 구조를 전제로 한 설명
- OMC에만 있는 파일 경로나 상태 디렉터리 구조

즉, **톤과 독자 경험은 가져오되, 제품 사실은 OMX 기준으로 다시 써야 합니다.**

## OMX 실무 적용 체크리스트

OMX 문서를 OMC 기준으로 다듬을 때는 아래 체크리스트를 권장합니다.

1. 이 페이지가 너무 카탈로그형인가?
2. 제목 아래 첫 문단이 "왜 읽어야 하는가"를 설명하는가?
3. 예시 / 표 / 단계형 흐름이 너무 늦게 나오지 않는가?
4. 한국어라면 더 읽히는 안내형 문장으로 바꿀 수 있는가?
5. 일본어/중국어라면 더 압축적이고 정돈된 구조가 맞는가?
6. 영어 product label을 유지하는 게 독자 경험에 도움이 되는가?
7. OMC 전용 사실을 실수로 가져오지는 않았는가?

## 권장 작업 순서

1. OMC 같은 로케일 페이지 읽기
2. 톤 / 구조 / 온보딩 방식 파악
3. 한국어 먼저 수정 및 승인
4. 승인된 구조를 영어에 반영
5. 일본어/중국어는 각 언어별 톤에 맞게 미세 조정
