---
title: Ultrawork Loop
description: The evidence-first work discipline behind LazyCodex.
---

ultrawork は LazyCodex が長い作業を扱うときの規律です。prompt に `ultrawork` または `ulw` が含まれるか、`$ulw-loop` を実行すると、agent は「終わったように見える」ではなく実行した証拠を基準に完了を判断します。

> _"計画し、実行し、検証し、証拠を一緒に残す。"_

原則は単純です。agent が終わったと言ったから作業が終わるわけではありません。**観測可能な証拠が検証**されたときに初めて終わります。

## What Ultrawork Enforces

ultrawork が有効になると、harness は希望的な「終わったようだ」では満足しません。

- 厳格な TDD: RED -> GREEN -> SURFACE -> CLEAN
- 作業リスクに合った現実的な QA scenarios
- 実際の manual QA channels: HTTP calls、tmux、browser、computer use、CLI stdout、data diff
- 作業が本当に終わるまで回る拘束力のある verification gates

## Relationship to `$ulw-loop`

`$ulw-loop` は ultrawork の規律を Commands として使う方法です。最新の流れでは `.omo/ulw-loop` の下に request、goals、success criteria、evidence ledger を残します。

| File | Role |
| --- | --- |
| `.omo/ulw-loop/brief.md` | 元の依頼と継続制約 |
| `.omo/ulw-loop/goals.json` | 目標と成功基準 |
| `.omo/ulw-loop/ledger.jsonl` | pass、fail、block、steering、checkpoint 記録 |

作業が終わったという言葉だけでは足りません。各 success criterion には実際の表面で確認した evidence が必要で、その evidence が通過して初めて loop が止まります。

正確な syntax と flags は [`$ulw-loop` command doc](/ja/docs/commands/ulw-loop) にだけ置きます。

## Failure Limits

loop は永遠に回らないよう制限されています。最新の `$ulw-loop` workflow は以下の上限を使います。

| Condition | Limit |
| --- | --- |
| 1 つの goal で全体通過なしに繰り返す | 5 cycles |
| 同じ criterion の同じ失敗 | 3 回 |

## Evidence Over Hope

loop は「少しコードを書いた」で止まりません。agent が期待する status report ではなく、どの検査が走り、何を示したかという evidence で結果が確認されたときに止まります。

## Position Among Commands

`$ulw-loop` は Commands の 1 つで、それぞれ異なる形の作業のためにあります。

一般的な流れは単純です。`$ulw-plan` が decision-complete plan を作り、`$start-work` が checkbox 単位で実行し、`$ulw-loop` が開いた作業を verifier が承認するまで続けます。各 command の詳細 syntax は [Commands](/ja/docs/commands) にまとめています。
