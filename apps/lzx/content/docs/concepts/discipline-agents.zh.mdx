LazyCodex(LZX)不是一个庞大的提示词，而是在 Codex 中安装一小批 **纪律智能体**，每个只负责一件事。Sisyphus 读取你的意图并分派，专家在各自的车道工作，循环持续推进，直到结果由证据验证，而非乐观的状态汇报。

这些智能体来自 **OmO(oh-my-openagent)** 框架。LazyCodex 是把该框架打包给 Codex 的薄分发层 —— "Codex 版 LazyVim"。

> **角色细节的来源。** LazyCodex README 只*列出*这些智能体的名字 —— "Sisyphus 编排 Hephaestus、Oracle、Librarian"。下列各角色的行为与工具边界由底层的 **OmO** 引擎定义，而非 LazyCodex README。权威来源请参阅 OmO 智能体文档。

## 角色一览

| 智能体 | 角色 | 边界 |
|---|---|---|
| **Sisyphus** | 编排者 | 读取意图、拆分工作、分派给专家、跟踪 todo，仅在任务完成时停止。 |
| **Hephaestus** | 实现 | 编写代码、调试、推理硬逻辑与综合的深度工作者。 |
| **Oracle** | 验证与评审 | 架构与调试的只读顾问。验证完成度；不写入、不编辑、不分派。 |
| **Librarian** | 检索 | 仅研究。拉取文档、远程仓库与 OSS 代码。不写入、不编辑、不分派。 |

Oracle 与 Librarian 的只读边界是框架的一部分，防止"聪明"的智能体在错误角色里做危险操作 —— 检索与评审绝不会悄悄改动你的代码库。

## 如何协作

Sisyphus 是入口编排者：读取意图，把请求拆成单元，交给合适的专家。

- **Hephaestus** 承担实现与硬推理单元。
- **Oracle** 验证结果，并以只读方式就架构和调试提供咨询。
- **Librarian** 检索工作者所需的上下文(文档、仓库、示例)。

Oracle 与 Librarian 不进行分派 —— 它们的工具访问按设计为只读。由于分派是显式的，多条车道可同时运行。

## 并行执行

当主会话无需结果即可继续时，LazyCodex 会并行运行专家。独立研究与自包含单元同时进行；当结果会阻塞下一个实现决策时则保持同步。每个单元都搭配适合其工作的模型 —— 见[多模型路由](/zh/docs/concepts/model-routing)。

## 它的位置

- 命令支柱驱动这些智能体：[`$ulw-plan`](/zh/docs/commands/ulw-plan) 做规划，[`$start-work`](/zh/docs/commands/start-work) 执行计划，[`$ulw-loop`](/zh/docs/commands/ulw-loop) 循环至 Oracle 验证完成。
- 让循环停下的，是经过验证的完成 —— 见 [ultrawork 循环](/zh/docs/concepts/ultrawork)。
