LazyCodex (LZX) 将 OmO agent harness 打包给 Codex 使用。用户通常只需要记住 `$init-deep`、`$ulw-plan`、`$start-work`、`$ulw-loop` 这些命令。内部则通过 Codex multi-agent 工具和可选角色，把探索、规划、评审、QA 分开。

以 `4.12.1` 为准，安装的角色包括 `explorer`、`librarian`、`plan`、`momus`、`metis`、`lazycodex-executor`、`lazycodex-code-reviewer`、`lazycodex-qa-executor`、`lazycodex-gate-reviewer`、`lazycodex-clone-fidelity-reviewer`。如果 Codex build 暴露 `agent_type`，就可以直接指定这些角色；否则会在消息里描述角色作为 fallback。

## Parent Session Ownership

即使角色很多，完成判断也不会整体交给子智能体。父 Codex 会话始终拥有目标、约束和最终判断。子智能体用于读地形、找缺口、评审或辅助 QA。

| 角色 | 主要用途 |
| --- | --- |
| `explorer` | 代码库结构、调用流、测试位置等内部上下文探索 |
| `librarian` | 外部文档、库契约、最新 API 调研 |
| `plan` | 计划草案和工作拆分 |
| `momus` / `metis` | 缺失决策、边界情况、风险评审 |
| `lazycodex-executor` | 执行计划中的特定工作单元 |
| `lazycodex-code-reviewer` | 实现后的代码质量评审 |
| `lazycodex-qa-executor` | 面向真实表面的 QA |
| `lazycodex-gate-reviewer` | 完成前验证门 |
| `lazycodex-clone-fidelity-reviewer` | clone 或同步任务的 fidelity review |

## Execution Loop

无论角色如何拆分，一个工作单元的基本循环相同。

1. **Explore**：绘制地形。用工具读代码，不靠猜测；有需要时并行启动只读子智能体。
2. **Plan**：选择路径。把要改的文件、具体变更和依赖记录到活动计划中。
3. **Implement**：精确实现。即使新项目会写成另一种样子，也要贴合现有代码库风格。
4. **Verify**：证明可用。尽可能并行运行诊断、相关测试和构建检查。
5. **Manual QA**：检查真实表面，然后写最终消息。

## Non-goals

- **不相信子智能体的自我报告。** 完成判断依赖独立验证。
- **用户要代码时，不只停在建议。** 除非用户明确要计划或头脑风暴，否则会实现。
- **不猜没读过的代码。** 探索便宜，假设昂贵。
- **不在 turn 末留下模糊工作。** 每个计划步骤都会整理为完成、有理由的阻塞，或有理由的移除。

## Orchestration and Delegation

LazyCodex 使用 Codex 内的子智能体角色。并行探索可使用只读角色，评审或 QA 可使用专门的 reviewer 角色。但最终报告仍以父会话收集的证据为准。

## Boulder State

`$start-work` 使用 `.omo/boulder.json` 保存进度，并通过 Stop-hook continuation 继续计划执行。这是用户能看到的核心持久行为。所有复选框完成后会输出 **ORCHESTRATION COMPLETE**。
