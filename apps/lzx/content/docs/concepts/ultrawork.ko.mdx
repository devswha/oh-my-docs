ultrawork는 대표 모드입니다. 프롬프트 어디에든 `ultrawork`(또는 짧은 별칭 `ulw`)를 입력하면, 최대 정밀도·결과 우선·증거 기반 오케스트레이션이 활성화됩니다.

> _"계획하고, 실행하고, 검증하고, 증거를 함께 남긴다."_

원칙은 단순합니다. 에이전트가 끝났다고 말한다고 해서 작업이 끝난 것이 아닙니다. **Oracle이 증거에 비추어 검증**할 때 비로소 끝난 것입니다.

## ultrawork가 강제하는 것

ultrawork가 활성화되면, 하니스는 희망 섞인 "끝난 것 같다"로 만족하지 않습니다.

- 엄격한 TDD: RED → GREEN → SURFACE → CLEAN
- 최소 3개의 현실적인 QA 시나리오
- 실제 수동 QA 채널(HTTP 호출, tmux, 브라우저)
- 작업이 진짜로 끝날 때까지 도는 구속력 있는 검증 게이트

## $ulw-loop: 검증된 완료

`$ulw-loop`는 검증된 완료까지 도는 자기 참조적 개발 루프입니다. 에이전트는 계속 작업하다 완료되었다고 판단하면 `<promise>DONE</promise>`을 내보냅니다. 하지만 그것으로 루프가 끝나지는 **않습니다**.

먼저 Oracle이 결과를 검증해야 합니다. 시스템이 Oracle의 검증을 확인한 뒤에야 루프가 끝납니다. 검증이 실패하면 다음 메시지와 함께 계속됩니다.

```text
Oracle verification failed. Continuing ULTRAWORK loop.
```

### 구문

```bash
$ulw-loop "task description" [--completion-promise=TEXT] [--strategy=reset|continue]
```

| 플래그 | 효과 |
| --- | --- |
| `--completion-promise=TEXT` | 검증 전 에이전트가 충족해야 할 약속 텍스트 지정 |
| `--strategy=reset` | 각 반복을 깨끗한 상태에서 시작 |
| `--strategy=continue` | 반복 간 상태를 이어감 |

### 반복 한도

루프는 영원히 돌지 않도록 제한됩니다.

- ultrawork 모드에서 **500**회
- 일반 모드에서 **100**회

## 희망 섞인 상태가 아니라 증거

루프는 "코드를 좀 썼다"에서 멈추지 않습니다. 에이전트가 기대하는 상태 보고가 아니라 증거 — 어떤 검사가 돌았고 무엇을 보여줬는지 — 로 결과가 확인될 때 멈춥니다. 바로 그래서 `<promise>DONE</promise>`과 루프 종료 사이에 Oracle이 자리합니다.

## 기둥들 사이에서의 위치

`$ulw-loop`는 세 개의 커맨드 기둥 중 하나이며, 각각 다른 형태의 작업을 위한 것입니다.

| 기둥 | 사용 시점 |
| --- | --- |
| `$ulw-plan` | 코드를 쓰기 전에 의사결정이 필요할 때. `plans/<slug>.md`를 쓰고 제품 코드는 절대 쓰지 않음. |
| `$start-work` | 계획이 준비되어 완료까지 실행해야 할 때. **ORCHESTRATION COMPLETE**를 출력. |
| `$ulw-loop` | 결과가 증거로 검증될 때까지 작업을 계속 진행해야 할 때. |

일반적인 흐름: `$ulw-plan`이 의사결정 완료 계획을 만들고, `$start-work`가 체크박스 단위로 실행하며, `$ulw-loop`가 열린 작업을 Oracle이 승인할 때까지 계속 돌립니다.
