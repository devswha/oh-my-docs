LazyCodex(LZX) 실행은 모든 하위 작업에 최고 모델을 쓰지 않습니다. 그 바탕이 되는 OmO 하니스는 **작업 카테고리**와 폴백 체인을 정의해, 각 작업 단위가 가장 적합한 GPT 모델에 도달하도록 합니다. 이는 무작위 모델 교체가 아니라 벤치마크 기반 라우팅입니다.

## 서로 다른 GPT 모델이 나타나는 이유

실행 중에 `gpt-5.2` + `xhigh`, `gpt-5.4-mini`, `gpt-5.3-codex`, 또는 `gpt-5.5` + `xhigh` 같은 더 새로운 등가 모델이 보여도 놀라지 마세요. 의도된 동작입니다.

하니스는 작업에 맞는 모델을 고릅니다:

| 카테고리 | 라우팅 대상 | 용도 |
|---|---|---|
| `quick` | `gpt-5.4-mini` | 작은 수정 |
| `ultrabrain` | 고추론 GPT 모델 | 어려운 로직 |
| agentic coding | Codex 튜닝 GPT 모델(예: `gpt-5.3-codex`) | 소프트웨어 엔지니어링 경로(가능할 때) |

## 쿼터 규율

핵심은 쿼터 규율입니다: 깊은 추론이 필요한 작업에는 가장 강한 모델을, 그 정도면 충분한 작업에는 더 저렴하고 빠른 모델을 쓰며, 루틴 단계에 프리미엄 쿼터를 태우는 대신 병렬 에이전트 작업을 효율적으로 유지합니다.

이는 [병렬 실행](/ko/docs/concepts/discipline-agents)과 짝을 이룹니다: 여러 레인이 각자 알맞은 비용으로 동시에 돕니다.

## 벤치마크 기반, 모델별로

라우팅은 OpenAI가 각 모델에 대해 문서화한 내용을 반영합니다:

- **GPT-5.2**는 코드 리뷰, 버그 탐색, 복잡한 도구 사용에 더 강하다고 OpenAI가 문서화했으며, 발표에서 최대 API 추론 강도가 `xhigh`를 쓴다고 밝혔습니다.
- **GPT-5.3-Codex**는 에이전트형 소프트웨어 엔지니어링을 위한 OpenAI의 Codex 튜닝 모델로, SWE-Bench Pro, Terminal-Bench 2.0, OSWorld Verified 같은 공개 코딩 에이전트 벤치마크가 발표에 보고되어 있습니다.
- **GPT-5.4 mini**는 효율적인 일상 코딩, 컴퓨터 사용, 서브에이전트에 적합하게 자리매김되어 있어 — 가벼운 작업이 프런티어 추론 모델을 쓰는 대신 여기에 떨어질 수 있는 이유입니다.
