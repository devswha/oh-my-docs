---
title: Agent Roles
description: Selectable Codex roles and the verification-first work loop.
---

LazyCodex (LZX) は OmO の agent harness を Codex で使えるようにパッケージします。ユーザーは通常 `$init-deep`、`$ulw-plan`、`$start-work`、`$ulw-loop` のような Commands だけ覚えれば十分です。内部では Codex の multi-agent tools と選択可能な roles を使い、探索、計画、レビュー、QA を分けます。

`4.12.1` 時点でインストールされる roles には `explorer`、`librarian`、`plan`、`momus`、`metis`、`lazycodex-executor`、`lazycodex-code-reviewer`、`lazycodex-qa-executor`、`lazycodex-gate-reviewer`、`lazycodex-clone-fidelity-reviewer` が含まれます。Codex build が `agent_type` を公開していれば直接指定し、公開していなければ message 内に role を説明する形へ fallback します。

## Parent Session Ownership

roles が増えても、完了判断を子エージェントへ丸ごと渡すわけではありません。親 Codex session は goal、constraints、final judgment を持ち続けます。子エージェントは地形を読む、ギャップを探す、レビューや QA を助けるために使います。

| Role | 主な用途 |
| --- | --- |
| `explorer` | コードベース構造、呼び出しフロー、テスト位置などの内部コンテキスト探索 |
| `librarian` | 外部ドキュメント、ライブラリ契約、最新 API 調査 |
| `plan` | 計画草案と作業分解 |
| `momus` / `metis` | 欠けた判断、エッジケース、リスクのレビュー |
| `lazycodex-executor` | 計画内の特定作業単位の実行 |
| `lazycodex-code-reviewer` | 実装後のコード品質レビュー |
| `lazycodex-qa-executor` | 実際の表面に対する QA |
| `lazycodex-gate-reviewer` | 完了前の検証ゲート |
| `lazycodex-clone-fidelity-reviewer` | clone・sync 作業の fidelity review |

## Execution Loop

roles がどう分かれていても、作業単位の基本ループは同じです。

1. **Explore**: 地形を把握します。推測せずツールでコードを読み、必要なら読み取り専用の子エージェントを並列に起動します。
2. **Plan**: 経路を決めます。変更ファイル、具体的な変更、依存関係を active plan に記録します。
3. **Implement**: 精密に作ります。新規なら別の形にしたくなる場合でも、既存コードベースのスタイルに合わせます。
4. **Verify**: 動作を証明します。診断、関連テスト、ビルド確認を可能な限り並列で実行します。
5. **Manual QA**: 実際の表面で結果を確認し、最終メッセージを書きます。

## Non-goals

- **子エージェントの自己報告を信じません。** 完了判断は独立検証で行います。
- **コードを求められたのに提案だけで終わりません。** 明示的に計画やブレストを求められた場合を除き、実装します。
- **読んでいないコードを推測しません。** 探索は安く、仮定は高くつきます。
- **ターン末に作業を曖昧に残しません。** 各計画ステップは完了、理由付きブロック、理由付き削除のいずれかに整理されます。

## Orchestration and Delegation

LazyCodex は Codex 内の子エージェント role を使います。並列探索には読み取り専用 role を、レビューや QA には専用 review role を使えます。ただし最終報告は親 session が集めた証拠を基準にまとめます。

## Boulder State

`$start-work` は `.omo/boulder.json` で進捗状態を残し、Stop-hook continuation で計画実行を続けます。これはユーザーに見える中核動作です。すべてのチェックボックスが終わると **ORCHESTRATION COMPLETE** を出力します。
