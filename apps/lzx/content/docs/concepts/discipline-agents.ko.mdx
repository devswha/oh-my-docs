title: Agent Roles
description: Selectable Codex roles and the verification-first work loop.
---

LazyCodex(LZX)는 OmO의 에이전트 하니스를 Codex에서 쓸 수 있게 패키징합니다. 사용자는 보통 `$init-deep`, `$ulw-plan`, `$start-work`, `$ulw-loop` 같은 커맨드만 기억하면 됩니다. 내부적으로는 Codex의 multi-agent 도구와 선택 가능한 역할을 이용해 탐색, 계획, 리뷰, QA를 나눕니다.

`4.12.1` 기준 설치되는 역할에는 `explorer`, `librarian`, `plan`, `momus`, `metis`, `lazycodex-executor`, `lazycodex-code-reviewer`, `lazycodex-qa-executor`, `lazycodex-gate-reviewer`, `lazycodex-clone-fidelity-reviewer`가 포함됩니다. Codex 빌드가 `agent_type`을 노출하면 해당 역할을 직접 지정하고, 노출하지 않으면 메시지 안에 역할을 설명하는 방식으로 fallback합니다.

## Parent Session Ownership

중요한 점은 역할이 많아져도 완료 판단을 하위 에이전트에게 통째로 넘기지 않는다는 것입니다. 부모 Codex 세션은 목표, 제약, 최종 판단을 계속 소유합니다. 하위 에이전트는 지형을 읽거나, 갭을 찾거나, 리뷰를 돕는 방식으로 쓰입니다.

| 역할 | 주로 쓰이는 곳 |
| --- | --- |
| `explorer` | 코드베이스 구조, 호출 흐름, 테스트 위치 같은 내부 맥락 탐색 |
| `librarian` | 외부 문서, 라이브러리 계약, 최신 API 조사 |
| `plan` | 계획 초안과 작업 분해 |
| `momus` / `metis` | 빠진 결정, 엣지 케이스, 위험 검토 |
| `lazycodex-executor` | 계획의 특정 작업 단위 실행 |
| `lazycodex-code-reviewer` | 구현 후 코드 품질 리뷰 |
| `lazycodex-qa-executor` | 실제 실행 기반 QA |
| `lazycodex-gate-reviewer` | 완료 전 검증 게이트 |
| `lazycodex-clone-fidelity-reviewer` | 복제·동기화 작업의 충실도 검토 |

## Execution Loop

역할이 어떻게 나뉘든 작업 단위의 기본 루프는 같습니다.

1. **Explore** — 지형을 파악합니다. 추측하지 않고 도구로 코드를 읽으며, 필요하면 읽기 전용 하위 에이전트를 병렬로 띄웁니다.
2. **Plan** — 경로를 정합니다. 수정할 파일, 구체적 변경, 의존성을 활성 계획에 기록합니다.
3. **Implement** — 정밀하게 만듭니다. 새 프로젝트라면 다르게 짰을 코드도 기존 코드베이스 스타일에 맞춰 수술적으로 고칩니다.
4. **Verify** — 작동을 증명합니다. 변경 파일 진단, 관련 테스트, 빌드 확인을 가능한 한 병렬로 실행합니다.
5. **Manual QA** — 실제 표면에서 결과를 확인한 뒤 최종 메시지를 작성합니다.

## Non-goals

- **하위 에이전트의 자체 보고를 믿지 않습니다.** 완료 판단은 독립 검증으로 합니다.
- **코드를 요청받았는데 제안만 하고 끝내지 않습니다.** 명시적으로 계획이나 브레인스토밍을 원한 경우가 아니라면 구현합니다.
- **읽지 않은 코드를 추측하지 않습니다.** 탐색은 싸고, 가정은 비쌉니다.
- **턴 끝에 일을 흐리게 남기지 않습니다.** 모든 계획 단계는 완료, 사유 있는 차단, 사유 있는 제거 중 하나로 정리됩니다.

## Orchestration and Delegation

LazyCodex는 Codex 안의 하위 에이전트 역할을 사용합니다. 병렬 탐색에는 읽기 전용 역할을 띄울 수 있고, 리뷰나 QA에는 별도 리뷰 역할을 붙일 수 있습니다. 하지만 최종 보고는 부모 세션이 수집한 증거를 기준으로 정리합니다.

## Boulder State

`$start-work`는 `.omo/boulder.json`을 사용해 진행 상태를 남기고, Stop-hook continuation으로 계획 실행을 이어 갑니다. 이 부분은 사용자에게 보이는 핵심 동작입니다. 체크박스가 모두 끝나면 **ORCHESTRATION COMPLETE**를 출력합니다.
