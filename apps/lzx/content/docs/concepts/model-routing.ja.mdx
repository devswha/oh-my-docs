---
title: マルチモデルルーティング
description: タスクカテゴリごとの自動モデル選択 —— クォータ規律を守り、ベンチマーク駆動で、ランダムな入れ替えではありません。
---

LazyCodex(LZX)の実行は、すべてのサブタスクに最良のモデルを使うわけではありません。基盤の OmO ハーネスは **タスクカテゴリ** とフォールバックチェーンを定義し、各作業ユニットを最も適切な GPT モデルに到達させます。これはランダムなモデル入れ替えではなく、ベンチマーク駆動のルーティングです。

## 異なる GPT モデルが現れる理由

実行中に `gpt-5.2` + `xhigh`、`gpt-5.4-mini`、`gpt-5.3-codex`、あるいは `gpt-5.5` + `xhigh` のような新しい同等モデルが見えても驚かないでください。これは意図的です。

ハーネスは仕事に合うモデルを選びます:

| カテゴリ | ルーティング先 | 用途 |
|---|---|---|
| `quick` | `gpt-5.4-mini` | 小さな編集 |
| `ultrabrain` | 高推論の GPT モデル | 難しいロジック |
| agentic coding | Codex 調整済み GPT モデル(例: `gpt-5.3-codex`) | ソフトウェアエンジニアリング経路(利用可能なとき) |

## クォータ規律

要点はクォータ規律です: 深い推論が必要なタスクには最強のモデルを、それで十分なタスクには安く速いモデルを使い、ルーチンの手順にプレミアムクォータを燃やす代わりに、並列のエージェント作業を効率的に保ちます。

これは[並列実行](/ja/docs/concepts/discipline-agents)と組み合わさります: 多くのレーンが、それぞれ適切なコストで同時に走ります。

## ベンチマーク駆動、モデルごとに

このルーティングは、OpenAI が各モデルについて文書化している内容を反映します:

- **GPT-5.2** は、コードレビュー、バグ発見、複雑なツール使用に強いと OpenAI が文書化しており、その発表は最大 API 推論強度が `xhigh` を使うと述べています。
- **GPT-5.3-Codex** はエージェント的ソフトウェアエンジニアリング向けの OpenAI の Codex 調整モデルで、SWE-Bench Pro、Terminal-Bench 2.0、OSWorld Verified などの公開コーディングエージェントベンチマークが発表で報告されています。
- **GPT-5.4 mini** は効率的な日常コーディング、コンピュータ操作、サブエージェント向けに位置づけられており —— だからこそ軽量なタスクが、フロンティア推論モデルを使わずにここへ落ちることがあります。
