title: Ultrawork Loop
description: The evidence-first work discipline behind LazyCodex.
---

ultrawork 是 LazyCodex 处理长任务时使用的工作纪律。当 prompt 中包含 `ultrawork` 或 `ulw`，或调用 `$ulw-loop` 时，智能体会根据已执行的证据判断完成，而不是根据“看起来完成了”。

> _"计划、执行、验证，并留下证据。"_

原则很简单：不是智能体说完成就完成。只有**可观察证据被验证**后，工作才算完成。

## What Ultrawork Enforces

ultrawork 激活后，harness 不接受乐观的“看起来完成了”。

- 严格 TDD：RED -> GREEN -> SURFACE -> CLEAN
- 与工作风险匹配的现实 QA 场景
- 真实手动 QA 通道，例如 HTTP 调用、tmux、浏览器、computer use、CLI stdout、数据 diff
- 持续运行到工作真正完成的强约束验证门

## Relationship to `$ulw-loop`

`$ulw-loop` 是把 ultrawork 纪律作为 Command 使用的方式。最新流程会在 `.omo/ulw-loop` 下记录请求、目标、成功标准和证据账本。

| 文件 | 作用 |
| --- | --- |
| `.omo/ulw-loop/brief.md` | 原始请求和持续约束 |
| `.omo/ulw-loop/goals.json` | 目标和成功标准 |
| `.omo/ulw-loop/ledger.jsonl` | pass、fail、block、steering、checkpoint 记录 |

仅仅说工作完成是不够的。每个成功标准都需要真实表面的证据，而且证据必须通过后循环才会停止。

准确语法和 flags 只放在 [`$ulw-loop` command 文档](/zh/docs/commands/ulw-loop)。

## Failure Limits

循环有上限，不会永远运行。最新 `$ulw-loop` 工作流使用：

| 条件 | 上限 |
| --- | --- |
| 一个 goal 没有完整通过时的重复 | 5 个 cycle |
| 同一 criterion 的同一种失败 | 3 次 |

## Evidence Over Hope

循环不会停在“写了一些代码”。只有用证据确认结果时才会停止：运行了哪些检查、显示了什么、为什么能证明结果。

## Position Among Commands

`$ulw-loop` 是 Commands 之一，每个 Command 面向不同形态的工作。

常见流程很简单：`$ulw-plan` 创建决策完备的计划，`$start-work` 按复选框执行，`$ulw-loop` 持续推进开放工作，直到验证者批准。每个命令的详细语法集中在 [Commands](/zh/docs/commands)。
