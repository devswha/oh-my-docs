ultrawork 是核心模式。在提示词的任意位置输入 `ultrawork`（或短别名 `ulw`），即可激活最高精度、结果优先、证据驱动的编排。

> _"规划、执行、验证，并把证据一并留存。"_

原则很简单：任务并非因为智能体声称完成就算完成，而是当 **Oracle 依据证据验证**通过时才算完成。

## ultrawork 强制什么

ultrawork 激活时，harness 不会满足于乐观的"看起来完成了"：

- 严格 TDD：RED → GREEN → SURFACE → CLEAN
- 至少 3 个真实的 QA 场景
- 真实的手动 QA 通道（HTTP 调用、tmux、浏览器）
- 一个有约束力的验证门，循环直到工作真正完成

## $ulw-loop：已验证的完成

`$ulw-loop` 是一个自引用开发循环，运行直到验证通过的完成。智能体持续工作，认为完成时会发出 `<promise>DONE</promise>`——但这**不会**结束循环。

必须先由 Oracle 验证结果。只有在系统确认 Oracle 已验证后，循环才结束。验证失败时，会带着以下消息继续：

```text
Oracle verification failed. Continuing ULTRAWORK loop.
```

### 语法

```bash
$ulw-loop "task description" [--completion-promise=TEXT] [--strategy=reset|continue]
```

| 标志 | 作用 |
| --- | --- |
| `--completion-promise=TEXT` | 设置验证前智能体必须满足的承诺文本 |
| `--strategy=reset` | 每次迭代从干净状态开始 |
| `--strategy=continue` | 在迭代间延续状态 |

### 迭代上限

循环有界，不会无限运行：

- ultrawork 模式 **500** 次
- 普通模式 **100** 次

## 以证据取代乐观状态

循环不止于"写了点代码"，而是在结果由证据——运行过的检查及其结果——确认时停止，而非智能体乐观的状态更新。这正是 `<promise>DONE</promise>` 与循环结束之间要有 Oracle 把关的原因。

## 在支柱中的位置

`$ulw-loop` 是三大命令支柱之一，各自面向不同形态的工作：

| 支柱 | 何时使用 |
| --- | --- |
| `$ulw-plan` | 写代码前需要决策时。写入 `plans/<slug>.md`，绝不写产品代码。 |
| `$start-work` | 计划就绪、需执行到完成时。打印 **ORCHESTRATION COMPLETE**。 |
| `$ulw-loop` | 任务应持续推进直到结果由证据验证时。 |

常见流程：`$ulw-plan` 产出决策完备的计划，`$start-work` 逐个复选框执行，`$ulw-loop` 让开放式任务持续运行直到 Oracle 签字认可。
