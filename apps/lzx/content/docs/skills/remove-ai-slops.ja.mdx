---
title: remove-ai-slops
description: 振る舞いを保ったまま AI 由来のコードの臭いを整理。
---

`remove-ai-slops` は、ブランチの変更または明示的なファイル一覧から、AI 由来のコードの臭い（スロップ）を、振る舞いを厳密に保ちつつ取り除きます。核心となる安全不変条件は、1 行を削除する前にグリーンな回帰テストで振る舞いを固定することです。チェックリストは安全装置ではなく、通過するテストこそが安全装置です。

トリガーには "remove ai slops"、"clean ai code"、"deslop"、"cleanup AI generated" が含まれます。

## スコープ

- **既定**: `merge-base main` に対するブランチ diff —— 引数不要。
- **任意**: 呼び出し側が渡す明示的なファイル一覧（例: ループワークフローの変更ファイル集合）。

削除・バイナリ・生成・ベンダリングされたファイルは除外されます。

## スロップのカテゴリ

このスキルは 10 のカテゴリを種類ごとに分けて評価します。

| グループ | カテゴリ |
| --- | --- |
| スタイル | 自明なコメント、過剰な防御コード、過度な複雑さ |
| 構造 | 不要な抽象、境界違反、デッドコード |
| 隠れたコスト | 重複、性能等価の最適化 |
| 振る舞いの網羅 | 欠落したテスト |
| 構造（サイズ） | 過大モジュール（純 250 LOC 以上） |

各カテゴリには明示的な KEEP ルールがあります。WHY を説明するコメント、システム境界での検証、実際の継ぎ目を与える抽象、意図的なパターンは保持します。性能修正は振る舞い等価が明白な場合のみ適用し、そうでなければ SKIP します。過大モジュールは単にフラグを立てるのではなく、完全なモジュール分割リファクタリングを引き起こし、各新ファイルは責務とする概念で命名します（`utils`、`helpers`、`common`、`part_1` は不可）。

## プロセス

<Steps>

<Step>

### スコープの決定

ファイルパスが渡されればそれがスコープです。なければブランチ diff を使います。

```bash
git diff $(git merge-base main HEAD)..HEAD --name-only
```

</Step>

<Step>

### 回帰テストで振る舞いを固定

スコープ内の各ファイルについて観測可能な振る舞いを特定し、既存の網羅を探し、編集前に現在の振る舞いを固定する最狭の回帰テストを書きます。ベースラインはグリーンでなければなりません。グリーンなベースラインを確立できなければ、停止して報告します。

</Step>

<Step>

### 計画してから並列で整理

最も安全な順から危険な順（コメント → デッドコード → 防御 → 重複 → 複雑さ → 抽象/境界 → 性能 → テスト → 過大モジュール）でファイルごとの整理計画を明示します。次に `ai-slop-remover` スキルを読み込んだ `deep` エージェントでファイルを処理し、5 件ずつ並列でバッチ実行します。失敗したファイルは 1 度だけ再試行し、それでも駄目ならエスカレートします。

</Step>

<Step>

### 品質ゲートで検証

回帰テスト、lint、型チェック（LSP 診断とプロジェクトの型チェッカー）、単体/結合テスト、静的/セキュリティスキャンを実行します。本当に N/A なゲートは `N/A` と報告し、黙ってスキップしません。その後、安全/振る舞い/品質のクリティカルレビューチェックリストを通します。

</Step>

<Step>

### 課題の修正

いずれかのゲートが失敗したら、問題の変更を `git checkout` か的を絞った編集で戻し、安全と証明できる分だけ再適用し、ゲートを再実行します。同じファイルで 3 回失敗したら停止しエスカレートします。

</Step>

</Steps>

## アンチパターン

- 回帰テスト段階の省略 —— 網羅のない地盤でのコード削除は振る舞い変更の時限爆弾。
- 無関係なリファクタリングを 1 つの整理コミットに束ねる。
- アルゴリズム変更を性能最適化に偽装する。
- 黙ったスキップ —— `N/A` とその理由を述べる。
- WHY を説明するコメントの削除。
- スコープ外のファイルに触れる。
