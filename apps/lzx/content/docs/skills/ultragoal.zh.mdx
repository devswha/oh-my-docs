---

`ultragoal` 技能是 LazyCodex（LZX）充当目标编排智能体，用于跨轮次与跨会话存续的多目标工作。每个目标携带自己的成功标准，每个标准由真正运行过的真实使用场景的可观测证据来证明，每次 pass、fail、block、转向变更与检查点都记入仅追加的台账审计。它随 LazyCodex 安装到 Codex 的 OmO（oh-my-openagent）插件一起提供。

绿色的测试套件只是支撑证据，绝非完成证明。仅靠测试永远不能证明完成。

## 持久化状态

状态驻留在仓库中，而非模型脑中。该技能在恢复、转向或检查点之前读取这些工件，且绝不在它们或 `omo ultragoal status --json` 之外臆造状态。

| 工件 | 用途 |
|---|---|
| `.omo/ultragoal/brief.md` | 原始简报与持久约束。 |
| `.omo/ultragoal/goals.json` | 目标列表，每个内嵌 `successCriteria`。 |
| `.omo/ultragoal/ledger.jsonl` | 每个结果与检查点的仅追加审计轨迹。 |

所有写入都走 `omo ultragoal` CLI 路径——状态文件绝不手工编辑。

## 手动 QA 通道

每个标准恰好通过一个通道执行，在记录 PASS 之前由智能体亲自运行。`--dry-run`、打印命令、"应当响应"、"看起来对" 一律不算数。

| 通道 | 方式 |
|---|---|
| HTTP 调用 | 用 `curl -i`（或 Playwright APIRequestContext）打实时端点；捕获状态行 + 头 + 正文。 |
| tmux | `tmux new-session`，用 `send-keys` 驱动，用 `capture-pane` 转储；记录即工件。 |
| 浏览器使用 | 用 Playwright / Puppeteer / Chromium 驱动真实页面；捕获操作日志 + 截图。 |
| 计算机使用 | 对运行中应用做 OS 级 GUI 自动化；捕获操作日志 + 截图。 |

纯 CLI stdout、数据库状态差异或解析后的配置转储可满足 CLI 或数据形态的标准，但绝不替代面向用户行为的通道场景。

## 运行流程

<Steps>

<Step>

### 引导

用 `omo ultragoal create-goals`（`--brief`、`--brief-file` 或 `--from-stdin`）从简报创建目标。打磨每个目标，使其携带覆盖正常路径、边界、回归与对抗风险的 3 个以上 `successCriteria`——每个带 `id`、`scenario`、`expectedEvidence`、对抗类别、停止条件，以及命名的手动 QA 通道。然后用 `omo ultragoal status --json` 检查状态。

</Step>

<Step>

### 执行循环

按目标循环，每目标上限 5 个周期、同一标准相同失败上限 3 次。用 `omo ultragoal complete-goals --json` 获取下一目标。对每个标准：依据场景与既往台账条目规划、登记原子待办、执行一次有界改动，然后真正运行命名的通道场景并捕获可观测工件。

</Step>

<Step>

### 先清理再记录

记录前先拆除场景产生的每个运行时工件——服务器 PID、tmux 会话、浏览器上下文、容器、占用端口、临时文件。在证据串中嵌入一行清理回执。用 `omo ultragoal record-evidence --status pass|fail|blocked` 恰好记录一个结果。缺少清理回执即为 BLOCKED，而非 PASS。

</Step>

<Step>

### 目标完成

用 `omo ultragoal criteria` 确认每个标准均为 `pass`，再用 `omo ultragoal checkpoint --status complete` 携带证据摘要与最新目标快照。被阻断或失败的目标用 `--status blocked` / `--status failed` 加诊断证据检查点。

</Step>

<Step>

### 最终质量闸门

在所有标准均通过的最后一个目标上：跑定向验证，对改动文件跑 `ai-slop-cleaner`，再次跑验证，然后运行 `$code-review`。`recommendation == "APPROVE"` 且 `architectStatus == "CLEAR"` 即为干净。干净的评审以 `--quality-gate-json` 检查点最终完成；不干净则改记录评审阻断项。

</Step>

</Steps>

## 动态转向

转向仅用于结构化、有证据支撑的变更——自然语言转向请求会被拒绝。每种类型有必填字段，并通过 `omo ultragoal steer --kind <kind> ... --evidence "<...>" --rationale "<...>" --json` 执行。类型包括 `add_subgoal`、`split_subgoal`、`reorder_pending`、`revise_pending_wording`、`revise_criterion`、`annotate_ledger`、`mark_blocked_superseded`。

## 停止规则

- 所有目标完成 + 所有标准 `pass` + 最终质量闸门干净：DONE。
- 同一标准失败 3 次，或一个目标 5 个周期未全通过：以失败检查点并呈现诊断。
- 安全边界（破坏性命令、密钥外泄、生产写入）：阻断并给出安全替代。
- 残留的 QA 状态（活进程、tmux 会话、浏览器上下文、占用端口、临时目录）：非 PASS——清理、附上回执，再继续。
- 用户发出 `/cancel`：干净释放进行中状态，且不自动恢复。
