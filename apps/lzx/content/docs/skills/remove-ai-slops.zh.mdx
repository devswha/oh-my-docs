---
title: remove-ai-slops
description: 在保留行为的前提下清理 AI 生成的代码异味。
---

`remove-ai-slops` 会从分支变更或明确给出的文件列表中清理 AI 生成的代码异味，同时严格保留行为。它最核心的安全不变式是：删除任何一行之前，先用通过的回归测试锁定当前行为。清单不是安全网；通过的测试才是。

触发词包括 "remove ai slops"、"clean ai code"、"deslop"、"cleanup AI generated"。

`remove-ai-slops` 是一个编排器。它确定范围，先用回归测试锁定行为，再把按类别划分的清理工作拆给并行 worker，最后用质量门验证结果。在 Codex 中，它不会直接调用 OpenCode 风格的 `deep` 示例，而是把它们翻译成带有自包含消息的 worker sub-agent。

## 范围

- **默认值**：相对 `merge-base main` 的分支 diff；不需要参数。
- **可选**：调用方传入的明确文件列表，例如 loop 工作流中的变更文件集。

删除文件、二进制文件、生成文件、vendor 文件会被过滤掉。

## Slop 类别

这个 Skill 会按组评估主要 slop 类别。

| 组 | 类别 |
| --- | --- |
| 风格 | 显而易见的注释、过度防御代码、过度复杂。 |
| 结构 | 不必要的抽象、边界违规、死代码。 |
| 隐性成本 | 重复、行为等价的性能清理。 |
| 行为覆盖 | 缺失测试。 |
| 结构，大小 | 超大模块，纯代码超过 250 LOC。 |

每个类别都有明确的 KEEP 规则。解释原因的注释、系统边界处的验证、提供真实边界的抽象、刻意保留的模式都要保留。性能修改只在行为等价明显时应用；否则跳过。超大模块不是简单提示，而会触发真正的模块化重构，新文件必须按其负责的概念命名。不要使用 `utils`、`helpers`、`common`、`part_1` 这样的名称。

## 流程

<Steps>

<Step>

### 确定范围

如果传入了文件路径，就以这些路径为范围。否则使用分支 diff。

```bash
git diff $(git merge-base main HEAD)..HEAD --name-only
```

</Step>

<Step>

### 用回归测试锁定行为

对范围内每个文件，识别可观察行为，寻找现有覆盖，并在编辑前写出能锁定当前行为的最窄回归测试。基线必须是 green。如果无法建立 green 基线，就停止并报告。

</Step>

<Step>

### 先计划，再并行清理

按从安全到高风险的顺序写出清理计划：注释、死代码、防御代码、重复、复杂度、抽象/边界、性能、测试、超大模块。然后以文件为单位批量运行 worker，每批最多五个。某个文件失败时重试一次，仍失败则升级处理。

</Step>

<Step>

### 用质量门验证

运行回归测试、lint、通过 LSP 诊断和项目类型检查器的 typecheck、单元/集成测试、静态/安全扫描。真正不适用的 gate 必须标成 `N/A`，不能悄悄跳过。之后再检查安全、行为、质量清单。

</Step>

<Step>

### 修复问题

如果 gate 失败，用定向编辑或 checkout 回退有问题的变更，只重新应用能证明安全的部分，并重新运行 gate。如果同一个文件失败三次，就停止并升级。

</Step>

</Steps>

## 反模式

- 跳过回归测试步骤。在没有覆盖的地方删代码，就是延迟发生的行为变更。
- 把无关重构混进一个清理提交。
- 把算法变化伪装成性能清理。
- 静默跳过。要写 `N/A` 和原因。
- 删除解释原因的注释。
- 修改约定范围之外的文件。
