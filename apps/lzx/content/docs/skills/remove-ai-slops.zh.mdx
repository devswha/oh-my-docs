`remove-ai-slops` 在严格保持行为不变的前提下，从分支改动或显式文件清单中移除 AI 生成的代码异味（slop）。核心安全不变式：在删除任何一行之前，先用通过的回归测试锁定行为。清单不是安全保障，通过的测试才是。

触发短语包括 "remove ai slops"、"clean ai code"、"deslop"、"cleanup AI generated"。

`remove-ai-slops` 是编排者：确定范围（分支 diff 或文件清单），先用回归测试锁定行为，再以 `deep` 智能体每批 5 个并行对各文件执行分类清理，最后用质量门验证。每个 `deep` 智能体都加载 `ai-slop-remover` 技能 —— 仅对单个文件移除 slop 的单文件 worker。多个文件通过每文件一个、并行调用该 worker 来处理。

## 范围

- **默认**：相对 `merge-base main` 的分支 diff —— 无需参数。
- **可选**：调用方传入的显式文件清单（例如循环工作流的改动文件集）。

被删除、二进制、生成与 vendored 文件会被过滤。

## slop 类别

该技能按种类分组评估十个类别。

| 分组 | 类别 |
| --- | --- |
| 风格 | 显而易见的注释；过度防御代码；过度复杂 |
| 结构 | 多余抽象；边界违规；死代码 |
| 隐性成本 | 重复；性能等价优化 |
| 行为覆盖 | 缺失测试 |
| 结构（体量） | 超大模块（纯 250 LOC 以上） |

每个类别都带有明确的 KEEP 规则。解释 WHY 的注释、系统边界的校验、提供真实接缝的抽象、有意为之的模式都会保留。性能修复仅在行为等价显而易见时才应用，否则 SKIP。超大模块触发完整的模块化重构，而非仅打标记，每个新文件按其所负责的概念命名（绝不用 `utils`、`helpers`、`common`、`part_1`）。

## 流程

<Steps>

<Step>

### 确定范围

若传入了文件路径，即为范围。否则使用分支 diff：

```bash
git diff $(git merge-base main HEAD)..HEAD --name-only
```

</Step>

<Step>

### 用回归测试锁定行为

对范围内每个文件，识别可观察行为，找出现有覆盖，并在编辑前写下锁定当前行为的最窄回归测试。基线必须为 green。若无法建立 green 基线，停止并报告。

</Step>

<Step>

### 先规划，再并行清理

按从最安全到最危险的顺序（注释 → 死代码 → 防御 → 重复 → 复杂度 → 抽象/边界 → 性能 → 测试 → 超大模块）给出每个文件的清理计划。然后用加载了 `ai-slop-remover` 技能的 `deep` 智能体处理文件，每批五个并行。若某文件失败，重试一次；否则上报。

</Step>

<Step>

### 用质量门验证

运行回归测试、lint、类型检查（LSP 诊断加项目类型检查器）、单元/集成测试以及任何静态/安全扫描。真正 N/A 的门报告为 `N/A`，绝不静默跳过。随后走一遍安全/行为/质量的批判性评审清单。

</Step>

<Step>

### 修复问题

若任一门失败，用 `git checkout` 或定向编辑回退问题改动，仅重新应用可证明安全的部分，再重跑该门。同一文件失败三次即停止并上报。

</Step>

</Steps>

## 反模式

- 跳过回归测试阶段 —— 在无覆盖的地基上删代码是行为变更的定时炸弹。
- 把无关的重构塞进一次清理提交。
- 把算法变更伪装成性能优化。
- 静默跳过 —— 要说 `N/A` 及原因。
- 移除解释 WHY 的注释。
- 触碰范围外的文件。
