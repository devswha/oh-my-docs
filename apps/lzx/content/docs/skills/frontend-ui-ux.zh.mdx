`frontend` 是 LazyCodex 在处理 Web UI、UX、样式、性能、可访问性和视觉 QA 时使用的 Skill。它不是单纯把组件堆到页面上，而是以现有设计系统和真实浏览器验证为基准来构建界面。

这个 Skill 更像一个路由器，而不是单一规则文件。它会根据任务性质，最小化读取 `design`、`perfection`、`ui-ux-db` 参考资料，并用这些标准完成实现与验证。

## 何时使用

- 前端实现、样式、布局、动画工作。
- 现有 UI 显得别扭或不一致，需要重新打磨。
- 需要在真实浏览器中检查 Lighthouse、Core Web Vitals、可访问性和 SEO。
- 需要在 React 项目中接入 `react-grab`、`react-scan`、`react-doctor` 等开发工具。
- 需要查找颜色、字体、图表、落地页结构或 UX 模式。

## 三个轴

| 轴 | 作用 |
| --- | --- |
| `design` | 设计系统、品牌语气、布局审美、基于浏览器的设计 QA。 |
| `perfection` | Lighthouse 100、性能、可访问性、SEO、Core Web Vitals。 |
| `ui-ux-db` | 风格、配色、字体组合、UX 指南搜索。 |

## 执行原则

- 不在没有检查设计系统的情况下创建新组件。
- 不为了 Lighthouse 分数削弱 UX。
- 图标使用 SVG 图标集，而不是 emoji。
- 动画优先使用 GPU-friendly 属性，例如 `transform`、`opacity`、`filter`。
- 在真实浏览器中确认之后，才算完成。

## 常见流程

<Steps>

<Step>

### Route

先判断请求属于 UI 实现、现有 UI 改进、性能审计，还是样式搜索。只读取必要的参考资料。

</Step>

<Step>

### Build

按照现有代码库和设计系统实现。需要新增视觉规则时，也应沉淀为 token 或可复用模式。

</Step>

<Step>

### Verify

在浏览器中检查 desktop/mobile 表面；如果任务需要，也一起检查性能与可访问性指标。

</Step>

</Steps>
