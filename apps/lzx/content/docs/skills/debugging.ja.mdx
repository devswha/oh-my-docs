`debugging` スキルは LazyCodex（LZX）が本物のランタイムデバッグを行うものです。クラッシュ、静かな失敗、誤った応答、止まったプロセス、メモリリーク、非同期の誤動作、説明のつかないタイミング、さらにはシンボルを剥がしたバイナリのリバースエンジニアリングまで扱います。仮説駆動のループを回し、観測した状態が根本原因を証明するまでバグを直したと宣言しません。LazyCodex が Codex に導入する OmO（oh-my-openagent）プラグインに同梱されます。

トリガーには "debug this"、"why is X not working"、"hanging"、"attach a debugger"、"reverse engineer"、"trace this bug"、"reproduce and fix"、"silent failure"、"HTTP 200 but empty"、"stuck process" などの語句と、`pwndbg`、`gdb`、`lldb`、`node inspect`、`tsx debug`、`pdb`、`dlv`、`delve`、`rust-gdb` といったツール名が含まれます。

## 毎回守る 2 つの規律

1. **ランタイムの真実はコード読解に勝る。** バグが起きる理由に関するすべての主張は観測した状態から来なければならず、コードを読んで作った筋の通った物語から来てはいけません。
2. **痕跡を残さない。** デバッグはアーティファクトを生みます。すべてのアーティファクトは作成前にジャーナルへ記録し、タスクを完了と宣言する前に除去します。

## HOW は references にある

このスキルは意図的に小さく作られています。実際のデバッグ知識は `references/` にあり、該当のリファレンスを読むことは推奨ではなくハードゲートです。あるリファレンス領域のコマンドを実行する前に、本セッションでそのリファレンスを読んでいる必要があります。

- **ランタイムリファレンス**はランタイム別に起動・アタッチ・ブレークポイント・検査の方法を扱います: Python、Node.js / tsx / Bun / Deno、Rust、Go、ネイティブバイナリ、バンドルアプリバイナリ（Bun/Node SEA、Deno compile、pkg、nexe、Electron、Tauri、PyInstaller）。各々が静かに数時間を浪費させる落とし穴を文書化します。例えば `tsx` + `node inspect` には行ブレークポイントが決して発火しない静かなソースマップ失敗があります。
- **専門ツール**はその領域での正しい道具です: ブラウザ提供の UI バグには Playwright CLI、`strings`/`objdump` で当てずっぽうする前に Ghidra のデコンパイラ、素の `gdb` の代わりに pwndbg、バイナリやネットワークとの再現可能なやり取りには pwntools。

ネイティブとバンドルを 30 秒で見分ける方法: `file ./target` は両方を ELF/Mach-O と呼ぶので、`du -h ./target`（50 MB 以上は疑わしい）と `strings -n 12 ./target | rg -iE 'bun|node_modules|webpack|esbuild|deno|pkg/lib|electron|pyinstaller|nexe|NODE_SEA_FUSE|tauri'` を併用します。ヒットがあればバンドル、きれいならネイティブです。

## フェーズループ

各フェーズにはちょうど 1 つのリファレンスがあり、記憶からではなくフェーズ入場時に読みます。

<Steps>

<Step>

### フェーズ 0-1 — セットアップとジャーナル

何かをアタッチする前に、ランタイム、ポート、シンボル、環境変数、ウォッチャーを把握します。すべてのアーティファクトを追跡する単一の `.debug-journal.md` を開き、確実なリバートを保証します。

</Step>

<Step>

### フェーズ 2-3 — 仮説と並列調査

直交する軸で最低 3 つの仮説を立て、それぞれに区別できる証拠を持たせます。並列に調査します。チームモードが有効なら `debug-squad` チームで、そうでなければ非同期サブエージェントで進めます。

</Step>

<Step>

### フェーズ 4 — Oracle Triple

連続 2 ラウンド失敗したら、直交するフレーミングで Oracle を 3 つ起動し、その答えを統合します。（抽出/監査タスク用の Verification Oracle とは別物です。）

</Step>

<Step>

### フェーズ 5-7 — エスカレーション、確定、修正

証拠が尽き、かつ判断に方針上の含意があるときだけユーザー判断へエスカレーションします。根本原因は、疑わしい原因をトグルするとバグもトグルするときにのみ確定します。その後、先に失敗するテストで固定します（red → 最小 green、スコープ拡張なし）。

</Step>

<Step>

### フェーズ 8-10 — QA、クリーンアップ、検証

システムを実際に使って QA します: CLI は tmux、ブラウザは Playwright、API は本物の `curl`、バイナリは本物の再現。ジャーナルをたどってすべてのアーティファクトをリバートし、`git diff` に修正とテストだけが残る状態にしてから、最後の証拠ゲート 4 つを通過します。

</Step>

</Steps>

## 安全不変条件

- ランタイム状態が唯一の真実の源です。観測値のない仮説は推測であり、推測は直しません。
- すべてのデバッグアーティファクトは作成前にジャーナルへ記録します。
- 先に失敗するテストなしに修正を出荷しません。red → green の遷移が必須です。
- 型チェックやコンパイルの成功は決して完了になりません。実際のユーザーシナリオだけが完了です。
- デバッグ中にエラーを静かに飲み込みません。システムが飲み込むなら、それ自体がバグであることが多いです。
- このスキル内で `git commit` は絶対にしません。コミットはユーザー確認後のフローの担当です。
