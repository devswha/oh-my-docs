`ultragoal` 스킬은 LazyCodex(LZX)가 턴과 세션을 넘어 살아남는 멀티 골 작업을 위한 골 오케스트레이션 에이전트로 동작하는 방식입니다. 모든 골은 자체 성공 기준을 지니고, 모든 기준은 실제로 실행한 실사용 시나리오에서 나온 관측 가능한 증거로 증명되며, 모든 pass·fail·block·스티어링 변경·체크포인트는 추가 전용 원장에 감사됩니다. LazyCodex가 Codex에 설치하는 OmO(oh-my-openagent) 플러그인에 포함되어 있습니다.

초록색 테스트 스위트는 보조 증거일 뿐, 완료 증명이 아닙니다. 테스트만으로는 절대 완료를 증명하지 못합니다.

## 지속형 상태

상태는 모델의 머릿속이 아니라 레포에 삽니다. 이 스킬은 재개·스티어링·체크포인트 전에 이 아티팩트들을 읽으며, 이들이나 `omo ultragoal status --json` 밖의 상태를 절대 지어내지 않습니다.

| 아티팩트 | 용도 |
|---|---|
| `.omo/ultragoal/brief.md` | 원래 브리프와 지속 제약. |
| `.omo/ultragoal/goals.json` | 골 목록, 각각 `successCriteria` 내장. |
| `.omo/ultragoal/ledger.jsonl` | 모든 결과와 체크포인트의 추가 전용 감사 추적. |

모든 쓰기는 `omo ultragoal` CLI 경로를 거칩니다. 상태 파일은 절대 손으로 편집하지 않습니다.

## 수동 QA 채널

각 기준은 정확히 하나의 채널로 실행되며, PASS를 기록하기 전에 에이전트가 직접 돌립니다. `--dry-run`, 명령 출력, "응답해야 함", "맞아 보임"은 절대 인정되지 않습니다.

| 채널 | 방법 |
|---|---|
| HTTP 호출 | `curl -i`(또는 Playwright APIRequestContext)로 라이브 엔드포인트를 치고 상태 줄 + 헤더 + 본문을 캡처. |
| tmux | `tmux new-session`으로 띄워 `send-keys`로 조작하고 `capture-pane`으로 덤프. 트랜스크립트가 아티팩트. |
| 브라우저 사용 | Playwright / Puppeteer / Chromium으로 실제 페이지를 조작하고 액션 로그 + 스크린샷을 캡처. |
| 컴퓨터 사용 | 실행 중 앱에 대한 OS 수준 GUI 자동화로 액션 로그 + 스크린샷을 캡처. |

순수 CLI stdout, DB 상태 diff, 파싱한 설정 덤프는 CLI나 데이터 형태의 기준을 충족하지만, 사용자 대면 동작에 대한 채널 시나리오를 대체하지는 못합니다.

## 동작 방식

<Steps>

<Step>

### 부트스트랩

브리프로부터 `omo ultragoal create-goals`(`--brief`, `--brief-file`, `--from-stdin`)로 골을 만듭니다. 각 골이 해피 패스, 엣지, 회귀, 적대적 위험을 아우르는 `successCriteria`를 3개 이상 지니도록 다듬습니다. 각 기준에는 `id`, `scenario`, `expectedEvidence`, 적대적 클래스, 정지 조건, 명명된 수동 QA 채널이 들어갑니다. 그런 다음 `omo ultragoal status --json`으로 상태를 점검합니다.

</Step>

<Step>

### 실행 루프

골마다 루프를 돌리며, 골당 5사이클, 동일 기준 동일 실패 3회로 제한합니다. `omo ultragoal complete-goals --json`으로 다음 골을 가져옵니다. 각 기준마다: 시나리오와 이전 원장 항목으로 계획하고, 원자적 투두를 등록하고, 경계 지어진 변경 하나를 실행한 뒤, 명명된 채널 시나리오를 실제로 돌려 관측 가능한 아티팩트를 캡처합니다.

</Step>

<Step>

### 정리 후 기록

기록하기 전에 시나리오가 띄운 모든 런타임 아티팩트를 정리합니다. 서버 PID, tmux 세션, 브라우저 컨텍스트, 컨테이너, 점유한 포트, 임시 파일까지요. 한 줄짜리 정리 영수증을 증거 문자열에 박습니다. `omo ultragoal record-evidence --status pass|fail|blocked`로 정확히 하나의 결과를 기록합니다. 정리 영수증이 없으면 PASS가 아니라 BLOCKED입니다.

</Step>

<Step>

### 골 완료

`omo ultragoal criteria`로 모든 기준이 `pass`인지 확인한 뒤, 증거 요약과 새 골 스냅샷과 함께 `omo ultragoal checkpoint --status complete`를 실행합니다. 막혔거나 실패한 골은 진단 증거와 함께 `--status blocked` / `--status failed`로 체크포인트합니다.

</Step>

<Step>

### 최종 품질 게이트

모든 기준이 통과한 마지막 남은 골에서: 표적 검증을 돌리고, 변경된 파일에 `ai-slop-cleaner`를 돌리고, 검증을 다시 돌린 뒤 `$code-review`를 실행합니다. `recommendation == "APPROVE"` 이고 `architectStatus == "CLEAR"` 이면 클린입니다. 클린 리뷰는 `--quality-gate-json`으로 최종 완료를 체크포인트하고, 클린이 아니면 리뷰 블로커를 대신 기록합니다.

</Step>

</Steps>

## 동적 스티어링

스티어링은 구조화된, 증거 기반 변형에만 쓰입니다. 자연어 스티어링 요청은 거부됩니다. 각 종류는 필수 필드를 가지며 `omo ultragoal steer --kind <kind> ... --evidence "<...>" --rationale "<...>" --json`으로 실행됩니다. 종류에는 `add_subgoal`, `split_subgoal`, `reorder_pending`, `revise_pending_wording`, `revise_criterion`, `annotate_ledger`, `mark_blocked_superseded`가 있습니다.

## 정지 규칙

- 모든 골 완료 + 모든 기준 `pass` + 최종 품질 게이트 클린: DONE.
- 동일 기준 3회 실패, 또는 한 골에서 전체 통과 없이 5사이클: 실패로 체크포인트하고 진단을 드러냅니다.
- 안전 경계(파괴적 명령, 시크릿 유출, 프로덕션 쓰기): 막고 안전한 대체안을 제시합니다.
- 남은 QA 상태(살아 있는 프로세스, tmux 세션, 브라우저 컨텍스트, 점유 포트, 임시 디렉터리): PASS 아님. 정리하고 영수증을 붙인 뒤 계속합니다.
- 사용자가 `/cancel`을 내림: 진행 중 상태를 깨끗이 풀고 자동 재개하지 않습니다.
