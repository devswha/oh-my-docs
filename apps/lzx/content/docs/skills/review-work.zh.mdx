---

`review-work` 是一个实现后审查编排器。完成有意义的实现之后，它会并行启动五个专门的 sub-agent，从互补角度检查工作。只有五个都通过，审查才通过。任何一个 lane 失败，整体审查就失败。

触发词包括 "review work"、"review my work"、"review changes"、"QA my work"、"verify implementation"、"check my work"、"validate changes"、"post-implementation review"。

## 五个审查 Lane

| # | Lane | 回答的问题 | Codex 中的执行方式 |
| --- | --- | --- | --- |
| 1 | 目标和约束验证 | 是否准确完成了用户要求？ | Self-contained reviewer sub-agent。 |
| 2 | QA 执行 | 是否在真实表面上可用？ | Hands-on QA sub-agent。 |
| 3 | 代码质量 | 代码是否一致、可维护？ | `lazycodex-code-reviewer` 或 reviewer sub-agent。 |
| 4 | 安全 | 输入、认证、密钥、数据暴露是否有风险？ | Security-focused reviewer sub-agent。 |
| 5 | 上下文挖掘 | git 历史、issue、周边上下文是否暴露了遗漏？ | Context-mining sub-agent。 |

原始 Skill 中仍有一些 OpenCode 风格示例，但 Codex 不会直接调用它们。Codex 会通过 `multi_agent_v1.spawn_agent` 发送描述角色的自包含消息，并在可用时使用 `lazycodex-code-reviewer`、`lazycodex-qa-executor`、`lazycodex-gate-reviewer` 等已安装角色。

## 工作方式

<Steps>

<Step>

### Phase 0 — 收集上下文

先收集输入，大多来自对话记录：原始 GOAL、CONSTRAINTS、BACKGROUND。CHANGED_FILES 和 DIFF 从 git 自动收集。读取每个变更文件的完整内容，并从 `package.json`、`Makefile`、`docker-compose.yml` 中推断 RUN_COMMAND。

```bash
git diff --name-only HEAD~1   # 或: git diff --name-only main...HEAD
git diff HEAD~1               # 或: git diff main...HEAD
```

不要在主工作树中 checkout PR 分支。用 `git worktree add` 新建一个 git worktree，并在那里工作。

</Step>

<Step>

### Phase 1 — 一次性启动五个 Lane

在一个 turn 中同时启动五个 lane，每个都放到后台。不要顺序执行，也不要中途等待。目标验证者检查需求覆盖，QA 执行者运行真实场景，代码审查者和安全审计者检查 diff 与相关文件，上下文挖掘者检查 git 历史、GitHub、周边上下文和代码交叉引用。

</Step>

<Step>

### Phase 2 — 等待并收集

启动后结束 turn。agent 完成时收集结果。五个全部完成之前，不输出最终报告。

</Step>

<Step>

### Phase 3 — 给出判定

五个全部 PASS，则为 REVIEW PASSED。任意一个 FAIL，则为 REVIEW FAILED。最终报告按 lane 列表呈现，汇总阻塞问题和关键发现；如果失败，要明确写出先修什么、再修什么。

</Step>

</Steps>

## 判定逻辑

- 每个 agent 都必须给出判定和置信度。
- 一次 FAIL 会让整体审查失败。
- timeout、空响应、ack-only、明确的 `BLOCKED:` 都不是 PASS。
- 失败时，发现事项必须指出问题、文件、修复方向。模糊的“改进 X”不够。
- 成功时，报告保持简短，只列出非阻塞建议。
