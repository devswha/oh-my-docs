---

`review-work` は実装後レビューのオーケストレーターです。重要な実装を終えたあと、5 つの専門サブエージェントを並列で起動し、補完的な角度から作業を検査します。レビューが合格するには 5 つすべてが合格する必要があります。1 つでも失敗すればレビューは失敗です。

トリガーには "review work"、"review my work"、"review changes"、"QA my work"、"verify implementation"、"post-implementation review" などの語句が含まれます。

## 5 つのエージェント

| # | エージェント | 種別 | 答える問い | レベル |
| --- | --- | --- | --- | --- |
| 1 | 目標・制約の検証者 | Oracle | ルールの範囲で求められた通りに作ったか？ | MAIN |
| 2 | QA 実行者 | unspecified-high | 実際に動かして機能するか？ | MAIN |
| 3 | コードレビュアー | Oracle | コードはよく書かれ一貫しているか？ | MAIN |
| 4 | セキュリティ監査者 | Oracle | セキュリティ脆弱性はあるか？ | SUB |
| 5 | コンテキストマイナー | unspecified-high | 見落とした文脈はないか？ | MAIN |

Oracle エージェントはファイルを読んだりコマンドを実行したりできないため、diff、ファイル全文、文脈をすべてプロンプトで受け取ります。`unspecified-high` エージェントは自律的で、ファイルを読み、コマンドを実行し、ツールを使うため、生のダンプではなく目標と手がかりを受け取ります。

## 実行の流れ

<Steps>

<Step>

### フェーズ 0 — コンテキスト収集

まず入力を集めますが、多くは会話履歴から取得します: 元の GOAL、CONSTRAINTS、BACKGROUND。CHANGED_FILES と DIFF は git から自動収集し、変更された各ファイルの全文を読み、RUN_COMMAND を `package.json`、`Makefile`、`docker-compose.yml` から検出します。

```bash
git diff --name-only HEAD~1   # または: git diff --name-only main...HEAD
git diff HEAD~1               # または: git diff main...HEAD
```

メインワークツリーで PR ブランチをチェックアウトしないでください。新しい git ワークツリー（`git worktree add`）を作り、そこで作業します。

</Step>

<Step>

### フェーズ 1 — 5 つを一度に起動

5 つのエージェントを 1 ターンで、それぞれバックグラウンドで起動します。逐次起動も、間の待機もありません。目標検証者は目標をサブ要件に分解し、各々を ACHIEVED / MISSED / PARTIAL で印を付けます。QA 実行者は 15〜30 のシナリオをブレインストームして補強し、タスク一覧を作って優先順に実行します。コードレビュアーは正確性、パターン一貫性、型安全性、性能、技術的負債を評価します。セキュリティ監査者は入力検証、認証、シークレット、データ露出を点検します。コンテキストマイナーは git 履歴、GitHub、コミュニケーションチャネル、コードの相互参照を掘り起こします。

</Step>

<Step>

### フェーズ 2 — 待機と収集

起動後はターンを終えます。各エージェントが完了するたびに結論を収集します。5 つすべてが完了するまで最終レポートは出しません。

</Step>

<Step>

### フェーズ 3 — 判定の提示

5 つすべて PASS なら REVIEW PASSED、いずれか FAIL なら REVIEW FAILED です。最終レポートは領域ごとに表化し、ブロッキング課題を集約し、主要な所見を列挙し、失敗時には何をどの順で直すかを正確に記します。

</Step>

</Steps>

## 判定ロジック

- すべてのエージェントが判定と確信度を返します。
- 1 つの FAIL がレビュー全体を失敗させます。
- 失敗時の所見は、問題・ファイル・修正案を指し示します。曖昧な「X を改善検討」はありません。
- 成功時はレポートを短く保ち、ブロッキングでない提案のみ記します。
