`review-work` は実装後レビューのオーケストレーターです。意味のある実装を終えた後、五つの専門 sub-agent を並列で起動し、互いに補完する角度から作業を検査します。レビューは五つすべてが通った場合だけ PASS です。一つでも失敗すればレビュー全体は失敗です。

トリガーは "review work"、"review my work"、"review changes"、"QA my work"、"verify implementation"、"check my work"、"validate changes"、"post-implementation review" などです。

## 五つのレビュー Lane

| # | Lane | 答える問い | Codex での実行 |
| --- | --- | --- | --- |
| 1 | 目標・制約の検証 | 依頼されたものを正確に作ったか？ | Self-contained reviewer sub-agent。 |
| 2 | QA 実行 | 実際の画面や表面で動くか？ | Hands-on QA sub-agent。 |
| 3 | コード品質 | コードは一貫していて保守可能か？ | `lazycodex-code-reviewer` または reviewer sub-agent。 |
| 4 | セキュリティ | 入力、認証、シークレット、データ露出のリスクはないか？ | Security-focused reviewer sub-agent。 |
| 5 | コンテキスト調査 | git 履歴、issue、周辺文脈から見落としはないか？ | Context-mining sub-agent。 |

元の Skill には OpenCode 風の例が一部残っていますが、Codex ではそれをそのまま呼びません。`multi_agent_v1.spawn_agent` で役割を説明する自己完結メッセージを送り、利用可能な場合は `lazycodex-code-reviewer`、`lazycodex-qa-executor`、`lazycodex-gate-reviewer` などのインストール済みロールを使います。

## 動き方

<Steps>

<Step>

### Phase 0 — コンテキスト収集

まず入力を集めます。多くは会話履歴から取得します: 元の GOAL、CONSTRAINTS、BACKGROUND。CHANGED_FILES と DIFF は git から自動収集します。変更された各ファイルの全文を読み、`package.json`、`Makefile`、`docker-compose.yml` から RUN_COMMAND を探します。

```bash
git diff --name-only HEAD~1   # または: git diff --name-only main...HEAD
git diff HEAD~1               # または: git diff main...HEAD
```

メイン worktree で PR ブランチを checkout しません。`git worktree add` で新しい git worktree を作り、そこで作業します。

</Step>

<Step>

### Phase 1 — 五つの Lane を一度に起動する

五つの lane を一つの turn ですべて、各々バックグラウンドで起動します。順番に実行したり、途中で待ったりしません。目標検証者は要件充足を確認し、QA 実行者は実シナリオを動かし、コードレビュアーとセキュリティ監査者は diff と関連ファイルを確認します。コンテキスト調査者は git 履歴、GitHub、周辺文脈、コードの相互参照を見ます。

</Step>

<Step>

### Phase 2 — 待機と収集

起動後に turn を終えます。agent が終わるたびに結果を収集します。五つすべてが終わるまでは最終報告を出しません。

</Step>

<Step>

### Phase 3 — 判定を伝える

五つすべてが PASS なら REVIEW PASSED、一つでも FAIL なら REVIEW FAILED です。最終報告では lane ごとの表、ブロッキング issue、主要な発見、失敗時には何をどの順で直すかを明確に書きます。

</Step>

</Steps>

## 判定ロジック

- すべての agent が判定と信頼度を出します。
- 一つの FAIL でレビュー全体が失敗します。
- timeout、空応答、ack-only、明示的な `BLOCKED:` は PASS ではありません。
- 失敗時の発見事項は、問題、ファイル、修正案を指摘します。曖昧な「X を改善してください」では足りません。
- 成功時の報告は短く保ち、ブロックしない提案だけを書きます。
