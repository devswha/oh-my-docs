`frontend` は、LazyCodex が Web UI、UX、スタイリング、性能、アクセシビリティ、ビジュアル QA を扱うときに使う Skill です。単に画面を埋めるのではなく、既存のデザインシステムと実ブラウザでの検証を基準に UI を作ります。

この Skill は単一のルールファイルというより、ルーターに近いものです。作業内容に応じて `design`、`perfection`、`ui-ux-db` の参照を必要最小限だけ読み、その基準で実装と検証を進めます。

## いつ使うか

- フロントエンド実装、スタイリング、レイアウト、アニメーション作業。
- 既存 UI がぎこちない、または一貫していないため整えたいとき。
- Lighthouse、Core Web Vitals、アクセシビリティ、SEO を実ブラウザで確認したいとき。
- React プロジェクトに `react-grab`、`react-scan`、`react-doctor` などの開発ツールを入れたいとき。
- 色、フォント、チャート、ランディング構成、UX パターンを探したいとき。

## 三つの軸

| 軸 | 役割 |
| --- | --- |
| `design` | デザインシステム、ブランドトーン、レイアウト感覚、ブラウザベースの Design QA。 |
| `perfection` | Lighthouse 100、性能、アクセシビリティ、SEO、Core Web Vitals。 |
| `ui-ux-db` | スタイル、パレット、フォント組み合わせ、UX ガイドライン検索。 |

## 実行原則

- デザインシステムを確認せずに新しいコンポーネントを作らない。
- Lighthouse スコアのために UX を弱めない。
- アイコンは emoji ではなく SVG アイコンセットを使う。
- アニメーションは `transform`、`opacity`、`filter` など GPU-friendly なプロパティを優先する。
- 完了と言う前に、実ブラウザで確認する。

## 一般的な流れ

<Steps>

<Step>

### Route

依頼が UI 実装、既存 UI 改善、性能監査、スタイル検索のどれかをまず分類します。必要な参照だけを読みます。

</Step>

<Step>

### Build

既存コードベースとデザインシステムに合わせて実装します。新しい視覚ルールが必要な場合も、token や再利用可能なパターンとして残します。

</Step>

<Step>

### Verify

ブラウザで desktop/mobile の表示を確認し、必要であれば性能・アクセシビリティ指標もあわせて見ます。

</Step>

</Steps>
