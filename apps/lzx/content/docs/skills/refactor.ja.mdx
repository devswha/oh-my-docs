---

`refactor` スキルは LazyCodex（LZX）が、やみくもな検索置換ではなくインテリジェントで決定論的なリファクタを行うものです。何かに触れる前にコードベースをマッピングし、テストカバレッジを評価して検証戦略を決め、入念に計画し、LSP と AST-grep で段階的に実行します。変更ごとにテストを走らせ、リファクタがリグレッションを生まないようにします。LazyCodex が Codex に導入する OmO（oh-my-openagent）プラグインに同梱されます。

トリガーには "refactor"、"refactoring"、"cleanup"、"restructure"、"extract"、"simplify"、"modernize" が含まれます。

## 使い方

```
/refactor <refactoring-target> [--scope=<file|module|project>] [--strategy=<safe|aggressive>]
```

- **refactoring-target** — ファイルパス、シンボル名、パターン、または「検証ロジックを別モジュールに抽出」のような記述。
- **`--scope`** — `file`、`module`（デフォルト）、`project` のいずれか。
- **`--strategy`** — `safe`（デフォルト、最大のテストカバレッジを要求）または `aggressive`（十分なカバレッジ下でより広い変更）。

## フェーズ

<Steps>

<Step>

### フェーズ 0 — 意図ゲート

いかなる行動の前にもリクエストを分類します。具体的なファイル・シンボルや「X を Y にリファクタ」は分析へ直行します。漠然とした「改善して」/「整理して」、曖昧なスコープ、不足した文脈は、何かを動かす前に必ず明確化質問をトリガーします。

</Step>

<Step>

### フェーズ 1 — コードベース分析

並列の explore エージェントを起動し、対象、その依存元、類似パターン、関連テスト、アーキテクチャ文脈を見つけます。実行中は直接ツールを使います: 定義・参照・シンボルには LSP、構造的パターン分析には AST-grep を使います。

</Step>

<Step>

### フェーズ 2 — コードマップ構築

分析から確定的なコードマップを作ります: コアファイル、依存グラフ（imports / imported-by / used-by）、リスクとカバレッジで格付けした影響ゾーン、リファクタが従うべき既存パターン。そこから、従うべき既存パターン、壊してはならないもの、安全に変えられるもの、移行が必要なものを印付けます。

</Step>

<Step>

### フェーズ 3 — テスト評価

テストインフラを検出してカバレッジを分析します。カバレッジが検証戦略を決めます: 高（>80%）は各ステップ後に既存テストを走らせ、中（50-80%）は安全アサーションを足し、低（&lt;50%）は止めてテスト追加を先に提案し、無は aggressive リファクタを即座にブロックします。

</Step>

<Step>

### フェーズ 4 — 計画生成

コードマップ、カバレッジ、制約とともに Plan エージェントを呼び、依存順に並んだ原子的で独立検証可能なステップを生成します。各ステップには正確なファイルと行範囲、ロールバック戦略が入ります。計画を細粒度の Todo に変換します。

</Step>

<Step>

### フェーズ 5 — 実行

各ステップを厳格なプロトコルで実行します: 現在状態の読み取り、ベースライン診断の確認、実行、その後の検証（診断クリーン、テスト合格、型チェッククリーン）。いかなる失敗も即座に停止してリバートします。テストが壊れたまま次のステップへ進みません。

</Step>

<Step>

### フェーズ 6 — 最終検証

完全なテストスイート、型チェック、lint、ビルド（該当時）、変更した全ファイルの最終診断を走らせ、何が変わったかと検証結果の要約を作ります。

</Step>

</Steps>

## 構造的検索と書き換え — AST-grep の在り処

リファクタこそ AST-grep が実際に動く場所です。`ast_grep_search` はテキストでなく形でコードを探し、`ast_grep_replace` はマッチしたパターンを書き換えます。重要な規則: `ast_grep_replace` は必ず `dryRun=true` で先に走らせ、プレビューを確認してから `dryRun=false` で実行します。

LSP と AST-grep は互いを補完します。LSP は実在のシンボルに関する意味的な問いに答えます。安全なワークスペース全体のリネームには `lsp_prepare_rename` のあと `lsp_rename`、編集ごとに `lsp_diagnostics`。AST-grep は単一シンボルのリネームでない変換のために、構文木レベルのパターンマッチと編集を行います。

## 重要なルール

- 変更後の診断チェックを飛ばさず、テスト失敗のまま進めません。
- `as any`、`@ts-ignore`、`@ts-expect-error` を使わず、通すためにテストを削除しません。
- 既存パターンを理解した後にのみリファクタし、適用前にプレビューします。
- 非推奨 API は `librarian` に現代的な代替を尋ねますが、ユーザーが移行を明示的に要求しない限りライブラリのバージョンを自動で上げません。
- 複雑なアーキテクチャ判断は（読み取り専用の）`oracle` に相談します。
- ゼロカバレッジ、公開 API の破壊、不明確なスコープ、連続 3 回の検証失敗では、中止してユーザーに相談します。
