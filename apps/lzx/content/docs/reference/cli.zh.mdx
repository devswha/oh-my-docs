LazyCodex 只提供一个 CLI：`lazycodex-ai` npm bin。它安装并诊断 OmO-based Codex light setup。没有全局安装；始终通过 `npx` 调用。

## Forwarding to OmO

bin 会读取参数并转发给 `oh-my-openagent`。`install` 子命令会被特殊处理，以固定 Codex platform；其余参数原样透传。

<Tabs items={["install", "other"]}>

<Tab value="install">

```bash
npx lazycodex-ai install
```

会展开为：

```bash
npx --yes --package oh-my-openagent omo install --platform=codex
```

`install` 后面的所有参数都会原样追加。例如：

```bash
npx lazycodex-ai install --no-tui --codex-autonomous
# -> npx --yes --package oh-my-openagent omo install --platform=codex --no-tui --codex-autonomous
```

</Tab>

<Tab value="other">

第一个参数不是 `install` 时，会不固定 platform，直接转发给 `omo`：

```bash
npx lazycodex-ai <args...>
# -> npx --yes --package oh-my-openagent omo <args...>
```

</Tab>

</Tabs>

## `--dry-run`

把 `--dry-run` 放在**第一个**参数时，会打印解析后的 `npx` 命令而不执行：

```bash
npx lazycodex-ai --dry-run install --no-tui --codex-autonomous
```

输出：

```text
npx --yes --package oh-my-openagent omo install --platform=codex --no-tui --codex-autonomous
```

`--dry-run` 会在转发前被移除，因此剩余参数的解析与真实执行完全一致。执行前可用它确认准确的 `omo` 调用。

## `doctor`

```bash
npx lazycodex-ai doctor
```

输出 health report，说明哪些已配置、哪些缺失以及原因。它会检查 plugin cache、hooks、MCP Servers、Agent Roles、Codex config state。hooks pending、skills 未加载或 routing 异常时先运行它。

## Marketplace Alternative

`npx` 安装是主路径。作为附加的实验路径，也可以从 Codex 安装插件。

```bash
codex plugin marketplace add https://github.com/code-yeongyu/lazycodex
codex plugin add omo@sisyphuslabs
```

在下一次 Codex startup review 中批准 omo hooks。第一次批准后的 session 会运行 background bootstrap，并在完成后提示重启。升级命令：

```bash
codex plugin marketplace upgrade sisyphuslabs
```

升级后插件文件发生变化，trust hash 需要重新批准，hooks 会显示为 **Modified**。再次批准后，下一个 session 会为新版本运行 bootstrap。

## Exit Behavior

bin 会用 inherited stdio spawn 解析后的命令，并以该命令的状态码退出。如果 `npx` 本身 spawn 失败，会打印错误并以非零码退出。

## In-session Commands

安装后，LazyCodex 会在 Codex 内注册 OmO commands。它们不是 shell 命令，而是在 Codex 输入框中使用的 `$command`。语法和使用流程集中在 [Commands](/zh/docs/commands)。
