title: $start-work

---

`$start-work` は [Prometheus 作業計画](/ja/docs/commands/ulw-plan)を、トップレベルのチェックボックスがすべて完了するまで実行します。対象は `.omo/plans/<slug>.md` で、進捗は `.omo/boulder.json` に保存されます。

## Syntax

```bash
$start-work [plan-name] [--worktree <absolute-path>]
```

- `plan-name` は任意です。`.omo/plans/<slug>.md` のファイル名または一部の名前を指定できます。
- `--worktree <absolute-path>` は任意です。隔離された git worktree で作業を実行します。

## How It Works

- `.omo/boulder.json` の **Boulder 状態**はターンやセッションをまたいで維持され、実行の間に進捗が失われません。
- **Stop-hook** が計画完了まで次のターンを再注入します。ループは自己継続します。
- **独立したサブタスクは並列 subagents へ展開されます。**
- 厳格な TDD に加えて **5 つの証拠ゲート**があります: 計画の再読、自動検証、手動 QA、敵対的 QA、クリーンアップ。
- チェックボックスが付くたびに、**証拠が `.omo/start-work/ledger.jsonl` に記録されます**。
- 実行者は子タスクの「完了」報告をそのまま信じず、独立検証が通ってからチェックボックスを完了扱いにします。

利用可能な計画がない状態でユーザーが明示的に `$start-work` を呼んだ場合、最新の流れではまず `$ulw-plan` を起動して実行可能な計画を作り、その後に実行へ進みます。

## Completion

すべてのチェックボックスが付くと、`ORCHESTRATION COMPLETE` ブロックを出力します。

```text
ORCHESTRATION COMPLETE
```

## Typical Flow

<Steps>

<Step>

### Prepare a Plan

`$ulw-plan` で計画を書くか、既存の `.omo/plans/<slug>.md` を `$start-work` に指定します。

```bash
$ulw-plan "add OAuth login with refresh tokens"
```

</Step>

<Step>

### Execute

```bash
$start-work oauth-refresh-tokens
```

独立した作業は並列 subagents に展開され、Boulder 進捗が中断に耐えます。

</Step>

<Step>

### Wait for Completion

`$start-work` はすべてのチェックボックスが付いたときだけ停止し、`ORCHESTRATION COMPLETE` を出力します。

</Step>

</Steps>
