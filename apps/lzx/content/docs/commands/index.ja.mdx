LazyCodex は Codex 入力欄で使う `$command` をインストールします。これは shell で実行するコマンドではなく、Codex セッション内のワークフロー入口です。インストール後に `$` を押したときに見える項目がここに該当します。

## Core Commands

| Command | 入力 | 役割 |
| --- | --- | --- |
| [`$init-deep`](/ja/docs/commands/init-deep) | `$init-deep` | 階層的な `AGENTS.md` プロジェクトメモリを生成します。大規模または変更直後のコードベースで実行します。 |
| [`$ulw-plan`](/ja/docs/commands/ulw-plan) | `$ulw-plan "what to build"` | 実装前に探索し、質問し、`.omo/plans/<slug>.md` に決定済みの計画を書きます。製品コードは書きません。 |
| [`$start-work`](/ja/docs/commands/start-work) | `$start-work [plan-name] [--worktree <path>]` | `.omo/plans` の計画をチェックボックス単位で実行します。完了時に **ORCHESTRATION COMPLETE** を出力します。 |
| [`$ulw-loop`](/ja/docs/commands/ulw-loop) | `$ulw-loop "task"` | `.omo/ulw-loop` の状態と証拠台帳を使い、完了が検証されるまで作業を続けます。 |

## How They Compose

通常は 3 段階で考えます。まずプロジェクトメモリを作り、判断が必要な作業は計画し、その計画を実行します。計画なしの開いた目標を証拠で完了まで押し切るときだけ `$ulw-loop` を使います。

<Steps>

<Step>

### Create Project Memory

大きなコードベースならまず `$init-deep` を実行します。以降のコマンドは `AGENTS.md` の階層を読み、よりよい文脈で始められます。

```bash
$init-deep
```

</Step>

<Step>

### Plan and Execute

実装前に判断が必要なら、まず計画を書き、その計画を `$start-work` に渡します。

```bash
$ulw-plan "add OAuth login with refresh tokens"
$start-work
```

`$ulw-plan` はインタビューし、コードベースを探索し、製品コードに触れず `.omo/plans/<slug>.md` を書きます。その後 `$start-work` が計画をチェックボックスごとに実行します。

</Step>

<Step>

### Verified Completion for Open Work

希望的な状態報告ではなく、証拠が完了を証明するまで動き続けるべき作業では、ループを回します。

```bash
$ulw-loop "make the flaky payment tests pass"
```

エージェントは作業を続け、完了の約束を出しますが、ループは Oracle が結果を検証した後にだけ終了します。検証に失敗すれば継続します。

</Step>

</Steps>
