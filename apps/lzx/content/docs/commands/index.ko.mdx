LazyCodex는 Codex 입력창에서 쓰는 `$command`를 설치합니다. 셸에서 실행하는 명령이 아니라, Codex 세션 안에서 입력하는 워크플로우 진입점입니다. 설치 후 `$`를 눌렀을 때 보이는 항목들이 여기에 해당합니다.

## 핵심 커맨드

| 커맨드 | 입력 | 하는 일 |
| --- | --- | --- |
| [`$init-deep`](/ko/docs/commands/init-deep) | `$init-deep` | 계층형 `AGENTS.md` 프로젝트 메모리를 생성합니다. 크거나 막 변경된 코드베이스에서 실행하세요. |
| [`$ulw-plan`](/ko/docs/commands/ulw-plan) | `$ulw-plan "what to build"` | 구현 전에 탐색하고 질문한 뒤, `.omo/plans/<slug>.md`에 결정 완료 계획을 씁니다. 제품 코드는 쓰지 않습니다. |
| [`$start-work`](/ko/docs/commands/start-work) | `$start-work [plan-name] [--worktree <path>]` | `.omo/plans`의 계획을 체크박스 단위로 실행합니다. 완료되면 **ORCHESTRATION COMPLETE**를 출력합니다. |
| [`$ulw-loop`](/ko/docs/commands/ulw-loop) | `$ulw-loop "task"` | `.omo/ulw-loop` 상태와 증거 원장을 사용해 완료가 검증될 때까지 작업을 이어 갑니다. |

## 어떻게 조합하나요

보통은 세 단계로 생각하면 됩니다. 먼저 프로젝트 메모리를 만들고, 결정이 필요한 일은 계획한 뒤, 계획을 실행합니다. 계획 없이 열린 목표를 끝까지 밀어야 할 때만 `$ulw-loop`를 씁니다.

<Steps>

<Step>

### 프로젝트 메모리 만들기

큰 코드베이스라면 먼저 `$init-deep`를 실행합니다. 이후 커맨드들이 `AGENTS.md` 계층을 읽고 더 나은 맥락에서 시작합니다.

```bash
$init-deep
```

</Step>

<Step>

### 계획하고 실행하기

구현 전에 결정이 필요한 작업이라면 먼저 계획을 세운 뒤 그 계획을 `$start-work`에 넘깁니다.

```bash
$ulw-plan "add OAuth login with refresh tokens"
$start-work
```

`$ulw-plan`은 인터뷰하고 코드베이스를 탐색한 뒤, 제품 코드는 건드리지 않고 `.omo/plans/<slug>.md`를 씁니다. 그다음 `$start-work`가 그 계획을 체크박스 하나씩 실행합니다.

</Step>

<Step>

### 열린 작업의 검증된 완료

희망 섞인 상태 보고가 아니라 증거가 완료를 증명할 때까지 계속 움직여야 하는 작업이라면 루프를 돌립니다.

```bash
$ulw-loop "make the flaky payment tests pass"
```

에이전트는 계속 작업하며 완료 약속을 내보내지만, 루프는 Oracle이 결과를 검증한 뒤에만 끝납니다. 검증이 실패하면 계속 이어집니다.

</Step>

</Steps>
