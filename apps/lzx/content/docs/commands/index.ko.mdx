---
title: 커맨드 기둥
description: 네 가지 LazyCodex 커맨드 — $ulw-plan, $start-work, $ulw-loop, $init-deep — 와 이들을 조합하는 방법.
---

LazyCodex는 작고 안정적인 커맨드 레이어를 Codex에 설치합니다. 커맨드 레이어는 의도적으로 단순하게 유지되며, 전문가적 판단은 [스킬](/ko/docs/skills)이 담당합니다. 이 커맨드들은 Codex용 OmO 커맨드로 제공되며, 인스톨러가 보여주는 `$command` 문법으로 호출합니다.

## 네 가지 기둥

| 커맨드 | 입력 | 하는 일 |
| --- | --- | --- |
| [`$ulw-plan`](/ko/docs/commands/ulw-plan) | `$ulw-plan "what to build"` | Prometheus 전략 플래너입니다. `plans/<slug>.md`에 계획을 씁니다. 제품 코드는 절대 쓰지 않습니다. |
| [`$start-work`](/ko/docs/commands/start-work) | `$start-work [plan-name] [--worktree <path>]` | 모든 체크박스가 끝날 때까지 계획을 실행합니다. **ORCHESTRATION COMPLETE**를 출력합니다. |
| [`$ulw-loop`](/ko/docs/commands/ulw-loop) | `$ulw-loop "task" [--completion-promise=TEXT] [--strategy=reset\|continue]` | Oracle이 검증한 완료까지 도는 자기 참조 루프입니다. ultrawork 모드에서 최대 500회, 일반 모드에서 100회 반복합니다. |
| [`$init-deep`](/ko/docs/commands/init-deep) | `$init-deep` | 계층형 `AGENTS.md` 프로젝트 메모리를 생성합니다. 크거나 막 변경된 코드베이스에서 실행하세요. |

## 어떻게 조합하나요

자연스러운 흐름이 두 가지 있습니다. 시작 시점에 얼마나 알고 있는지로 고릅니다.

<Steps>

<Step>

### 먼저 계획하고, 그다음 실행

구현 전에 결정이 필요한 작업이라면 먼저 계획을 세운 뒤 그 계획을 `$start-work`에 넘깁니다.

```bash
$ulw-plan "add OAuth login with refresh tokens"
$start-work
```

`$ulw-plan`은 인터뷰하고 코드베이스를 탐색한 뒤, 제품 코드는 건드리지 않고 `plans/<slug>.md`를 씁니다. 그다음 `$start-work`가 그 계획을 체크박스 하나씩, 지속 가능한 Boulder 진행 상태와 함께 완료까지 끌고 갑니다.

</Step>

<Step>

### 열린 작업의 검증된 완료

희망 섞인 상태 보고가 아니라 증거가 완료를 증명할 때까지 계속 움직여야 하는 작업이라면 루프를 돌립니다.

```bash
$ulw-loop "make the flaky payment tests pass"
```

에이전트는 계속 작업하며 완료 약속을 내보내지만, 루프는 Oracle이 결과를 검증한 뒤에만 끝납니다. 검증이 실패하면 계속 이어집니다.

</Step>

</Steps>

## 프로젝트 메모리가 먼저입니다

큰 코드베이스라면 두 흐름 중 무엇이든 시작하기 전에 `$init-deep`을 한 번 실행하세요. 복잡한 디렉터리에 점수를 매기고 계층형 `AGENTS.md`를 작성해, 이후 모든 커맨드가 충분한 맥락에서 시작하게 합니다.

```bash
$init-deep
```
