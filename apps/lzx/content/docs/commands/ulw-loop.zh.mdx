---

`$ulw-loop` 是 LazyCodex 的持久化命令。它不会靠乐观状态汇报停止，而是持续工作，直到真实使用表面捕获的证据证明结果完成。

当任务是开放式的，并且你希望以证据而不是智能体自己的“完成”声明来判断时使用它。

## 语法

```bash
$ulw-loop "task description" [--completion-promise=TEXT] [--strategy=reset|continue]
```

| 组成 | 含义 |
| --- | --- |
| `"task description"` | 要达成的结果。写期望结果，而不是步骤。 |
| `--completion-promise=TEXT` | 兼容语法。需要在完成前满足特定承诺文本时使用。 |
| `--strategy=reset` | 兼容语法。需要从新状态重新开始循环时使用。 |
| `--strategy=continue` | 兼容语法。需要在迭代之间延续状态时使用。 |

实际运行状态会保存在 `.omo/ulw-loop` 下。

| 文件 | 作用 |
| --- | --- |
| `.omo/ulw-loop/brief.md` | 原始请求与持续约束 |
| `.omo/ulw-loop/goals.json` | 目标与成功标准 |
| `.omo/ulw-loop/ledger.jsonl` | pass、fail、block、steering、checkpoint 的证据账本 |

## 工作方式

该循环刻意设计为不能只凭智能体自我报告结束。

<Steps>

<Step>

### 持续工作

智能体把目标拆成成功标准，规划需要的工作，并收集证据。最新流程会使用 `omo ulw-loop create-goals`、`complete-goals`、`record-evidence`、`checkpoint` 等内部 CLI 状态。

</Step>

<Step>

### 输出完成承诺

即使智能体认为工作已经完成，那也只是意图信号，**不会**结束循环。每个成功标准都需要可观察证据。

</Step>

<Step>

### Oracle 验证

验证者根据证据审查结果。测试本身并不总是足够；还需要来自真实表面的证据，例如 HTTP 调用、tmux transcript、浏览器截图、computer-use 日志、CLI stdout、数据 diff。

</Step>

<Step>

### 失败时继续

验证失败时，循环会重跑该标准，或把失败/阻塞状态写入账本，然后进入下一步判断。

</Step>

</Steps>

## 失败上限

最新工作流设置了限制，避免循环无限运行。

| 条件 | 上限 |
| --- | --- |
| 一个 goal 没有完整通过时的重复 | 5 个 cycle |
| 同一 criterion 的同一种失败 | 3 次 |

## 何时使用

- 工作是**开放式**的：例如“让不稳定测试套件通过”“让构建变绿”“完成这次迁移”。
- 你希望以**证据**而不是状态汇报来判断完成。
- 你想让智能体持续迭代，直到验证者批准。

需要在写代码前先做决策时用 `$ulw-plan`。已有书面计划并要执行到最后一个复选框时用 `$start-work`。
