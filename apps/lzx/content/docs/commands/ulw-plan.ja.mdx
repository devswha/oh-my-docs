---
title: $ulw-plan
description: "Prometheus 戦略プランナー: 製品コードを書かず、.omo/plans に決定済みの計画を書きます。"
---

`$ulw-plan` は Prometheus 戦略計画コンサルタントです。アイデアを決定済みの作業計画へ変えます。実装者では**なく**プランナーです。「X をやって」と言うと、X の計画を作るだけで、製品コードは絶対に書きません。

成果物は `.omo/drafts/<slug>.md` と `.omo/plans/<slug>.md` です。draft は探索と承認状態を残す再開地点で、plan は `$start-work` がそのまま実行できるチェックリストです。

## Syntax

```bash
$ulw-plan "what to build"
```

## When To Use It

実装前に判断が必要な作業に `$ulw-plan` を使います。範囲が不明確、可能なアプローチが複数ある、または誰かがコードに触れる前に順序を決めるべき横断的な変更などです。

手早い回答や小さな編集だけなら普通にプロンプトします。計画がすでにあるなら、そのまま [`$start-work`](/ja/docs/commands/start-work) へ進みます。

## Flow

<Steps>

<Step>

### Socratic Interview

Prometheus は計画前に曖昧さを取り除くために必要な質問をします。

</Step>

<Step>

### Parallel Codebase Exploration

関連コードを並列に探索し、実際に存在するものの上に計画を立てます。

</Step>

<Step>

### Metis Gap Analysis

欠けている判断、エッジケース、リスクを見つけるギャップ分析を行います。

</Step>

<Step>

### Plan Writing

作業計画を `.omo/plans/<slug>.md` に書きます。スキャフォールドスクリプトが構造を作り、計画内容は定められた領域に追加されます。

</Step>

<Step>

### Review and Approval

必要なら Metis/Momus 系のレビューを通します。意図が不明確な作業では、planner が best-practice の既定値を採用し、より強いレビューを自動で付けることがあります。ユーザー承認なしに計画が実行へ進むことはありません。

</Step>

</Steps>

## What You Get

質問、調査、そして `$start-work` にそのまま渡せる `.omo/plans/<slug>.md` の決定済み作業計画です。

```bash
$ulw-plan "migrate the auth service to OAuth with refresh tokens"
# → .omo/plans/oauth-refresh-tokens.md を書き出す
$start-work
```
