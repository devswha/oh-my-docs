`$ulw-loop` は LazyCodex の永続性コマンドです。希望的な状態報告ではなく、実際の利用表面から取った証拠で結果が検証されるまで作業を続けます。

タスクが開いた形で、エージェント自身の「完了」宣言ではなく証拠で完了を判断したいときに使います。

## Syntax

```bash
$ulw-loop "task description" [--completion-promise=TEXT] [--strategy=reset|continue]
```

| Component | 意味 |
| --- | --- |
| `"task description"` | 到達したい結果。手順ではなく、ほしい結果を書きます。 |
| `--completion-promise=TEXT` | 互換構文。完了前に満たすべき約束テキストを指定するときに使います。 |
| `--strategy=reset` | 互換構文。新しい状態から繰り返しを始めるときに使います。 |
| `--strategy=continue` | 互換構文。繰り返しの間で状態を引き継ぐときに使います。 |

実行状態は `.omo/ulw-loop` の下に残ります。

| File | Role |
| --- | --- |
| `.omo/ulw-loop/brief.md` | 元の依頼と継続する制約 |
| `.omo/ulw-loop/goals.json` | 目標と成功基準 |
| `.omo/ulw-loop/ledger.jsonl` | pass、fail、block、steering、checkpoint の証拠台帳 |

## How It Works

このループは、エージェント自身の判断だけでは終われないよう意図的に設計されています。

<Steps>

<Step>

### It Keeps Working

エージェントは目標を成功基準へ分解し、必要な作業を計画し、証拠を集めます。最新の流れでは `omo ulw-loop create-goals`、`complete-goals`、`record-evidence`、`checkpoint` などの内部 CLI 状態を使います。

</Step>

<Step>

### It Emits Completion Promises

作業が終わったと判断しても、それは意図のシグナルにすぎず、ループを終わらせ**ません**。各成功基準には観測可能な証拠が必要です。

</Step>

<Step>

### Oracle Verifies

検証者は証拠をもとに結果をレビューします。テストだけでは十分ではなく、HTTP 呼び出し、tmux transcript、ブラウザ screenshot、computer-use log、CLI stdout、data diff など、実際の表面の証拠が必要です。

</Step>

<Step>

### It Continues on Failure

検証に失敗すると、同じ基準を再実行するか、失敗・ブロック状態を ledger に記録して次の判断へ進みます。

</Step>

</Steps>

## Failure Limits

最新ワークフローは無限に回らないよう失敗上限を持ちます。

| Condition | Limit |
| --- | --- |
| 1 つの goal で全体通過なしに繰り返す | 5 cycles |
| 同じ criterion の同じ失敗 | 3 回 |

## When To Use It

- 作業が**開いた形**のとき: 「不安定なテストスイートを通す」「ビルドをグリーンにする」「このマイグレーションを終える」。
- 完了を状態報告ではなく**証拠**で判断したいとき。
- 検証者が承認するまでエージェントに反復を任せたいとき。

コード前に計画が必要な判断中心の作業には `$ulw-plan` を使います。作成済みの計画を最後のチェックボックスまで実行するには `$start-work` を使います。
