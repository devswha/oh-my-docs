LazyCodex(LZX)保持精简的公开表面:四个命令支柱(`$ulw-plan`、`$start-work`、`$ulw-loop`,以及 `/init-deep`)和叠加其上的技能。诀窍在于选择与工作形态相匹配的命令,然后让它把任务带到经过验证的完成。

LazyCodex 将 [OmO(oh-my-openagent)](https://github.com/code-yeongyu/oh-my-openagent) 打包为 Codex 智能体框架 —— 可视作 Codex 版的 LazyVim。底层由 Sisyphus 编排 Hephaestus、Oracle 和 Librarian,你只需选择入口。

## 选择合适的命令

多数任务无需完整循环。让命令匹配工作的真实需求:

| 需求 | 使用 | 原因 |
| --- | --- | --- |
| 写代码前先决策 | `$ulw-plan` | Prometheus 将模糊想法变为 `plans/<slug>.md` 中的决策完备计划。 |
| 执行已批准的计划 | `$start-work` | 以持久的 Boulder 进度运行清单,直到每个复选框完成。 |
| 开放式、必须验证 | `$ulw-loop` | 持续推进,直到 Oracle 以证据验证结果。 |
| 初次接触大型仓库 | `/init-deep` | 在任何编辑前生成分层的 `AGENTS.md` 项目记忆。 |

`$ulw-plan` 从不编写产品代码 —— 它以一份可交给 `$start-work` 的计划收尾。当你希望以验证而非乐观的状态汇报来衡量进度时,使用 `$ulw-loop`。

## 用例

### 计划优先工作

经典流程:用 `$ulw-plan` 确定方案,再用 `$start-work` 推进。

- **何时使用:** 在实现开始前需要敲定决策、排序或取舍时。
- **命令:** `$ulw-plan` 然后 `$start-work`。

<Steps>

<Step>

#### 用 `$ulw-plan` 规划工作

Prometheus 进行苏格拉底式访谈,并行探索代码库,运行 Metis 缺口分析,然后将决策完备的计划写入 `plans/<slug>.md`。它从不触碰产品代码。

```bash
$ulw-plan "添加带刷新令牌轮换的 OAuth 登录"
```

你带走的是一份计划文件,而非代码改动。

</Step>

<Step>

#### 用 `$start-work` 执行计划

将计划交给 `$start-work`。它以 `.omo/boulder.json` 中的持久 Boulder 状态运行每个顶层复选框,将独立子任务分散到并行子智能体,并应用严格的 TDD。Stop-hook 会在计划完成前持续重新注入下一轮,且五道证据关卡(重读计划、自动验证、手动 QA、对抗式 QA、清理)必须全部通过。

```bash
$start-work add-oauth-login
```

当每个复选框被勾选后,它会打印 **ORCHESTRATION COMPLETE** 块。

</Step>

</Steps>

整个流程合为一块:

```bash
$ulw-plan "添加带刷新令牌轮换的 OAuth 登录"
$start-work add-oauth-login
```

如果需求已清晰到可直接规划,可跳过偏访谈的规划阶段,在已有计划时从 `$start-work` 开始。

### 开放式、可验证的工作

- **何时使用:** 任务应持续推进直到结果被证明,而非智能体宣称完成即止。
- **命令:** 配合 `--completion-promise` 的 `$ulw-loop`。

`$ulw-loop` 是自引用的。智能体认为任务完成时会发出承诺,但这并不结束循环 —— Oracle 必须先验证结果。若验证失败,循环继续。迭代上限为 ultrawork 模式 500 次、普通模式 100 次。

```bash
$ulw-loop "让 CI 上不稳定的结账测试通过" --completion-promise="结账套件连续 3 次绿色" --strategy=continue
```

用 `--strategy=reset` 在每次迭代时重置上下文,或用 `--strategy=continue` 保留累积上下文。

### 上手大型代码库

- **何时使用:** 仓库太大或太旧,无法凭记忆解释,后续智能体在编辑前需要路标时。
- **命令:** `/init-deep`。

`/init-deep` 为复杂目录评分,并在需要的代码附近写入分层 `AGENTS.md` 上下文,让之后每次运行都以项目记忆而非猜测起步。

```bash
/init-deep
```

代码库形态变化时再次运行。

### 工作树隔离执行

- **何时使用:** 希望在验证前与主检出隔离的结构性改动或高风险实验。
- **命令:** `$start-work --worktree <path>`。

将执行指向隔离的 Git 工作树,让计划运行时主检出保持干净。

```bash
$start-work refactor-payments --worktree /home/you/wt-refactor-payments
```

`--worktree` 需传入绝对路径。计划在该工作树内执行。打印 `ORCHESTRATION COMPLETE` 后再审查并合并。

### 实现后审查

- **何时使用:** 代码已写完,合并前想多角度检查。
- **命令:** `review-work` 技能。

实现后运行 `review-work` 技能获取多角度的实现后审查。可搭配 `remove-ai-slops`(在不改变行为的前提下清理 AI 痕迹代码)或 `comment-checker`(编辑类操作后给出反馈)。
