LazyCodex(LZX)は公開表面を小さく保ちます。4 つのコマンド柱(`$ulw-plan`、`$start-work`、`$ulw-loop`、そして `$init-deep`)と、その上に重なるスキルがすべてです。コツは作業の形に合うコマンドを選び、検証済みの完了までそのコマンドにタスクを運ばせることです。

LazyCodex は [OmO(oh-my-openagent)](https://github.com/code-yeongyu/oh-my-openagent) を Codex のエージェントハーネスとしてパッケージ化したものです — Codex 版の LazyVim と考えてください。内部では Sisyphus が Hephaestus、Oracle、Librarian をオーケストレーションし、ユーザーは入口を選ぶだけです。

## 適切なコマンドを選ぶ

ほとんどのタスクはフルループより少ない手数で済みます。作業が実際に必要とするものにコマンドを合わせてください:

| 必要なこと | 使うもの | 理由 |
| --- | --- | --- |
| コード前の意思決定 | `$ulw-plan` | Prometheus が曖昧な発想を `plans/<slug>.md` の意思決定完了プランに変えます。 |
| 承認済みプランの実行 | `$start-work` | 全チェックボックスが終わるまで耐久性のある Boulder 進捗でチェックリストを実行します。 |
| オープンエンド、要検証 | `$ulw-loop` | Oracle が証拠で結果を検証するまで進み続けます。 |
| 大規模リポジトリへの初接触 | `$init-deep` | 編集前に階層的な `AGENTS.md` プロジェクトメモリを生成します。 |

`$ulw-plan` は製品コードを書きません — `$start-work` に渡すプランで終わります。希望的な状況報告ではなく検証で進捗を測りたいときは `$ulw-loop` を使ってください。

## ユースケース

### 計画優先の作業

定番の流れです。`$ulw-plan` で方針を決め、`$start-work` で運びます。

- **いつ使うか:** 実装開始前に意思決定・順序付け・トレードオフを固めたいとき。
- **コマンド:** `$ulw-plan` の次に `$start-work`。

<Steps>

<Step>

#### `$ulw-plan` で作業を計画する

Prometheus がソクラテス式インタビューを行い、コードベースを並列に探索し、Metis ギャップ分析を回したうえで、意思決定完了のプランを `plans/<slug>.md` に書き出します。製品コードには一切触れません。

```bash
$ulw-plan "リフレッシュトークンローテーション付き OAuth ログインを追加"
```

手にするのはコード変更ではなくプランファイルです。

</Step>

<Step>

#### `$start-work` でプランを実行する

プランを `$start-work` に渡します。`.omo/boulder.json` の耐久性ある Boulder 状態で全トップレベルチェックボックスを実行し、独立したサブタスクを並列サブエージェントへ分散し、厳格な TDD を適用します。Stop-hook がプラン完了まで次のターンを再注入し、5 つの証拠ゲート(プラン再読、自動検証、手動 QA、敵対的 QA、クリーンアップ)をすべて通過する必要があります。

```bash
$start-work add-oauth-login
```

全チェックボックスがチェックされると **ORCHESTRATION COMPLETE** ブロックを出力します。

</Step>

</Steps>

一連の流れを 1 ブロックで:

```bash
$ulw-plan "リフレッシュトークンローテーション付き OAuth ログインを追加"
$start-work add-oauth-login
```

要件がすでに直接計画できるほど明確なら、インタビュー中心の計画段階を飛ばし、プランがある状態で `$start-work` から始めても構いません。

### オープンエンドで検証付きの作業

- **いつ使うか:** エージェントが完了を主張した時点ではなく、結果が証明されるまで進み続けるべきタスク。
- **コマンド:** `--completion-promise` を付けた `$ulw-loop`。

`$ulw-loop` は自己参照的です。エージェントは完了したと判断するとプロミスを出しますが、それだけではループは終わりません — Oracle がまず結果を検証する必要があります。検証が失敗すればループは続きます。反復上限は ultrawork モードで 500 回、通常モードで 100 回です。

```bash
$ulw-loop "CI で不安定なチェックアウトテストを通す" --completion-promise="チェックアウトスイートが 3 回連続グリーン" --strategy=continue
```

反復ごとにコンテキストを新規に始めるなら `--strategy=reset`、蓄積したコンテキストを維持するなら `--strategy=continue` を使います。

### 大規模コードベースのオンボーディング

- **いつ使うか:** リポジトリが記憶だけで説明するには大きすぎる・古すぎて、以降のエージェントが編集前に目印を必要とするとき。
- **コマンド:** `$init-deep`。

`$init-deep` は複雑なディレクトリをスコアリングし、必要なコードの近くに階層的な `AGENTS.md` コンテキストを書き込みます。これにより以降のすべての実行が推測ではなくプロジェクトメモリから始まります。

```bash
$init-deep
```

コードベースの形が変わったら再実行してください。

### ワークツリー隔離実行

- **いつ使うか:** 検証まではメインチェックアウトから切り離したい構造的変更やリスクのある実験。
- **コマンド:** `$start-work --worktree <path>`。

実行を隔離された Git ワークツリーへ向け、プラン実行中もメインチェックアウトをクリーンに保ちます。

```bash
$start-work refactor-payments --worktree /home/you/wt-refactor-payments
```

`--worktree` には絶対パスを渡してください。プランはそのワークツリー内で実行されます。`ORCHESTRATION COMPLETE` が出力されたらレビューしてマージしてください。

### 実装後レビュー

- **いつ使うか:** コードを書き終え、マージ前に多角的に点検したいとき。
- **コマンド:** `review-work` スキル。

実装後に `review-work` スキルを実行して多角的な実装後レビューを得ます。挙動を変えずに AI っぽいコードを整理する `remove-ai-slops` や、編集系操作の後にフィードバックを返す `comment-checker` と組み合わせると効果的です。
