LazyCodex(LZX)는 공개 표면을 작게 유지합니다. 네 개의 커맨드 기둥(`$ulw-plan`, `$start-work`, `$ulw-loop`, 그리고 `/init-deep`)과 그 위에 얹히는 스킬들이 전부입니다. 핵심은 작업의 모양에 맞는 커맨드를 고르고, 그 커맨드가 검증된 완료까지 일을 끌고 가도록 맡기는 것입니다.

LazyCodex는 [OmO(oh-my-openagent)](https://github.com/code-yeongyu/oh-my-openagent)를 Codex 에이전트 하니스로 패키징한 것입니다 — Codex를 위한 LazyVim이라고 보시면 됩니다. 아래에서는 Sisyphus가 Hephaestus, Oracle, Librarian을 오케스트레이션하고, 사용자는 진입점만 고르면 됩니다.

## 올바른 커맨드 고르기

대부분의 작업은 전체 루프보다 적은 것으로 충분합니다. 작업이 실제로 필요로 하는 것에 커맨드를 맞추세요:

| 필요 | 사용할 것 | 이유 |
| --- | --- | --- |
| 코드 작성 전 의사결정 | `$ulw-plan` | Prometheus가 모호한 아이디어를 `plans/<slug>.md`의 의사결정 완료 계획으로 바꿉니다. |
| 승인된 계획 실행 | `$start-work` | 모든 체크박스가 끝날 때까지 내구성 있는 Boulder 진행으로 체크리스트를 실행합니다. |
| 오픈엔드, 반드시 검증 | `$ulw-loop` | Oracle이 증거로 결과를 검증할 때까지 계속 진행합니다. |
| 큰 저장소와의 첫 접촉 | `/init-deep` | 편집 전에 계층적 `AGENTS.md` 프로젝트 메모리를 생성합니다. |

`$ulw-plan`은 절대 제품 코드를 작성하지 않습니다 — `$start-work`에 넘길 계획으로 끝납니다. 진행을 희망 섞인 상태 보고가 아니라 검증으로 측정하고 싶을 때 `$ulw-loop`을 사용하세요.

## 유스케이스

### 계획 우선 작업

대표적인 흐름입니다. `$ulw-plan`으로 접근 방식을 정한 뒤 `$start-work`로 끌고 갑니다.

- **언제 쓰나:** 구현을 시작하기 전에 의사결정, 시퀀싱, 트레이드오프를 먼저 정리해야 할 때.
- **커맨드:** `$ulw-plan` 그다음 `$start-work`.

<Steps>

<Step>

#### `$ulw-plan`으로 작업 계획하기

Prometheus가 소크라테스식 인터뷰를 진행하고, 코드베이스를 병렬로 탐색하고, Metis 갭 분석을 돌린 뒤, 의사결정 완료 계획을 `plans/<slug>.md`에 작성합니다. 제품 코드는 절대 건드리지 않습니다.

```bash
$ulw-plan "리프레시 토큰 회전이 포함된 OAuth 로그인 추가"
```

코드 변경이 아니라 계획 파일을 들고 나오게 됩니다.

</Step>

<Step>

#### `$start-work`로 계획 실행하기

계획을 `$start-work`에 넘기세요. `.omo/boulder.json`의 내구성 있는 Boulder 상태로 모든 최상위 체크박스를 실행하고, 독립적인 하위 작업은 병렬 서브에이전트로 분산하며, 엄격한 TDD와 증거 게이트를 적용합니다.

```bash
$start-work add-oauth-login
```

모든 체크박스가 체크되면 **ORCHESTRATION COMPLETE** 블록을 출력합니다.

</Step>

</Steps>

전체 흐름을 한 블록으로:

```bash
$ulw-plan "리프레시 토큰 회전이 포함된 OAuth 로그인 추가"
$start-work add-oauth-login
```

요구사항이 이미 바로 계획할 만큼 명확하다면, 인터뷰 위주의 계획 단계를 건너뛰고 계획이 있는 상태에서 `$start-work`로 시작해도 됩니다.

### 오픈엔드, 검증된 작업

- **언제 쓰나:** 에이전트가 다 됐다고 주장할 때가 아니라 결과가 증명될 때까지 계속 진행되어야 하는 작업.
- **커맨드:** `--completion-promise`와 함께 쓰는 `$ulw-loop`.

`$ulw-loop`은 자기참조적입니다. 에이전트는 작업이 완료됐다고 판단하면 약속을 내보내지만, 그것만으로 루프가 끝나지는 않습니다 — Oracle이 먼저 결과를 검증해야 합니다. 검증이 실패하면 루프는 계속됩니다. 반복 상한은 ultrawork 모드에서 500회, 일반 모드에서 100회입니다.

```bash
$ulw-loop "CI에서 불안정한 체크아웃 테스트를 통과시키기" --completion-promise="체크아웃 스위트가 3회 연속 그린" --strategy=continue
```

각 반복마다 컨텍스트를 새로 시작하려면 `--strategy=reset`을, 누적된 컨텍스트를 유지하려면 `--strategy=continue`를 사용하세요.

### 대규모 코드베이스 온보딩

- **언제 쓰나:** 저장소가 기억만으로 설명하기에 너무 크거나 오래되어, 이후 에이전트가 편집 전에 이정표를 필요로 할 때.
- **커맨드:** `/init-deep`.

`/init-deep`은 복잡한 디렉터리를 점수화하고 필요한 코드 근처에 계층적 `AGENTS.md` 컨텍스트를 작성합니다. 이후의 모든 실행이 추측 대신 프로젝트 메모리로 시작합니다.

```bash
/init-deep
```

코드베이스의 모양이 바뀌면 다시 실행하세요.

### 워크트리 격리 실행

- **언제 쓰나:** 메인 체크아웃에서 떼어두고 싶은 구조적 변경이나 위험한 실험.
- **커맨드:** `$start-work --worktree <path>`.

실행을 격리된 Git 워크트리로 향하게 해서, 계획이 도는 동안 메인 체크아웃을 깨끗하게 유지하세요.

```bash
$start-work refactor-payments --worktree ../wt-refactor-payments
```

계획은 해당 워크트리 안에서 실행됩니다. `ORCHESTRATION COMPLETE`가 출력되면 리뷰하고 머지하세요.

### 구현 후 리뷰

- **언제 쓰나:** 코드를 다 작성했고 머지 전에 여러 각도로 점검하고 싶을 때.
- **커맨드:** `review-work` 스킬.

구현 후 `review-work` 스킬을 실행해 여러 각도의 구현 후 리뷰를 받으세요. 동작을 바꾸지 않고 AI 티 나는 코드를 정리하는 `remove-ai-slops`나, 편집류 작업 후 피드백을 주는 `comment-checker`와 함께 쓰면 좋습니다.
