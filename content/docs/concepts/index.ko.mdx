OmO는 거대한 프롬프트 한 덩어리가 아닙니다. 요청이 진짜로 무슨 뜻인지, 누가 처리해야 하는지, 일이 끝날 때까지 어떻게 루프를 굴릴지를 정하는 작은 장치들의 모음입니다.

## 의도부터 본다

OmO는 모든 프롬프트를 글자 그대로만 다루지 않습니다. 사용자가 진짜로 원하는 게 설명인지, 조사인지, 구현인지, 계획인지, 리뷰인지, 위임인지를 먼저 묻습니다. 같은 문장도 맥락에 따라 다른 일이 됩니다.

| 사용자가 한 말 | 하네스가 해석한 일 |
|---|---|
| "왜 깨졌어?" | 재현하고, 근본 원인을 찾고, 가능하면 고칩니다. |
| "한번 봐줄래?" | 요약에서 멈추지 않고 조사와 해결까지 갑니다. |
| "가장 좋은 방법은?" | 방법을 고르고, 코드 맥락이 행동을 가리키면 실행합니다. |
| "리뷰해줘" | 버그, 위험, 빠진 테스트, 검증 공백을 먼저 찾습니다. |

## 계획과 실행을 나눈다

Prometheus, Metis, Momus, Atlas는 결정이 구현 레인 안에서 즉흥으로 만들어지지 않게 하는 역할입니다.

| Agent | 경계 |
|---|---|
| Prometheus | 인터뷰하고, 조사하고, 계획을 씁니다. production code는 건드리지 않습니다. |
| Metis | 계획이 굳기 전에 숨은 의도, 모호함, 빠진 제약, AI 실패 지점을 끌어냅니다. |
| Momus | 파일 참조, acceptance criteria, 검증이 구체적이지 않으면 계획을 거절합니다. |
| Atlas | 승인된 계획을 좁은 작업으로 실행하고, 학습을 누적하고, 단계마다 검증합니다. |
| Sisyphus-Junior | 카테고리로 생성되는 executor. 할당된 한 단위에만 집중하고 재위임은 못 합니다. |

핵심 계약은 "Decision Complete"입니다. 계획이 끝나면 구현자에게 숨은 제품/아키텍처 결정이 남아 있으면 안 됩니다.

## 모델 계열

OmO는 모델마다 동작이 달라서 에이전트와 카테고리로 라우팅합니다.

| 계열 | 어울리는 일 |
|---|---|
| Claude-like | 구조화된 오케스트레이션과 긴 규칙 준수. Claude Opus, Sonnet, Kimi K2.5, GLM 5가 Sisyphus 스타일에 맞습니다. |
| GPT | 명시적 추론과 아키텍처. GPT-5.4가 Hephaestus, Oracle, Momus, hard logic 카테고리를 돌립니다. |
| Gemini | 시각/프론트엔드 작업처럼 다른 추론 스타일이 도움이 되는 일. |
| Fast utility | Explore, Librarian, quick fix, grep 식 작업, 저렴한 background research. |

모든 작업을 가장 비싼 모델로 "올리지" 마세요. Explore와 Librarian은 일부러 빠르고 저렴합니다. 검색이 오케스트레이션 토큰을 태우면 안 됩니다.

## Category + Skill

Category는 일을 설명합니다. Skill은 playbook과 필요할 때 MCP 서버를 주입합니다.

| Category | 용도 |
|---|---|
| `visual-engineering` | Frontend, UI/UX, design, styling, animation. |
| `ultrabrain` | hard logic, architecture, deep reasoning. |
| `deep` | 자율 조사와 구현. |
| `artistry` | 창의적이거나 비정형적인 문제 해결. |
| `quick` | 단일 파일 수정, typo fix, 간단한 변경. |
| `unspecified-low` | 다른 카테고리에 안 맞는 저노력 작업. |
| `unspecified-high` | 다른 카테고리에 안 맞는 고노력 작업. |
| `writing` | 문서, prose, technical writing. |

좋은 위임은 카테고리와 스킬을 같이 명시합니다.

```ts
task({
  category: "visual-engineering",
  load_skills: ["frontend-ui-ux", "playwright"],
  run_in_background: false,
  prompt: "Implement and browser-verify the responsive sidebar.",
});
```

## 하네스 기능으로서의 연속성

Sisyphus라는 비유는 그냥 비유가 아닙니다. 끝나지 않은 일을 시스템이 다시 굴립니다.

| 장치 | 막는 실패 |
|---|---|
| Todo continuation | todo가 남았는데 에이전트가 멈춤. |
| Ralph loop / ulw loop | 자기 이어가기를 여러 번 해야 하는 긴 작업. |
| Boulder state | `/start-work`가 세션을 넘어 Prometheus 계획을 다시 잡음. |
| Session recovery | tool result 누락, 도구 부재, empty content, context-limit 실패. |
| Runtime fallback | 모델/공급자 실패 시 다음 fallback으로 이동. |
