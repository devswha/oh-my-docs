## 행동보다 의도

OmO는 모든 문장을 글자 그대로 실행하지 않습니다. 먼저 사용자가 진짜 원하는 것이 설명인지, 조사인지, 구현인지, 계획인지, 리뷰인지, 위임인지 판단합니다.

| 표면 요청 | 하네스의 해석 |
|---|---|
| “왜 깨졌어?” | 재현하고, 원인을 찾고, 가능하면 고칩니다. |
| “한번 봐줄래?” | 단순 요약이 아니라 조사와 해결을 의미합니다. |
| “가장 좋은 방법은?” | 코드베이스 맥락이 있으면 방법을 고르고 구현합니다. |
| “리뷰해줘” | 요약보다 버그, 위험, 누락된 테스트, 검증 공백을 먼저 찾습니다. |

## 계획과 실행 분리

Prometheus, Metis, Momus, Atlas는 결정을 구현에서 분리하기 위해 있습니다.

| Agent | 경계 |
|---|---|
| Prometheus | 인터뷰, 조사, 계획 작성. production code는 수정하지 않습니다. |
| Metis | 계획이 굳기 전에 숨은 의도, 모호함, 빠진 제약, AI 실패 지점을 찾습니다. |
| Momus | 파일 참조, acceptance criteria, 검증 방법이 구체적일 때까지 계획을 거절합니다. |
| Atlas | 승인된 계획을 focused task로 실행하고, learnings를 누적하며, 각 단계를 검증합니다. |
| Sisyphus-Junior | category로 생성되는 executor. 하나의 작업 단위에 집중하고 재위임하지 않습니다. |

핵심 규칙은 “Decision Complete”입니다. 좋은 계획은 구현자에게 숨은 제품/아키텍처 결정을 남기지 않습니다.

## 모델 계열

OmO는 agent와 category별로 모델을 라우팅합니다. 모델마다 잘하는 행동이 다르기 때문입니다.

| 계열 | 주 용도 |
|---|---|
| Claude-like | 구조화된 오케스트레이션과 규칙 준수. Claude Opus, Sonnet, Kimi K2.5, GLM 5가 Sisyphus 계열 흐름에 잘 맞습니다. |
| GPT | 명시적 추론과 아키텍처. GPT-5.4는 Hephaestus, Oracle, Momus, hard logic category에 주로 쓰입니다. |
| Gemini | visual/frontend, creative work처럼 다른 추론 스타일이 도움이 되는 작업. |
| 빠른 utility model | Explore, Librarian, quick fix, grep식 조사, 저렴한 background research. |

모든 작업을 가장 비싼 모델로 “업그레이드”하지 마세요. Explore와 Librarian은 retrieval에 집중하므로 빠르고 저렴한 모델을 쓰는 것이 의도입니다.

## Category + Skill

Category는 작업 성격이고, Skill은 도메인 지식과 필요한 MCP를 주입합니다.

| Category | 용도 |
|---|---|
| `visual-engineering` | Frontend, UI/UX, design, styling, animation. |
| `ultrabrain` | Hard logic, architecture, deep reasoning. |
| `deep` | Autonomous research and implementation. |
| `artistry` | 창의적이거나 비정형적인 문제 해결. |
| `quick` | 단일 파일 수정, typo, 단순 변경. |
| `unspecified-low` | 다른 category에 맞지 않는 낮은 노력 작업. |
| `unspecified-high` | 다른 category에 맞지 않는 높은 노력 작업. |
| `writing` | 문서, 산문, 기술 글쓰기. |

좋은 위임은 둘 다 명시합니다.

```ts
task({
  category: "visual-engineering",
  load_skills: ["frontend-ui-ux", "playwright"],
  run_in_background: false,
  prompt: "Implement and browser-verify the responsive sidebar.",
});
```

## 연속성 자체가 기능

Sisyphus라는 이름은 비유가 아니라 동작 방식입니다. 끝나지 않은 작업을 다시 밀어 올립니다.

| 장치 | 막아주는 실패 |
|---|---|
| Todo continuation | todo가 남았는데 agent가 멈추는 상황. |
| Ralph loop / ulw loop | 긴 작업이 반복적인 자기 continuation을 필요로 하는 상황. |
| Boulder state | `/start-work`가 Prometheus plan을 세션 너머로 이어가는 상황. |
| Session recovery | tool result 누락, unavailable tool, empty content, context limit 실패. |
| Runtime fallback | 모델/provider 실패 시 설정된 fallback으로 이동해야 하는 상황. |
