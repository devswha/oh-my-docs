## 핵심 순서

OpenCode에서 core agent의 Tab 순서는 고정입니다: Sisyphus, Hephaestus, Prometheus, Atlas. 나머지 agent는 그 뒤를 따릅니다.

| Agent | 역할 | 메모 |
|---|---|---|
| Sisyphus | 메인 orchestrator. 의도 파악, 위임, todo 추적, 완료까지 진행. | Claude Opus/Kimi/GLM식 orchestration에 잘 맞고 GPT-5.4 경로도 있습니다. |
| Hephaestus | GPT-native autonomous deep worker. | 아키텍처, 깊은 디버깅, cross-domain synthesis에 사용합니다. |
| Prometheus | 전략 planner와 interviewer. | 계획을 만들며 production code를 수정하지 않아야 합니다. |
| Atlas | 승인된 계획 실행. | Prometheus plan을 읽고 boulder state를 만들며 작업 단위를 위임하고 검증합니다. |
| Oracle | Read-only architecture/debugging consultant. | 낯선 패턴, 보안/성능 tradeoff, 어려운 debugging에 적합합니다. |
| Librarian | 문서, remote repo, OSS 예제 조사. | retrieval은 빠르고 저렴한 모델을 쓰는 것이 의도입니다. |
| Explore | 빠른 codebase search. | “X가 어디 있어?”와 pattern discovery에 적합합니다. |
| Multimodal-Looker | 이미지, PDF, 다이어그램, screenshot. | vision-capable model로 추출/요약합니다. |
| Metis | 계획 전 gap analyzer. | 숨은 의도, 모호함, 누락 제약을 찾습니다. |
| Momus | 계획/결과 reviewer. | 모호하거나 검증 불가능한 계획을 거절합니다. |
| Sisyphus-Junior | Category로 생성되는 executor. | 하나의 작업 단위에 집중하며 재위임하지 않습니다. |

## 도구 경계

| Agent | 제약 |
|---|---|
| Oracle | Read-only consultation. write/edit/delegation 없음. |
| Librarian | Research-only. write/edit/delegation 없음. |
| Explore | Search-only. write/edit/delegation 없음. |
| Multimodal-Looker | Media analysis, read 중심. |
| Atlas | 계획 실행을 orchestrate하지만 구현 단위는 위임합니다. |
| Sisyphus-Junior | 하나의 배정된 단위를 구현하고 검증 후 반환합니다. |

이 제약은 하네스의 일부입니다. 똑똑한 agent가 잘못된 역할에서 위험한 작업을 하지 않게 막습니다.

## Category Worker

`task` tool에 `category`가 들어오면 OmO는 해당 category의 model/profile을 가진 Sisyphus-Junior 계열 worker를 만듭니다. Category는 domain을 설명하고, skill은 지시와 MCP를 더합니다.

| Category | Worker 동작 |
|---|---|
| `visual-engineering` | Frontend/UI/design. `frontend-ui-ux`와 browser verification을 함께 씁니다. |
| `ultrabrain` | Hard reasoning과 architecture. prompt는 goal-oriented로 둡니다. |
| `deep` | Autonomous research + implementation. 복잡하지만 coherent한 작업에 적합합니다. |
| `artistry` | 창의적이거나 비정형적인 접근. |
| `quick` | 작은 변경. visual work를 diff가 작다는 이유로 quick에 넣지 않습니다. |
| `writing` | 문서와 prose. |

## Background Agents

Background agent를 쓰면 메인 세션은 계속 구현하면서 specialist가 병렬로 조사합니다.

```ts
task({
  subagent_type: "explore",
  load_skills: [],
  run_in_background: true,
  prompt: "Find all auth policy implementations and summarize file paths.",
});
```

독립적인 조사는 background mode를 쓰고, 다음 구현 결정을 막는 결과는 synchronous mode를 사용합니다.
