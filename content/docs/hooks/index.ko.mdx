## Hook이 하는 일

Hook은 OpenCode session 바깥의 harness layer입니다. instruction을 넣고, slash command를 라우팅하고, 깨진 state를 복구하고, 끝나지 않은 작업을 다시 이어줍니다.

| 계열 | 예시 | 목적 |
|---|---|---|
| Routing | `auto-slash-command`, `keyword-detector`, `category-skill-reminder` | 사용자 의도를 맞는 workflow로 보냄 |
| Continuation | `ralph-loop`, `todo-continuation-enforcer`, `task-reminder` | 미완료 작업이 조용히 멈추는 것을 막음 |
| Recovery | `session-recovery`, `edit-error-recovery`, `json-error-recovery` | 흔한 model/tool 실패 복구 |
| Tool integration | `interactive-bash-session`, `claude-code-hooks`, MCP hooks | tmux, 외부 도구, agent별 tooling 연결 |
| Context | `directory-agents-injector`, AGENTS-style injection | repo나 directory convention 주입 |

## 실전에서는 네 가지만 보면 된다

1. 작업 시작 전에 필요한 instruction이 들어왔는가
2. 맞는 specialist 또는 category가 선택됐는가
3. tool/model glitch에서 복구할 수 있는가
4. todo와 verification이 끝날 때까지 이어지는가

## 안전하게 바꾸기

Hook이나 experimental option을 바꿀 때는 작게 바꾸세요.

- 한 번에 하나의 동작만 바꿉니다.
- config 변경 후 `bunx oh-my-opencode doctor --verbose`를 실행합니다.
- harness 자체를 debugging하는 게 아니면 continuation hook은 켜 둡니다.
- repo 전용 behavior는 project config에 둡니다.
- hook assumption은 `.opencode/skills/`나 docs에 남깁니다.

## 관련 Config

자주 보이는 experimental option은 context/recovery 쪽입니다: `aggressive_truncation`, `auto_resume`, `preemptive_compaction`, `truncate_all_tool_outputs`. 정확한 지원 여부는 설치된 version의 `doctor`와 schema를 기준으로 보세요.
