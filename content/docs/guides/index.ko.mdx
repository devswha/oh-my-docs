## workflow 선택

| 상황 | 사용 | 이유 |
|---|---|---|
| 작은 수정 또는 직접 질문 | 일반 prompt | overhead가 가장 낮습니다. |
| 복잡하지만 맡기고 싶을 때 | `ulw` / `ultrawork` | Sisyphus가 탐색, 위임, 구현, 검증, continuation을 처리합니다. |
| 복잡하고 정확도가 중요할 때 | Prometheus `@plan` → `/start-work` | 구현 전에 결정을 끝내고 Atlas가 계획을 실행합니다. |
| GPT-native deep reasoning | Hephaestus | 아키텍처, 어려운 디버깅, cross-domain reasoning용 deep worker. |
| 좁은 구현 단위 하나 | `task(category=..., load_skills=[...])` | 맞는 model profile로 Sisyphus-Junior가 집중 실행합니다. |

## Ultrawork

`ulw`는 “그냥 해줘” 경로입니다. 목표와, 가능하면 성공을 증명할 표면을 같이 주세요.

```text
ulw fix the login error message and verify success/failure in the browser
```

좋은 `ulw` prompt에는 다음이 들어갑니다.

- 바꿔야 하는 사용자 관찰 동작.
- 검증 표면: browser, CLI, API, tests, docs build.
- 바꾸면 안 되는 경계.

## Prometheus + Atlas

속도보다 정확도가 중요하면 Prometheus를 사용합니다.

1. Sisyphus에서 `@plan "..."`을 입력하거나 Tab으로 Prometheus를 선택합니다.
2. 요구사항이 명확해질 때까지 질문에 답합니다.
3. Prometheus가 `.sisyphus/plans/` 아래에 계획을 씁니다.
4. `/start-work`를 실행합니다.
5. Atlas가 계획을 읽고 boulder state를 만들며, 위임 실행하고 나중에 이어갈 수 있습니다.

`/start-work`가 active plan을 찾지 못하면 먼저 Prometheus plan을 만들거나 stale `.sisyphus/boulder.json`을 정리해야 합니다.

## Hephaestus vs Sisyphus

| 기준 | Hephaestus | Sisyphus + `ulw` |
|---|---|---|
| 모델 스타일 | GPT-native autonomous deep worker. | Category와 skill로 위임하는 orchestrator. |
| 적합한 일 | 아키텍처, 어려운 디버깅, cross-domain reasoning. | 대부분의 복잡한 “맡아서 처리해줘” 작업. |
| 계획 | 실행하면서 자체 계획. | Prometheus plan을 쓰거나 자율 탐색. |
| 위임 | Explore/Librarian으로 context를 많이 모읍니다. | Category worker와 specialist를 사용합니다. |

기본 추천: 대부분은 `ulw`, GPT-5.4식 깊은 추론이 필요할 때 Hephaestus.

## 설치를 에이전트에게 맡길 때

upstream README는 OmO 설치를 에이전트에게 맡기는 흐름을 권장합니다. 설치를 돕는 agent는 다음을 해야 합니다.

1. 사용자가 가진 subscription/provider를 묻습니다.
2. 답변에 맞춰 `bunx oh-my-opencode install --no-tui ...`를 실행합니다.
3. 필요한 경우 interactive terminal에서 `opencode auth login`을 안내합니다.
4. `bunx oh-my-opencode doctor`를 실행합니다.
5. model setup과 safe override를 설명합니다.

사용자가 명시하지 않았다면 model 설정을 바꾸거나 hook/MCP를 끄지 마세요.

## 모델 선택 원칙

- Sisyphus는 Claude Opus, Kimi K2.5, GLM 5 계열에서 가장 안정적입니다.
- Hephaestus, Oracle, Momus는 GPT-friendly 또는 GPT-native입니다.
- Visual engineering은 Gemini식 시각/frontend reasoning이 유리합니다.
- Explore와 Librarian은 빠르고 저렴해야 합니다.
- Runtime fallback과 `fallback_models`는 실패 시 workflow를 멈추지 않고 다음 모델로 이동하기 위한 장치입니다.
